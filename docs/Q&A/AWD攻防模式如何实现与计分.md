# Q&A：如何实现攻防模式（AWD）？学生如何防守/进攻？得分如何计算？

本文面向“平台实现者 + 参赛学生”解释 AWD（Attack With Defense）模式的核心机制、参与方式与计分口径。以仓库现有后端设计文档为准，不新增架构边界。

- AWD 数据模型：`ctf/docs/architecture/backend/02-database-design.md`（§8 AWD 模块）
- 竞赛配置（awd_config）：`ctf/docs/architecture/backend/02-database-design.md`（JSONB：`contests.config.awd_config`）

## 1. AWD 是什么（平台视角）

AWD 竞赛的最小实现要素是：

1. **队伍**：参赛单位是 team（不是单个 user）。
2. **每队一套同构服务**：每道 AWD 题目/服务，对每个队伍都启动一个“相同初始镜像”的服务实例。
3. **轮次（Round）驱动**：比赛按固定间隔（例如 5 分钟一轮）推进。
4. **Flag 轮换**：每轮为每队生成新的 Flag，并注入到该队服务实例中。
5. **健康检查（Checker）**：平台定时检测每队服务是否存活（HTTP `/health` 或 TCP dial）。
6. **攻击提交**：队伍从其他队伍服务中夺取 Flag 并提交到平台。
7. **计分**：每轮根据攻击成功次数、服务存活、被攻破次数等汇总得分，更新排行榜并推送。

> 关键点：平台必须把“选手可攻击的服务网络”和“平台内部服务（API/DB/Redis）”做严格隔离。否则 AWD 会天然变成“打平台”而不是“打对手”。

---

## 2. 学生如何防守（Defend）

防守目标只有两类：

1. **让你的服务持续存活**：保证平台 checker 在每轮都能判定你队服务为 up（例如 `/health` 200 或端口可连通）。
2. **让对手拿不到你的 Flag**：修补漏洞、加固配置、加鉴权或限速等，但不能把服务修到不可用（否则会被判 down）。

典型防守动作（按常见赛题）：

- 修补输入校验、SQL 注入、命令注入、文件包含等漏洞点。
- 增加鉴权/随机化令牌，避免未授权读到 `/flag/flag.txt`（或服务内存储的 flag）。
- 加 WAF/限速/黑名单（注意不能误杀 checker 的访问）。
- 修复默认口令、关闭调试接口、减少暴露端口。

平台侧“允许学生防守”的前提（需要实现支持）：

- 每队必须有**仅对本队可用**的管理通道去修改自己的服务（例如 WebTerminal/SSH/补丁上传）。
- 该管理通道必须与对抗网络隔离，避免被其他队伍利用成后门。

## 3. 学生如何进攻（Attack）

进攻的动作非常单一：

1. 访问对手队伍对外暴露的服务端口。
2. 利用漏洞拿到对手在“当前轮次”的 Flag（平台每轮会轮换）。
3. 把该 Flag 提交到平台（平台会校验该 Flag 属于哪个队伍、哪道题、哪一轮）。

平台侧“允许学生进攻”的前提：

- 对抗网络必须允许跨队访问服务端口（否则无法对抗）。
- 同时仍要阻断对平台内部网段/宿主机管理端口的访问（防止越界）。

## 4. 平台如何实现 AWD（实现链路）

实现可以按“轮次编排”落地（详见 `05-key-flows.md` §7）：

### 4.1 轮次触发与幂等

- 定时器触发 `round_number=N`。
- 以 Redis 锁 `awd:round:lock:{contest_id}:{N}` 做幂等，避免多实例/抖动重复执行同一轮。

### 4.2 阶段 1：Flag 轮换注入

- 对每个队伍实例生成该轮 Flag（设计给出的思路是 HMAC(secret, team_id, contest_id, round)）。
- 通过 Docker API `CopyToContainer` 写入容器内固定路径（设计示例为 `/flag/flag.txt`），避免命令拼接注入风险，也无需重启容器。
- 将本轮 Flag 按 team 存入 Redis（hash），便于 O(1) 校验与过期清理。

### 4.3 阶段 2：服务存活检测（Checker）

- 对每队每服务做 HTTP `/health` 或 TCP dial，记录 is_alive/latency 等。
- 写入 `awd_team_services.check_result`（JSONB）或独立检测结果表（以实现为准）。

### 4.4 阶段 3：收集攻击结果

- 从 `awd_attack_logs` 中聚合本轮成功攻击记录（或在提交时就写入并打分）。
- 需要去重规则（强烈建议）：同一轮内同一 `attacker_team -> victim_team -> challenge` 只计一次分，重复提交记日志不加分。

### 4.5 阶段 4：计算防守分

防守分的常见口径与 `05-key-flows.md` 的说明一致：

- 服务存活：`+alive_score`（或 `defense_score`）
- 服务宕机：`+0`
- 被攻破：`-breach_penalty * (攻破你的不同队伍数)`

> `breach_penalty` 建议作为 `contests.config.awd_config` 的参数；`attack_score/defense_score` 可直接使用 `awd_rounds` 表中的字段。

### 4.6 阶段 5：写入与广播

- 在事务里写入本轮得分明细与队伍总分（数据库为事实来源）。
- 同步更新 Redis 排行榜缓存（便于实时展示）。
- WebSocket 广播轮次结果/排行榜更新事件。

## 5. 得分如何计算（建议采用“每轮汇总”口径）

按设计文档里的变量名表达，一个可解释、易实现的计分公式是：

```
round_total(team, challenge) =
  attack_score_per_capture * unique_captures(team, challenge)
+ defense_alive_score * is_alive(team, challenge)
- breach_penalty * unique_attackers_against(team, challenge)
```

然后对本轮所有 challenge 求和，得到队伍本轮总分；再累加到竞赛总分。

### 5.1 Flag 的有效性与去重（必须明确）

为了比赛公平与可实现性，平台需要明确：

- Flag 必须属于“当前轮次”（必要时允许上一轮短窗口容错，详见 `05-key-flows.md` 的竞态处理建议）。
- 同一轮次对同一受害队伍同一题的重复提交不加分（但记录到 `awd_attack_logs` 用于审计与检测滥用）。
- 过期/错误 Flag 不加分，并返回明确错误原因（错误/过期/重复）。

## 6. 常见坑（实现与参赛方都容易踩）

- Checker 被误伤：防守时加鉴权/限速把 `/health` 挡了，导致整轮被判 down。
- Flag 写入不可用：服务从硬编码位置读 flag，但平台写入路径不同；建议统一约定 `/flag/flag.txt`。
- 对抗网络越界：靶机/对抗容器能打到平台 DB/Redis 或宿主机管理端口，属于严重安全事故。
- 计分竞态：Flag 刚轮换时新旧 flag 混用导致误判；需要容错窗口或“注入完成标记”。

## 7. 如果出现“攻不破的堡垒”怎么办？

“攻不破”通常有两类原因，处理方式不同。

### 7.1 先判断：是真的难，还是规则/环境问题

平台侧建议按以下顺序排查（避免把“网络打不通”误判为“队伍太强”）：

1. 连通性：对抗网络是否允许跨队访问该服务端口？是否存在错误的网络隔离/端口映射？
2. Checker 视角：该队服务是否只对白名单（如 checker）开放，把其他队伍都挡掉了？
3. 服务一致性：该队是否“违规改协议/改接口”导致别人 exploit 失效，但 checker 仍判 up？
4. 日志证据：查看本轮攻击日志（失败原因分布）、服务存活矩阵（是否异常稳定或异常波动）。

如果是 1/2/3 类问题，属于竞赛规则或平台配置缺陷，应优先修正平台或执行仲裁。

### 7.2 竞赛规则层的解决方案（推荐提前写进规则）

AWD 里“堡垒”并不总是坏事，但需要避免比赛退化成“纯 uptime 比赛”。常见规则手段：

- 防守分封顶：单轮 `defense_alive_score` 有上限，避免纯防守滚雪球。
- 去重与限次：同一轮同一 victim 同一服务只允许攻击得分一次，防止刷分，同时让“广撒网”更有收益。
- 多服务/多题并行：让比赛不是押宝单一服务，降低单点堡垒对整体排名的决定性影响。
- 违规防守判定：明确“服务必须可被对抗网络访问且满足协议约定”，禁止只放行 checker 的“假存活”。

这些规则通常可以通过 `contests.config.awd_config` 参数化（例如 attack/defense/breach 的分值、轮次长度、检查频率等），并在赛前公示。

### 7.3 平台实现层的约束（避免“只对白名单开放”的假堡垒）

平台侧建议补足两类“公平性检查”，否则 checker 很容易被针对：

- 多视角存活检测：checker 不仅从管理网络检测，也从对抗网络侧（或模拟选手网络）检测服务可达性。
- Checker 随机化：checker 的源 IP/UA/路径可随机化（在不破坏协议的前提下），降低被硬编码放行的概率。

> 以上属于“竞赛公平性与对抗性”需要的工程约束，是否实现取决于你们一期/二期的目标与成本。

### 7.4 比赛进行中如何处理（管理员应急预案）

当确认存在“规则漏洞导致的堡垒”或“题目设计失衡”时，常见处置方式（需记录审计并公告）：

- 临时修正 checker：例如把健康检查从单点改为多点，或修正被错误放行的探测特征。
- 重置/回滚服务：对指定队伍或全体队伍重置到同一基线镜像（强制一致性），避免违规改动扩大影响。
- 调整计分参数：例如降低 `defense_alive_score` 或提高 `breach_penalty`（最后手段，需公告并解释）。
- 替换服务：移除失衡服务或新增一项更可打的服务（对进度影响大，谨慎使用）。

### 7.5 学生视角的策略建议（不依赖裁判改规则）

如果你确认目标队伍是真的“难攻”：

1. 优先保证己方服务稳定存活，别因为进攻把防守搞崩。
2. 把进攻目标转向“可攻破且回报更高”的队伍或服务，减少在单一堡垒上的时间消耗。
3. 利用轮次窗口：有些防守只在短期有效（重启、轮换、缓存、临时封禁），观察其规律再选择时机。
