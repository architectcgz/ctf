# Q&A：题目包上传后，学生是怎么启动题目实例的？

本文回答“教师/管理员上传题目包（Zip）后，学生点击启动实例”的端到端链路。默认存储方案为宿主机文件系统（LocalFS），题目包规范为 `challenge-pack-v1`。

关联文档：

- 题目包规范：`ctf/docs/contracts/challenge-pack-v1.md`
- 文件存储设计：`ctf/docs/architecture/backend/06-file-storage.md`

---

## 1. 上传后发生了什么（从 Zip 到可启动题目）

> 关键点：**学生启动的不是“Zip”，而是“题目记录 + 可用镜像/构建产物 + 运行参数”。**

### Step 1：教师/管理员上传题目包

1. 后端接收 Zip 上传（教师/管理员权限）。
2. Zip 原文件落盘（例如 `${STORAGE_ROOT}/challenge_packs/`），记录 `sha256/size/uploader/created_at` 等元数据。

### Step 2：导入与静态校验

1. 在隔离临时目录解包并定位题目包根目录。
2. 强制安全校验：拒绝 zip-slip（`..`/绝对路径）、拒绝 symlink、限制文件数/大小、拒绝 zip bomb。
3. 解析 `manifest.yml` 并校验字段（category/difficulty/暴露端口/资源限制/flag 模式等）。
4. 提取附件：只允许从 `attachments/` 目录提取，校验 `sha256`，然后写入 `${STORAGE_ROOT}/attachments/`（建议按内容寻址 key 存储）。

### Step 3：产出“可运行的镜像来源”

根据 `manifest.yml runtime.image.source` 决定学生实例启动时用什么镜像：

- `registry`：
  - 题目记录关联一个镜像引用 `image_ref`（禁止 `latest`），学生启动时按需 pull 并 run。
- `dockerfile`：
  - 题目包携带 `docker/Dockerfile` 是“可复现/审计材料”。
  - 是否平台在线构建取决于策略：
    - 推荐：教师/CI 构建后推送 registry，题目实际运行仍走 `registry`。
    - 若启用平台构建：必须异步构建队列 + 隔离 builder；构建成功后才允许题目发布/可启动。
- `tar`：
  - 导入镜像 tar 并登记为可用镜像，学生启动时直接 run（或按需 load）。

### Step 4：题目发布 gating

题目只有在满足以下条件时，才应对学生“可见/可启动”：

- 题目状态为 `active`（或设计文档定义的上线态）
- 镜像来源可用（registry ref 合法且可 pull，或构建产物 ready）
- 运行参数通过安全校验（expose 端口白名单、资源限制不超过上限等）

## 2. 学生点击“启动实例”时发生了什么（从题目到容器实例）

> 目标：为该学生（或该队伍）创建一个隔离的题目实例，并返回访问地址。

### Step 1：鉴权与配额检查

后端处理“启动实例”请求时至少做：

- 鉴权：学生身份 + 是否有权访问该题（例如题目已上线、竞赛已开始、队伍已加入等）。
- 并发限制：`max_instances_per_user` 或 `max_per_team`。
- 状态检查：题目/竞赛当前是否允许启动（未发布/已结束/已封禁等）。

### Step 2：分配端口与网络隔离

- 从可配置端口段分配一个宿主机端口（避免冲突，建议加锁/唯一约束兜底）。
- 为该实例创建独立 Docker network（或按竞赛模式规则共享 network），确保：
  - 不同用户/队伍实例之间默认不可达
  - 靶机网络与平台服务网络不可达

### Step 3：创建并启动容器

容器启动参数来自“题目记录 + manifest 派生配置”，通常包含：

- 镜像（registry/ref 或构建产物）
- 端口映射：`host_port -> container_port`
- 资源限制：CPU/内存/pids/只读根文件系统（如启用）
- 安全限制：禁止 privileged、capabilities 最小化、非 root 用户
- 环境变量注入（示例）：
  - 动态 Flag 相关：例如 `CTF_FLAG`（具体注入字段以平台实现为准）
  - 实例标识：`INSTANCE_ID`、`USER_ID`/`TEAM_ID`、`CHALLENGE_ID`

启动成功后，后端写入实例记录（DB + 可选缓存），包含：

- `instance_id`、归属（user/team）、challenge、容器 ID、宿主机端口、过期时间、状态 `running`

### Step 4：返回访问地址

后端返回给前端：

- 访问地址（如 `http://<host>:<mapped_port>` 或通过 Nginx 反代的路径）
- 实例 TTL / 到期时间 / 延时规则（如果有）

## 3. 学生为什么不能直接拿到题目包？

因为题目包包含 Dockerfile/源码/配置等信息，一旦对学生可下载，等价于：

- 题目被“开源”，难度与公平性崩溃
- 更严重：如果题目包里不慎包含敏感信息，会造成平台级泄漏

因此通常的策略是：

- 学生只可下载题目附件（`attachments/` 中的白名单文件），且下载必须鉴权
- 原始题目包 Zip、Dockerfile、构建日志仅教师/管理员可下载（审计/复现）

---

## 4. 最小可验收的端到端检查清单

- 上传 Zip 后，平台能给出“导入成功/失败”的字段级错误定位。
- 题目未构建完成（或镜像不可用）时，学生无法启动，返回明确错误（非 500）。
- 学生启动实例返回的地址可访问，且访问到的是自己的实例（不串线）。
- 实例到期后会被回收，端口与 network 不残留。
