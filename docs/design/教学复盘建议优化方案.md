# 教学复盘建议优化方案

> 状态：Draft
> 事实源：中间方案、信号层与建议层取舍，不覆盖当前 `docs/architecture/features/` / `docs/contracts/` / 代码事实
> 替代：无

## 定位

本文档保留“教学证据 -> 能力画像补充信号 -> 推荐理由 -> 教学复盘建议”的中间方案。

- 当前文档仍包含方案比较、范围边界和验证设想，不能直接当作当前已落地事实源。
- 如果后续方案落地，应把最终 owner、接口和建议口径回收到专题架构文档或契约文档。

## 1. 背景

当前平台已经具备能力画像、推荐练习、班级复盘、个人复盘归档、攻击证据链和攻击会话这些核心能力，论文主线里“攻防演练 -> 技能评估 -> 推荐练习 -> 教学复盘”这条链路已经能跑通。

现在真正偏弱的部分不是“有没有复盘页”，而是“复盘建议是否足够贴近训练事实、是否足够可解释”。现有建议更多是页面级提示，离论文里可论证的“基于实训数据生成教学建议”还有一层差距。

## 2. 毕设边界

这次优化只围绕下面四条闭环展开，不做产品外扩：

1. `实训/竞赛事实 -> 教学证据`
2. `教学证据 -> 能力画像补充信号`
3. `能力画像/过程信号 -> 推荐练习`
4. `训练结果/复盘材料 -> 教学复盘建议`

明确不做：

- 不引入聊天助手或 LLM 自动点评
- 不做泛化学习路径系统
- 不做新的大前端专题页面
- 不把教师建议扩成“教务管理系统”

## 3. 现状缺口

### 3.1 画像只强在覆盖率，弱在过程质量

当前 `skill_profiles` 主要根据“该维度已解题得分 / 该维度总分”计算，适合表达覆盖面，但不足以回答下面这些教学问题：

- 学生是在高效完成训练，还是靠大量反复试错才命中
- 学生有没有形成“实操 -> 提交 -> 复盘材料”的闭环
- 学生在 AWD 中是否有真实攻击参与，而不只是团队共享结果

### 3.2 弱项维度缺少置信度

当前学生目录和班级复盘会直接取 `skill_profiles` 最低分维度作为 `weak_dimension`。这会带来两个问题：

- 当多个维度长期为 `0` 时，弱项很容易退化成“按字典序命中”
- 当某个维度几乎没有训练证据时，系统仍会把它当成明确弱项

这也是为什么本地样本数据需要手动写 `skill_profiles` 才能让建议稳定。

### 3.3 推荐题理由过于模板化

当前推荐题理由基本只有“针对薄弱维度：XXX”。它能说明推荐来自哪个维度，但不能解释：

- 为什么是现在推荐这道题
- 为什么推荐这个难度
- 这道题是在补覆盖，还是在纠正训练过程问题

### 3.4 班级复盘建议过于阈值化

`GetClassReview` 现在主要依赖：

- 近 7 天活跃率
- 最近事件数
- 已解题数
- 最低画像维度

这能产出基础提示，但还没有把证据链、提交稳定性、复盘闭环、AWD 个人实操证据真正转成教学动作。

### 3.5 个人复盘观察项区分度不够

`buildReviewArchiveObservations` 已经开始用证据链生成观察项，但还比较粗：

- `hint_usage` 目前是固定 good 文案，实际上没有事实支撑
- `training_closure`、`submission_stability`、`hands_on_activity` 仍偏单点判断
- 没有把“维度弱项”和“过程弱项”分开表达

## 4. 方案比较

### 方案 A：继续在现有 service 里加阈值和文案

优点：

- 改动最小
- 交付快

缺点：

- 规则继续散落在 `teaching_readmodel`、`assessment`、`recommendation` 多处
- 难以解释“建议来自哪些事实”
- 很难写成论文里稳定的设计章节

结论：不采用。

### 方案 B：直接接入 LLM 生成建议

优点：

- 文案看起来会更自然
- 可以快速生成长段说明

缺点：

- 不利于论文答辩中的可验证性和可复现实验
- 需要额外处理成本、漂移和安全问题
- 现阶段平台并没有稳定的 prompt、评测和兜底体系

结论：不采用。

### 方案 C：增加“教学信号层 + 规则型建议层”

核心做法：

- 保留现有事实表和读模型
- 在建议生成前先把证据整理成结构化教学信号
- 再用统一规则层产出班级建议、个人观察和推荐理由

优点：

- 可解释、可归因、可测试
- 与论文里“基于实训数据生成能力画像与教学复盘建议”的表述一致
- 能复用已有证据链、画像、writeup、AWD 事实，不偏离当前系统边界

结论：采用。

## 5. 选定方案

### 5.1 总体思路

在现有 `internal/teaching/evidence` 旁边新增一个共享的“教学建议内核”，把建议生成拆成三层：

1. `事实读取层`
   - 继续由 `teaching_readmodel`、`assessment`、`challenge` 各自读取自己拥有的事实
2. `原始教学事实层`
   - 由 owning 模块把事实装配成统一的原始计数/统计快照
   - 这里不做 `confidence`、`is_weak`、`severity` 这类派生判断
3. `教学建议层`
   - 由 `internal/teaching/advice` 对原始事实快照做评估
   - 统一生成弱项维度、置信度、严重度、建议和推荐理由

这样可以避免继续把建议规则直接写死在：

- `teaching_readmodel.QueryService.GetClassReview`
- `assessment.ReportService.buildReviewArchiveObservations`
- `assessment.RecommendationService`

### 5.2 建议统一的信号族

本次完整设计覆盖以下几类信号：

#### 1. 活跃信号

- 近 7 天活跃天数
- 近 7 天事件数
- 最近活跃时间

用途：

- 判断是否掉出训练节奏
- 支撑班级层“活跃度”与“优先介入学生”

#### 2. 提交稳定性信号

- 正确提交数
- 错误提交数
- 连续错误提交长度
- 首次成功前尝试次数

用途：

- 区分“做不出来”和“试错成本过高”
- 给出更贴近过程质量的观察项

#### 3. 训练闭环信号

- 已解题但无 writeup
- 已解题且有 writeup
- manual review 通过数
- 推荐题解/复盘材料数量

用途：

- 判断学生是否形成“实操 -> 提交 -> 复盘输出”的闭环
- 给出“补题”和“补复盘材料”两类不同建议

#### 4. 实操参与信号

- `instance_access`
- `instance_proxy_request`
- `awd_attack_submission`
- `awd_traffic`

用途：

- 判断是否有真实动手过程
- 区分普通解题记录与 AWD 实战参与记录

#### 5. 维度事实

在 `skill_profile` 之外，再补原始统计字段：

- `evidence_count`：该维度下实际参与过多少训练事实
- `attempt_count`：该维度下有多少提交尝试

用途：

- 作为 `advice` 计算弱项置信度的输入
- 避免在 `teaching_readmodel`、`assessment`、`recommendation` 各自派生“弱项”

### 5.3 建议输出层

建议输出不再只是自由文案，而是统一为“规则码 + 证据摘要 + 动作建议”。

本次完整设计统一覆盖三类输出：

#### 1. 班级复盘建议

输出重点：

- 班级节奏是否下滑
- 哪个维度是“高置信度薄弱项”
- 哪些学生需要优先跟进
- 当前更适合“统一补基础题”还是“督促补复盘材料”

#### 2. 个人复盘观察

替换掉当前无事实支撑的 `hint_usage` 固定文案，改成：

- `training_closure`
- `submission_stability`
- `hands_on_depth`
- `dimension_focus`
- `awd_participation`

其中每个观察项都要求能追溯到具体事实计数。

#### 3. 推荐题理由

推荐题仍然围绕薄弱维度，但“哪个维度算弱、是否证据充足、当前更适合什么难度”统一由 `advice` 判定；推荐服务只负责拿候选题和装配结果。理由要补足三层信息：

- 为什么推荐这个维度
- 为什么推荐这个难度
- 证据来自哪些训练表现

例如：

- 维度覆盖低，且最近 7 天在该维度只有失败尝试，建议先补 `beginner/easy`
- 已有该维度基础成功记录，但闭环不足，建议布置一题需要 writeup 的进阶题

## 6. 结构收敛原则

这次优化不允许把建议规则继续散落在多个 service 里。应显式收敛到一个共享 owner。

推荐落点：

- 新增 `internal/teaching/advice`

建议由它统一承载：

- 原始教学事实的评估 contract
- 弱项置信度计算
- 弱项维度识别与排序
- 班级建议规则
- 个人观察规则
- 推荐理由生成规则
- 严重度统一语义

各模块保留的职责：

- `teaching_readmodel`
  - 读取班级与教师实时分析所需事实
  - 负责访问控制和教师读接口组织
- `assessment`
  - 读取画像、归档、报告相关事实
  - 负责报告导出和个人归档装配
- `recommendation service`
  - 不再拥有弱项维度判定逻辑
  - 只保留候选题查询、已解过滤和结果装配
- `challenge`
  - 继续负责推荐候选题读取

## 7. 范围边界

本次设计聚焦“建议层升级”，不碰以下内容：

- 不重写证据链和攻击会话架构
- 不改训练排名、班级得分和 solved 统计口径
- 不新增数据库事实表
- 不改教师端页面整体布局

本次完整设计直接收口以下内容：

1. 弱项维度从“最低分”升级为“最低分 + 证据置信度”
2. 班级复盘建议改为由共享规则层生成
3. 个人复盘观察项改为由共享规则层生成
4. 推荐题理由增加维度、难度和证据解释
5. 直接替换旧的 `Accent`、`Level` 等展示型字段，统一改为 `severity` 主语义字段

## 8. 验证方式

这次设计天然适合用现有样本数据做验证。验证重点不是“页面能打开”，而是“不同画像学生是否得到不同建议”。

建议使用 6 名样本学生至少覆盖：

- 高闭环学生
- 连续错误提交学生
- 低活跃学生
- 有实操但无 writeup 学生
- AWD 个人攻击参与学生
- 某维度训练证据不足学生

验收时需要检查：

1. 弱项维度是否仍出现无证据随机命中
2. 班级建议是否能区分“补活跃度”和“补闭环”
3. 推荐题理由是否能解释推荐原因，而不是只回显维度名
4. 个人观察项是否能明显区分不同学生样本
5. 同一学生在班级建议、个人观察和推荐题三条链路里，证据不足维度不能被任一链路判成明确弱项

## 9. 结论

这次最值得做的优化，不是把教学复盘页面再做复杂，而是把“建议生成”从阈值文案升级成一个基于教学信号、可解释、可验证的规则层。

这样处理后，平台在毕设里能够更稳地回答两个问题：

1. 系统如何把实训事实转成能力评估与推荐练习
2. 教师如何基于平台输出进行教学复盘，而不只是看若干原始日志和静态统计
