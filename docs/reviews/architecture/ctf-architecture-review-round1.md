## 必须修复的高严重性问题（31 项，按主题聚合）

### 一、安全类（12 项）

| 编号 | 来源 | 问题 | 影响 | 修正建议 |
|------|------|------|------|----------|
| S-01 | 03-H1 | Flag 文件 bind mount 注入，明文落盘宿主机 `/var/ctf/flags/` | Flag 批量泄露 | 改用 tmpfs + docker exec 注入，避免宿主机落盘；文件权限收紧为 0400 |
| S-02 | 03-H2 | AWD Flag 轮换通过 `sh -c` 拼接命令，存在命令注入风险 | 容器内 root 任意命令执行 | 改用 Docker API `CopyToContainer` 直接写入文件 |
| S-03 | 03-H3 | iptables 规则仅阻止特定端口，未阻断宿主机物理 IP 访问 | 靶机可访问 PostgreSQL/Redis/API | 增加 `-d <HOST_IP> -j DROP` 规则，或白名单策略默认 DROP 出站 |
| S-04 | 03-H6 | 动态 Flag 使用全局固定 salt，salt 泄露后所有 Flag 可计算 | 竞赛公平性完全丧失 | HMAC 输入加入实例级随机 nonce；每场竞赛独立 salt |
| S-05 | 01-H4 | 动态 Flag 算法缺少全局 secret 分离，salt 安全存储未规定 | Flag 可被逆向推导 | 加入 global_secret；salt 随机 32 字节+，加密存储 |
| S-06 | 01-H5 | 容器网络隔离缺少具体 iptables 规则设计 | 平台核心安全边界缺失 | 补充完整规则模板，或独立 06-security-design.md |
| S-07 | 03-H5 | 容器池预创建的容器已映射端口但无 iptables 保护 | 预创建容器可被扫描访问 | 预创建时不映射端口，分配时再添加；或预注入 DROP 规则 |
| S-08 | 03-H7 | 多容器题目的同子网 ACCEPT 规则需动态注入，遗漏则内部不通 | 多容器题目无法使用 | 在 CreateIsolatedNetwork 中强制注入 ACCEPT 规则 |
| S-09 | 02-H2 | 静态 Flag 明文存储在数据库和 Redis 中 | 拖库即泄露所有 Flag | 存储为 SHA-256(flag + salt) 哈希值，验证时比对哈希 |
| S-10 | 04-H3 | JWT 默认 HS256 对称签名，生产环境应使用非对称签名 | Token 伪造风险 | 生产环境强制 RS256/ES256；硬编码允许的 alg 白名单 |
| S-11 | 04-H8 | WebSocket Token 通过 URL 查询参数传递 | Token 泄露到日志/代理/Referer | 改用连接后首条消息认证，或短期一次性 ticket |
| S-12 | 03-H4 | seccomp 白名单从零构建仅 30 个系统调用，靶机无法运行 | 平台不可用 | 基于 Docker 默认 profile 做减法，禁止高危调用即可 |

### 二、并发与一致性类（8 项）

| 编号 | 来源 | 问题 | 影响 | 修正建议 |
|------|------|------|------|----------|
| C-01 | 05-H1 | Flag 提交乐观锁重试时事务回滚导致唯一约束幂等失效 | 同一用户同一题可能被计分两次 | 改用原子累加 `score = score + ?`；或重试时只重试 UPDATE 不回滚 INSERT |
| C-02 | 05-H2 | 动态计分分批更新破坏原子性，部分批次失败导致分值不一致 | 排行榜出现不公平的中间状态 | 校园级场景单事务批量 UPDATE 即可；如需分批则引入断点续传机制 |
| C-03 | 05-H3 | 动态计分分布式锁 TTL 10s 不够，锁内操作可能超时 | 并行计分导致 solve_count 错误 | 使用可续期锁（watchdog）；或将 Redis 更新移到锁外，缩短持锁时间 |
| C-04 | 02-H3 | submissions/user_scores/challenges 多表联动缺少并发安全机制 | 竞赛高峰并发提交计分错误 | user_scores 增加 version 字段；冗余计数用原子操作 |
| C-05 | 02-H5 | 排行榜 ZSET score 只存 total_score，无法处理同分排序 | 同分排序不正确 | score 编码为复合值：`total_score * 10^10 + (MAX_TS - last_solve_ts)` |
| C-06 | 02-H6 | 缺少竞赛维度的「是否已解出」快速判断索引 | 200 并发下每次提交全表扫描 | 增加部分唯一索引 `UNIQUE(user_id, challenge_id, contest_id) WHERE result='correct'` |
| C-07 | 05-H4 | AWD Flag 轮换期间存在时间窗口，新旧 Flag 混合验证问题 | AWD 模式公平性受损 | 先写 Redis 再注入容器，原子切换有效轮次；验证时同时检查当前和上一轮 |
| C-08 | 01-H3 | 进程内事件总线丢事件，排行榜更新走异步可能丢失 | 竞赛排行榜与实际得分不一致 | 关键路径（计分+排行榜）同步处理；事件总线仅用于非关键异步通知 |

## 建议优先修复顺序

### P0 — 阻塞开发（开发前必须修复）

1. **R-01** 容器上限 100 vs 需求 200：决定单机方案还是多机方案，影响整体架构
2. **R-03** 难度枚举对齐：影响数据库 DDL 和所有关联逻辑
3. **R-04/R-05** 用户管理 + 教师权限：影响 RBAC 设计和所有接口权限标注
4. **R-06/F-03** 竞赛题目管理 + 竞赛实例启动：核心功能链路断裂

## 总体评价

五篇架构文档的完成度较高，分层清晰、模块划分合理、技术选型匹配场景。主要风险集中在三个方面：

1. **安全边界**（12 项高）：CTF 靶场的安全要求远高于普通 Web 应用，当前 iptables 规则、Flag 存储、seccomp profile 等设计停留在原则层面，缺少可落地的具体方案，建议独立产出安全设计文档。

2. **并发一致性**（8 项高）：Flag 提交计分、动态计分、排行榜更新是竞赛核心路径，当前的乐观锁、分布式锁、事件总线设计存在多个竞态窗口，需要在实现前逐一修正。

3. **需求覆盖度**（API 文档 33 项问题）：API 设计文档的接口覆盖度不足，用户管理、教师权限、竞赛题目管理等核心接口缺失，需要优先补齐。

建议按 P0 → P1 → P2 顺序修复后再进入实现阶段。
