# 校园级 CTF AWD 模式完整设计

## 1. 文档目标

这份文档给出当前项目的 **最终 AWD 方案**，用于统一后续的：

- 系统设计文档
- 论文系统设计章节
- 答辩 PPT 主线
- 实现计划拆分

它不是某个单页面或单接口的局部设计，而是面向校内训练/校赛场景的一整套 Attack-Defense 平台方案。

## 2. 设计前提

### 2.1 场景边界

本方案默认面向：

- 校内训练 / 校赛
- 20-50 支队伍
- 平台需要同时支持教学、比赛、复盘和评估

### 2.2 题型边界

最终方案默认支持：

- `Web / HTTP` 服务型题目
- `Pwn / 二进制网络服务` 题目
- `多容器综合靶场` 题目

不把 AWD 设计限制为单一的 Web 题模式。

### 2.3 运行模式

最终方案采用 **混合模式**：

- 平台内负责比赛编排、裁判、监控、计分、复盘、轻量运维入口
- 深度攻防动作通过 `VPN / SSH / 本地工具` 完成

平台不是重型 Web IDE，也不是只提供一个提交框的比赛网站。

## 3. 课题约束与设计目标

### 3.1 毕业设计课题要求

根据项目内现有课题与开题材料，本平台必须同时覆盖：

- Docker 技术快速生成靶机
- 团队对抗模式
- 自动监控攻击流量
- 统计防守成功次数
- 生成实时排行榜
- 集中监控与管理
- 赛后教学复盘与实训数据沉淀

这意味着 AWD 不是一个“附加页面”，而必须是一套完整的比赛子系统。

### 3.2 总体目标

本方案的核心目标是：

1. 让 AWD 与 Jeopardy 在业务语义上彻底分离
2. 建立适合校内比赛的完整 Attack-Defense 裁判内核
3. 支持多题型、多服务形态、多轮次的统一比赛管理
4. 同时兼顾比赛可用性、教学价值和答辩表达完整性

- 公开赛的裁判内核和网络边界思路
- 高校平台的运营与教学闭环思路
- 适配校内规模的复杂度控制

## 5. 核心决策

本方案在开始实现前先固定 6 个不会回退的设计选择。

### 5.1 AWD 与 Jeopardy 彻底分池

最终方案中：

- Jeopardy 使用独立题库
- AWD 使用独立题库

不再采用“统一题库里补 AWD 字段”的方案。

### 5.2 AWD 的核心对象不是题目，而是 Service

Jeopardy 的核心对象是“解题题目”，AWD 的核心对象是“可对抗服务”。

因此 AWD 题库对象统一命名为：

- `AWDServiceTemplate`

### 5.3 采用混合模式

- 平台内给出战场态势、连接入口、提交入口、轻量运维动作
- 深度攻击、防守修补、调试和联调在外部作战面完成

### 5.4 以轮次和 checker 为裁判内核

AWD 不是“题目列表加 stolen flag 提交”，而是：

- 轮次驱动
- flag 轮换
- checker 裁决
- 三段分计分
- service 状态演进

### 5.5 平台只认可平台记录的官方事件

分数和裁判结果只来源于：

- 平台 checker 结果
- 平台攻击提交流程
- 平台记录的官方轮次状态
- 平台记录的管理员干预

### 5.6 平台不仅办赛，还做教学闭环

最终平台不仅要能比赛，还要支持：

- 赛后复盘
- 队伍与题目表现分析
- 能力评估
- 报告导出

## 6. 非目标

为控制复杂度，本方案明确不追求以下内容：

- 公开赛级多机房高可用
- 独立 checker 集群和镜像仓库分发系统
- 面向百队以上的复杂自动编排
- 浏览器内全功能 Web IDE
- 浏览器内通用 root shell
- 全量原始流量长期归档
- 完整工单式申诉系统

这些能力可以作为后续扩展，但不属于本轮最终方案的必要条件。

## 7. 总体架构

### 7.1 一句话架构

本系统的 AWD 模式是一套面向校内训练/校赛的混合式 Attack-Defense 比赛系统：

- 以独立 AWD Service 题库为基础
- 以轮次和 checker 为裁判内核
- 以平台控制面负责编排、监控、计分与复盘
- 以 VPN/SSH/本地工具承载深度攻防动作

### 7.2 六层结构

系统按职责分成 6 层。

#### 资源层

负责“比赛用什么”。

包含：

- `AWDServiceTemplate`
- 镜像与拓扑模板
- checker 模板
- flag 注入模板
- readiness 规则

#### 比赛配置层

负责“这场比赛怎么用”。

包含：

- `AWDContest`
- `ContestAWDService`
- `ContestTeam`
- 轮次参数
- 分值参数
- 冻结榜参数

#### 运行编排层

负责“比赛怎么跑起来”。

包含：

- service 实例创建与回收
- 轮次推进
- flag 生成与注入
- rebuild / restart / rollback

#### 裁判计分层

负责“平台如何认定结果”。

包含：

- checker 执行
- 提交判定
- 三段分计算
- 排行榜更新
- 裁判干预

#### 观测与告警层

负责“比赛哪里出了问题”。

包含：

- service 健康矩阵
- 攻击日志
- 流量事件
- 告警中心
- 事件时间线

#### 展示与复盘层

负责“如何被使用和如何沉淀”。

包含：

- 选手战场工作台
- 管理员控制台
- 比赛大屏
- 赛后复盘
- 报告导出

## 8. 角色模型与权限边界

### 8.1 选手

选手负责真实攻防动作。

平台内可做：

- 查看战场态势
- 查看我的服务状态
- 查看目标目录
- 提交 stolen flag
- 获取连接信息
- 上传补丁
- 执行重启 / 回滚 / 预设修复动作
- 记录队伍协作状态

平台外完成：

- 深度攻击
- 深度修补
- Pwn 调试
- 多容器联调
- 本地抓包和 exploit 脚本运行

### 8.2 管理员

管理员负责比赛可运行性与裁判可控性。

职责包括：

- 赛前 readiness 审核
- 轮次控制
- 告警处置
- service 重建与重跑 checker
- 冻结、暂停、恢复比赛
- 管理审计与导出

### 8.3 教师

教师侧更关注：

- 队伍表现
- service 表现
- 赛后复盘
- 教学建议
- 报告导出

教师不是赛时主控制角色。

### 8.4 大屏 / 公开视图

公开视图只展示：

- 排名
- 态势
- 摘要事件
- 流量趋势
- 告警摘要

不展示：

- SSH/VPN 入口
- flag
- 容器内部信息
- 管理员操作入口

## 9. 题库体系

### 9.1 Jeopardy 题库

对象：

- `JeopardyChallenge`

语义：

- 教学练习、普通比赛题目

生命周期围绕：

- 题面
- 附件
- 解题 flag
- writeup

### 9.2 AWD 题库

对象：

- `AWDServiceTemplate`

语义：

- 可用于攻防对抗的服务模板

生命周期围绕：

- 镜像或拓扑
- checker
- 动态 flag
- 防守入口
- 恢复策略
- readiness

### 9.3 为何必须彻底分池

因为普通题和 AWD service 的约束完全不同。

AWD 题库必须额外具备：

- service 类型
- 暴露入口
- checker 配置
- 动态 flag 策略
- 防守入口
- 观测点
- 恢复能力

这些不应作为 Jeopardy 题目的附属字段。

## 10. AWD Service 模型

### 10.1 核心对象

`AWDServiceTemplate` 是独立 AWD 题库里的核心对象。

### 10.2 基础字段

至少包含：

- `id`
- `name`
- `slug`
- `category`
- `difficulty`
- `description`
- `service_type`
- `deployment_mode`
- `version`
- `status`

### 10.3 服务类型

建议固定为：

- `web_http`
- `binary_tcp`
- `multi_container`

### 10.4 部署相关字段

包含：

- 镜像引用
- 拓扑模板引用
- 公开入口服务
- 暴露端口列表
- service 依赖关系
- 资源限制建议

### 10.5 Checker 相关字段

包含：

- `checker_type`
- `checker_config`
- 超时设置
- 重试策略
- 是否支持 `putflag`
- 是否支持 `getflag`
- 是否支持 `havoc`

### 10.6 Flag 相关字段

包含：

- `flag_mode`
- 注入位置或注入协议
- 校验方式
- 上一轮兼容策略

### 10.7 防守相关字段

包含：

- `defense_entry_mode`
- SSH/VPN 接入方式
- patch 上传入口配置
- 重启动作模板
- 回滚策略

### 10.8 观测与恢复字段

包含：

- 可采集的日志点
- 可采集的流量入口
- 健康检查入口
- rebuild 策略
- rollback 策略

### 10.9 准入状态

必须包含：

- `readiness_status`
- `readiness_report`
- 最近验证时间
- 最近验证人

## 11. AWD 比赛模型

### 11.1 核心对象

AWD 比赛至少包含：

- `AWDContest`
- `ContestTeam`
- `ContestAWDService`
- `AWDRound`

### 11.2 AWDContest

负责：

- 比赛窗口
- 比赛模式
- 轮次参数
- 队伍规则
- 冻结榜规则
- 比赛状态流转

状态建议固定为：

- `draft`
- `ready`
- `running`
- `paused`
- `frozen`
- `ended`

### 11.3 ContestAWDService

表示某场比赛中实际使用的 service。

来源：

- 从 `AWDServiceTemplate` 派生

负责：

- service 排序
- 比赛内名称覆盖
- 分值参数覆盖
- 可见性
- 轮次级裁判参数覆盖

### 11.4 ContestTeam

负责：

- 队伍名称
- 成员
- 队伍邀请码
- 比赛总分
- 最近得分时间

### 11.5 AWDRound

负责：

- 轮次编号
- 开始结束时间
- 状态
- 本轮 attack / defense / SLA 参数

### 11.6 AWDContest 状态机

比赛状态流转建议固定为：

- `draft -> ready -> running -> frozen -> ended`
- `running <-> paused`

约束如下：

- `draft`
  - 允许编辑比赛配置、队伍、service 和轮次参数
- `ready`
  - 表示 readiness 检查已通过，等待开赛
- `running`
  - 正常推进轮次、计分和事件流
- `paused`
  - 暂停轮次推进与官方计分，保留现场态势快照
- `frozen`
  - 比赛仍继续运行，但公开榜停止更新
- `ended`
  - 停止轮次推进，允许复盘、导出和裁判复算

### 11.7 比赛进度模型

为了支撑选手端、管理员端和大屏统一展示比赛进度，`AWDContest` 应维护：

- `current_round`
- `total_rounds`
- `round_progress_percent`
- `contest_progress_percent`
- `time_to_next_round`
- `freeze_countdown`
- `remaining_attack_window`
- `remaining_contest_time`

这些字段统一由服务端计算，前端只消费，不自行推导官方比赛进度。

- `up/down/compromised`

### 12.6 ServiceAlert

表示告警对象。

包含：

### 12.7 AdminOperationAudit

表示管理员在 AWD 赛场内执行的一次官方干预。

### 12.8 TeamDefenseAction

表示队伍在平台内执行的一次受控防守动作。

## 13. 比赛运行模型

### 13.1 基本单位

真正被裁判系统判断的不是“某队解了一题”，而是：

- 某一轮里
- 某队的某个 service
- 是否存活
- 是否被攻破
- 是否守住

### 13.2 一轮固定执行顺序

每轮建议按以下顺序推进：

1. 创建或切换当前轮
2. 为每个 `team + service` 生成当前轮 flag
3. 注入 flag 到 service
4. 执行 checker
5. 开放本轮攻击提交
6. 记录攻击结果并更新受害方状态
7. 汇总本轮三段分
8. 推送榜单与事件流
9. 轮次结束并进入下一轮

### 13.3 轮次驱动原则

- 所有赛时状态必须挂在轮次上
- 所有计分和裁判结果必须可回放到轮次
- 前端不能自行拼装官方状态

## 14. Flag 生命周期设计

### 14.1 基本原则

每个 `team + service + round` 对应一个唯一 flag。

它必须满足：

- 当前轮唯一
- 可验证
- 可短窗口兼容上一轮
- 可注入 service
- 可追溯

### 14.2 注入方式

平台应支持至少两类注入：

- 文件注入
- 协议级或 checker 写入

以覆盖：

- Web
- Pwn
- 多容器综合靶场

### 14.3 有效窗口

建议支持：

- 当前轮有效
- 可配置的上一轮宽限窗口

## 15. Checker 模型

### 15.1 三动作模型

统一 checker 语义为：

- `putflag`
- `getflag`
- `havoc`

### 15.2 动作语义

`putflag`

- 证明 service 能接收当前轮 flag

`getflag`

- 证明 service 能正确返回当前轮 flag

`havoc`

- 证明业务链路仍然可用，不只是入口 200

### 15.3 状态判定

- `up`
  - 关键动作全部成功
- `down`
  - 不可达、超时、关键动作失败
- `compromised`
  - service 在线，但 flag 不正确
  - 或已被成功攻击写回失陷态

### 15.4 Checker 类型

最终方案至少支持：

- `http_standard`
- `tcp_script`
- `custom_script`

## 16. 安全边界与网络设计

### 16.1 四条硬规则

1. 每队有独立赛场网络
2. 队伍之间只能访问允许的公开攻击面
3. 平台控制面与比赛流量面隔离
4. 管理员入口与选手作战入口分离

### 16.2 三层网络面

#### 控制面

给平台自己使用：

- 轮次调度
- flag 注入
- checker 执行
- 容器/拓扑编排

#### 服务面

给队伍相互攻击使用：

- Web 公开入口
- Pwn TCP/UDP 端口
- 多容器题公开 service

#### 作战接入面

给选手防守自己使用：

- VPN
- SSH
- 平台受控 patch / restart 入口

### 16.3 攻击边界

攻击只允许通过：

- 对方 service 的公开暴露面

不允许：

- 通过平台后门直进对手容器
- 通过控制面链路攻击对方

### 16.4 防守边界

防守只允许通过：

- 本队 VPN / SSH
- 本队受控平台入口

不允许：

- 全局开放容器终端
- 跨队进入他人防守入口

## 17. 计分与裁判机制

### 17.1 官方裁判原则

- 平台是唯一官方裁判
- 分数只基于平台认可的事件
- 所有分数必须可回放、可复算、可解释

### 17.2 三段分

最终总分固定为：

- `sla_score`
- `defense_score`
- `attack_score`

总分：

- `total_score = sla_score + defense_score + attack_score`

### 17.3 SLA 分

若本轮 service 满足 checker 要求，则获得 SLA 分；否则为 0。

### 17.4 Defense 分

本轮未被成功打穿则保留；一旦本轮被成功攻击，防守分归零。

### 17.5 Attack 分

攻击方提交目标 service 当前轮有效 flag，且是本轮首次成功提交时获得攻击分。

### 17.6 去重规则

同轮、同攻击方、同受害方、同 service：

- 首次成功计分
- 后续成功仅记日志，不重复加分

### 17.7 冻结榜

支持：

- 配置冻结时间
- 冻结后真实榜继续计算
- 公开展示使用冻结榜
- 管理员看到真实榜

### 17.8 异常干预

允许管理员执行：

- 重跑 checker
- 重注 flag
- 重建实例
- 排行榜重算
- 暂停 / 恢复比赛

但必须：

- 全部留痕
- 记录原因
- 可赛后复盘

## 18. 选手端设计

### 18.1 总体定位

选手端是“战场工作台”，不是重型在线开发环境。

### 18.2 主导航

建议包含：

- 战场总览
- 我的服务
- 目标目录
- 攻击记录
- 队伍协作

### 18.3 主流程

选手主流程固定为：

1. 入场准备
2. 攻击流程
3. 防守流程
4. 协作记录
5. 轮次结束反馈

### 18.4 入场准备

包含：

- 加入队伍
- 进入战场
- 查看当前轮
- 确认本队 service 是否就绪
- 获取 VPN / SSH / patch 入口

### 18.5 攻击流程

包含：

- 选择 service
- 浏览目标队伍目录
- 使用本地工具攻击公开服务
- 获取 flag
- 在平台提交 stolen flag
- 收到命中反馈

### 18.6 防守流程

包含：

- 接收告警 / 被打反馈
- 查看失败原因、流量摘要、最近攻击
- 通过 SSH/VPN 修补
- 在平台执行 patch / restart / rollback
- 等待 checker 恢复

### 18.7 协作流程

平台内只做轻量协作：

- 谁修哪题
- 谁在攻击哪队
- 当前补丁版本
- 是否已恢复

不做重聊天或重工单系统。

## 28. 安全与公平性原则

最终方案建议在答辩中固定强调以下 9 条原则：

1. AWD 与 Jeopardy 在业务上彻底分池
2. AWD 的核心对象是 service，不是题目
3. 平台采用混合模式，不强行把深度攻防搬进浏览器
4. 攻击只允许通过目标 service 暴露面发生
5. 防守只允许通过本队受控接入面发生
6. 平台是唯一官方裁判
7. 得分只基于平台认可的事件
8. 所有管理员干预必须留痕
9. 比赛数据最终沉淀为教学资产

### 28.1 关键取舍说明

本方案明确采用以下取舍：

- 不把 AWD 题目与 Jeopardy 题目混库
  - 因为两者对象模型、运行方式和裁判逻辑完全不同
- 不让平台提供进入对手容器的后门入口
  - 因为这会破坏 AWD 的真实攻击边界，也会破坏公平性
- 不把深度攻防全部搬进浏览器
  - 因为 Web IDE 难以覆盖 Pwn、多容器和本地工具链需求
- 不追求公开赛级超大规模架构
  - 因为毕业设计和校内场景更重视闭环完整性而不是极限扩展性

### 28.2 主要实施风险

实际落地时重点关注以下风险：

- checker 误报或漏报
  - 需要通过 readiness、自测样例和赛中重跑机制降低误判
- 动态 flag 注入失败
  - 需要保留注入日志、重注入口和上一轮宽限窗口
- 多容器综合题编排复杂
  - 需要限制一期支持的拓扑复杂度，避免过早做重编排
- 流量与事件量过大
  - 需要采用时间桶聚合、缓存快照和明细按需回源
- 管理员干预过多影响比赛公信力
  - 需要强制留痕，并在赛后报告中完整展示干预原因和结果

## 29. 验收标准

完成后，平台应满足：

1. AWD 题库与 Jeopardy 题库彻底分离
2. 平台可以配置并运行一场多轮次 AWD 比赛
3. 平台可对 `Web / Pwn / 多容器` 三类 service 执行统一裁判流程
4. 平台可执行 flag 轮换、checker 检查、攻击提交流程、三段分计分
5. 平台提供选手战场工作台、管理员控制台和比赛大屏
6. 平台支持流量摘要、告警、轮次状态和排行榜实时展示
7. 平台支持赛后复盘与完整导出
8. 所有核心动作都可留痕并可赛后解释

## 30. 实施优先级建议

若后续进入实现阶段，建议按以下顺序推进：

1. 先落地 AWD 题库与比赛配置模型
2. 再落地轮次、flag、checker、计分内核
3. 再做选手战场工作台
4. 再做管理员控制台
5. 再做大屏、告警和导出

这样可以先保证裁判闭环正确，再逐步补齐展示与教学能力。

## 31. 最终答辩口径

建议答辩时使用以下统一表述：

本系统将 Jeopardy 与 AWD 两类赛制在业务上彻底分离，其中 AWD 以独立的 Service 模板库为基础，通过轮次调度、动态 Flag 注入、标准化 checker、攻击提交判定和三段分计分机制构建完整的攻防对抗闭环。平台负责比赛编排、实时监控、告警处置、排行榜更新和赛后复盘，选手则通过平台工作台获取态势信息和连接入口，并结合 VPN/SSH 与本地工具完成真实攻防操作。该设计既满足校内训练与竞赛的可实现性，也满足毕业设计对团队对抗、流量监控、实时排行榜和教学复盘的要求。
