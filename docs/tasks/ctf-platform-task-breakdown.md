## 第一期：核心基础（MVP）

### 阶段 A：项目骨架与基础设施（T1 ~ T4）

目标：搭建可运行的项目框架，后续所有任务在此基础上开发。

#### T1 — 后端项目骨架初始化

| 字段 | 说明 |
|------|------|
| 依赖 | 无 |
| 需求映射 | 基础设施 |
| 优先级 | Must |

**交付物：**
- Go 项目初始化（go mod、目录结构）
- 分层目录：`cmd/` `internal/handler/` `internal/service/` `internal/repository/` `internal/model/` `internal/middleware/` `pkg/`
- 配置管理（Viper / YAML 配置文件加载）
- 日志框架集成（Zap / Zerolog）
- 统一错误响应结构体
- Makefile / 基础构建脚本
- `.gitignore`、`docker-compose.dev.yml`（开发环境 PostgreSQL + Redis）

**验收标准：**
- `make run` 可启动 HTTP 服务，返回健康检查 200
- 配置文件可切换 dev/prod 环境
- 日志输出格式化且包含请求 ID

#### T3 — 用户认证与 RBAC 权限

| 字段 | 说明 |
|------|------|
| 依赖 | T2 |
| 需求映射 | F0.1、F0.2 |
| 优先级 | Must |

**交付物：**
- 用户注册接口（学号、姓名、班级、密码）
- 用户登录接口（JWT 签发，Access Token + Refresh Token）
- 密码加密存储（bcrypt）
- 三种角色：学员（student）、教师（teacher）、管理员（admin）
- RBAC 中间件：基于角色的接口访问控制
- 登录失败锁定（连续 N 次失败锁定 M 分钟，N/M 可配置）
- CAS 统一认证接口预留（interface 定义，不实现对接）

**验收标准：**
- 注册 → 登录 → 获取 Token → 携带 Token 访问受保护接口，全流程通过
- 学员无法访问管理员接口，返回 403
- 登录失败 5 次后账户锁定 15 分钟（默认值）
- Token 过期后返回 401，Refresh Token 可续签

### 阶段 B：Docker 容器管理引擎（T5 ~ T7）

目标：实现容器全生命周期管理，为靶场和竞赛提供底层容器能力。

#### T5 — Docker 容器生命周期管理

| 字段 | 说明 |
|------|------|
| 依赖 | T1 |
| 需求映射 | F2.2、F2.3 |
| 优先级 | Must |

**交付物：**
- Docker SDK（Go）封装层：创建、启动、停止、删除容器
- 容器状态查询（running / exited / created）
- 容器最大存活时间管理：到期自动销毁（后台定时扫描 + 回收）
- 延时机制：学员可手动延时，延时次数上限可配置
- 单用户并发实例数限制（默认 3 个，可配置）
- 容器启动超时处理（30s 超时返回失败）

**验收标准：**
- API 调用后容器在 30s 内启动并返回访问地址
- 超时容器被自动回收，数据库状态同步更新
- 同一用户超过并发上限时拒绝创建，返回明确提示

---

#### T6 — 容器资源限制与安全隔离

| 字段 | 说明 |
|------|------|
| 依赖 | T5 |
| 需求映射 | F1.8、安全需求 |
| 优先级 | Must |

**交付物：**
- 容器资源限制：CPU（--cpus）、内存（--memory）、PID 数（--pids-limit）、磁盘 IO
- 禁止特权模式（`--privileged=false` 强制）
- Linux Capabilities 白名单（Drop ALL，仅保留必要项）
- seccomp profile 配置
- 容器运行用户限制（非 root）

**验收标准：**
- 容器内执行 Fork Bomb 不影响宿主机和其他容器
- 容器内存超限后被 OOM Kill，不影响平台
- 容器无法访问宿主机网络和 Docker Socket

---

#### T7 — 容器网络隔离

| 字段 | 说明 |
|------|------|
| 依赖 | T5 |
| 需求映射 | 安全需求（网络隔离） |
| 优先级 | Must |

**交付物：**
- 每个用户实例创建独立 Docker Network
- 容器仅暴露必要端口到宿主机（动态端口映射）
- 不同用户实例间网络不可达
- 容器禁止访问宿主机管理接口（Docker API、SSH 等）
- 网络清理：实例销毁时同步清理对应 Network

**验收标准：**
- 用户 A 的靶机实例无法 ping 通用户 B 的靶机实例
- 容器内无法访问宿主机 172.17.0.1 和 Docker API 端口
- 实例销毁后 `docker network ls` 中无残留网络

### 阶段 C：靶场管理 — 基础（T8 ~ T11）

目标：管理员可以创建、配置、发布靶机，学员可以浏览可用靶场。

#### T8 — 靶机镜像管理

| 字段 | 说明 |
|------|------|
| 依赖 | T5 |
| 需求映射 | F1.1 |
| 优先级 | Must |

**交付物：**
- 镜像管理 CRUD 接口（名称、描述、版本标签、来源类型）
- 支持三种镜像来源：本地上传 tar 包、从 Registry 拉取、从 Dockerfile 构建
- 镜像构建状态追踪（pending / building / success / failed）
- 构建失败时返回构建日志
- 重复镜像名称冲突检测
- 镜像列表分页查询

**验收标准：**
- 上传 tar 包后镜像出现在 `docker images` 和平台列表中
- Dockerfile 构建失败时可查看错误日志
- 同名镜像上传提示冲突

---

#### T9 — 靶机 CRUD 与难度分级

| 字段 | 说明 |
|------|------|
| 依赖 | T8 |
| 需求映射 | F1.2、F1.3 |
| 优先级 | Must |

**交付物：**
- 靶机 CRUD 接口（名称、描述、关联镜像、漏洞类型标签、难度等级、分值、状态）
- 难度等级枚举：入门 / 简单 / 中等 / 困难 / 地狱
- 难度与基础分值的映射关系（可配置）
- 靶机状态管理：草稿 → 已发布 → 已下线
- 发布前校验：必须关联镜像、至少一个漏洞标签、已配置 Flag
- 靶机列表分页 + 按难度/漏洞类型筛选

**验收标准：**
- 创建靶机并发布，学员端可见
- 未配置 Flag 的靶机无法发布，返回校验错误
- 按难度筛选返回正确结果

---

#### T10 — Flag 管理（静态 + 动态）

| 字段 | 说明 |
|------|------|
| 依赖 | T9 |
| 需求映射 | F1.5 |
| 优先级 | Must |

**交付物：**
- 静态 Flag 配置：管理员手动设置固定 Flag 字符串
- 动态 Flag 生成：基于用户 ID + 靶机 ID + 盐值的 HMAC 算法，每个实例唯一
- Flag 注入机制：容器启动时通过环境变量或挂载文件注入 Flag
- Flag 验证接口（供 T13 调用）
- Flag 格式校验（统一前缀如 `flag{...}`）

**验收标准：**
- 静态 Flag 配置后，提交正确字符串验证通过
- 动态 Flag：同一靶机不同用户实例的 Flag 不同
- 动态 Flag 不可通过算法逆向推导（HMAC + 随机盐）

### 阶段 D：攻防演练 — 核心（T12 ~ T15）

目标：学员可以浏览靶场、启动靶机、提交 Flag、获得实时计分。

#### T12 — 靶场浏览与搜索

|------|------|
| 依赖 | T9 |

| 优先级 | Must |

**交付物：**
- 学员端靶场列表接口：分页、按难度/漏洞类型/关键词筛选排序
- 返回字段：名称、描述、难度、分值、完成人数、当前用户是否已完成
- 靶场详情接口：完整描述、提示数量（不展示内容）、资源要求

**验收标准：**

#### T13 — 靶机实例启动与 Flag 提交验证

| 字段 | 说明 |
|------|------|
| 依赖 | T5、T6、T7、T10 |
| 需求映射 | F2.2、F2.4 |
| 优先级 | Must |

**交付物：**
- "开始挑战"接口：创建容器实例、注入 Flag、返回访问地址和剩余时间
- 实例状态查询接口（运行中 / 已销毁 / 启动失败）
- 手动销毁接口
- Flag 提交接口：校验正确性、记录提交历史（时间、提交内容、结果）
- 防暴力破解：1 分钟内错误提交 > 20 次，锁定提交 5 分钟（阈值可配置）
- 重复提交已通过 Flag 返回"已完成"，不重复计分
- 提交记录持久化（用于后续技能评估）

**验收标准：**
- 点击开始挑战后 30s 内返回靶机访问地址
- 正确 Flag 提交后即时加分，错误 Flag 提示失败并记录
- 暴力破解触发频率限制，返回锁定提示和剩余等待时间

#### T14 — 实时计分与个人得分

| 字段 | 说明 |
|------|------|
| 依赖 | T13 |
| 需求映射 | F2.7 |
| 优先级 | Must |

- 并发提交不会导致重复计分（幂等保证）

### 阶段 E：竞赛管理 — Jeopardy 模式（T16 ~ T18）

目标：支撑 Jeopardy 模式校赛，包含竞赛创建、组队、排行榜。

#### T16 — 竞赛创建与团队管理

| 字段 | 说明 |
|------|------|

| 优先级 | Must |

**交付物：**

**验收标准：**

#### T17 — 实时排行榜与动态计分

| 字段 | 说明 |
|------|------|
| 依赖 | T14、T16 |
| 需求映射 | F3.4、F3.5 |
| 优先级 | Must |

**交付物：**
- 排行榜服务：基于 Redis Sorted Set 实现实时排名
- 支持个人排行和团队排行两种视图
- 动态计分：题目分值随解出人数递减（公式可配置，如 `max(min_score, base_score × decay_factor ^ solve_count)`）
- 分值变更时同步更新所有已解出该题的队伍得分
- 榜单冻结：管理员可配置冻结时间点，冻结后前端停止更新，赛后揭榜
- WebSocket 推送排行榜变更事件

**验收标准：**
- 得分变更后排行榜 5s 内刷新
- 动态计分公式生效，解出人数增加时分值递减
- 冻结时间点后排行榜前端不再更新，赛后揭榜数据正确

## 第二期：进阶功能

### 阶段 F：靶场管理 — 进阶（T19 ~ T22）

#### T19 — 自定义网络拓扑编排

| 字段 | 说明 |
|------|------|
| 依赖 | T7、T9 |
| 需求映射 | F1.4 |
| 优先级 | Should |

**交付物：**
- 多容器拓扑定义模型（JSONB 存储拓扑结构）
- 拓扑编排接口：添加节点、配置节点间网络连接（可达/不可达）

#### T21 — 靶机分类与标签体系
