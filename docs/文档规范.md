# CTF 项目文档规范

## 定位

本文档是 CTF 项目的文档写作和归属规范。新增或修改文档时，先按本文判断文档类型、目录归属、必需结构和验证方式，再进入具体内容编写。

规范依据以下公开实践收敛：

- GitHub README / CONTRIBUTING 约定：入口文档用于说明项目用途、上手路径、贡献预期，并应放在仓库可发现的位置。
- GitHub Docs 内容仓库：大型文档需要目录索引、元数据、相对链接和自动化检查来保持可维护。
- Google developer documentation style guide：优先遵循项目内风格；没有项目内规则时，再参考通用技术写作规范。
- Diátaxis：按读者需求区分教程、操作指南、参考资料和解释性文档，避免不同目的的内容混写。
- MADR / ADR：重要架构决策应记录背景、决策和后果，不能只保留结论。

## 基本原则

- 文档是事实源，不是临时草稿堆放区。
- 一个文档只承担一种主要用途：最终事实、实现方案、评审证据、操作手册、外部参考、反馈记录或过程资料。
- 入口文档负责导航，不承载长篇细则；细则进入对应事实源。
- 旧文档不能与新事实源长期冲突。结论迁移后，旧文档必须标注 `Superseded by ...`，或从索引中移除活动入口。
- 文档变更要和代码、脚本、测试、架构边界保持同步；不能只更新说明而不更新实际 guardrail。

## 目录归属

- `docs/architecture/`：当前架构与最终设计事实源。
- `docs/contracts/`：API、OpenAPI、事件、题包格式、导入导出结构等契约。
- `docs/design/`：仍在推演的中间设计稿、设计索引和过期说明。
- `docs/plan/impl-plan/`：结构性实现方案、阶段计划、执行清单和验证步骤。
- `docs/reviews/`：代码、架构、UI、流程 review 记录和 findings。
- `docs/requirements/`：需求基线、范围说明、差距分析和立项约束。
- `docs/tasks/`：任务分解、拆解清单和阶段性工作列表。
- `docs/todos/`：尚未收口的 backlog、延期项和后续跟踪。
- `docs/operations/`：运维手册、联调步骤、演练记录和运行侧说明。
- `docs/reports/`：阶段性报告、汇总结论、差距报告和分析输出。
- `docs/Q&A/`：会被重复引用的问答式说明。
- `feedback/`：agent 工作流、review、prompt、policy 和协作方式相关反馈。
- `references/`：外部文章、仓库、工具和研究资料索引。
- `prompts/`：已验证提示词、工作流和可复用 prompt 资产。
- `works/`：可复用模板、地图、教程和 good practices。
- `practice/`：实验记录、迁移过程、历史计划索引和实践说明。
- `thinking/`：尚未落地、但需要保存的判断、质疑和取舍分析。
- `concepts/`：项目级长期概念、原则和 harness 定义。

不再新增 `docs/improvements/`、`docs/superpowers/`、`docs/refs/`、`docs/skills/` 作为活动入口。

## 文档类型

### 文档状态字段

适用：`docs/architecture/`、`docs/contracts/`、`docs/design/` 中会被 agent 读取并参与架构判断的文档。

推荐在文档标题下方保留状态行：

```md
> 状态：Current
> 事实源：代码 / 契约 / 已落地功能
> 替代：无
```

状态含义：

- `Current`：当前事实，可直接作为架构图、实现和 review 的依据。
- `Draft`：仍在推演，只能作为设计候选；不得画成当前架构事实。
- `Superseded`：已被替代，必须写明 `Superseded by ...` 或 `替代：...`。
- `Archive`：仅保留历史价值，不作为活动入口。

约束：

- `docs/architecture/` 中的活动事实文档不得长期保留 `状态：Draft` 或 `状态：初稿`。
- `docs/design/` 中的 Draft 被采用后，稳定结论必须回收到 `docs/architecture/` 或 `docs/contracts/`。
- 索引文件列出的活动入口必须优先指向 `Current` 文档。

### 架构事实

适用：模块边界、调用关系、页面最终设计、设计系统、长期运行模型。

必需结构：

- `定位`：这份文档覆盖什么，不覆盖什么。
- `当前设计`：已经采用的事实。
- `边界`：谁负责什么，谁不能依赖谁。
- `接口或数据影响`：涉及 API、事件、数据库、路由或状态时必须写明。
- `Guardrail`：对应测试、脚本、hook 或人工 review 检查点。
- `历史迁移`：如果替代旧文档，写 `Superseded by ...` 或链接旧入口。

`当前设计` 写作硬规则：

- 每个组件都要写“负责 / 不负责”。
- 每条主流程都要能对应代码路径或 API。
- 每个状态都要写触发条件和退出条件。
- 每个副作用都要写失败后的处理。
- 每个外部依赖都要写“谁调用、失败如何处理、是否重试”。
- 不能出现没有落点的词，例如“统一处理”“自动同步”“平台负责”“系统管理”；如果必须使用，必须紧跟代码路径、API、数据结构或测试依据。
- 如果写了“支持”，必须说明入口、数据结构、状态变化和测试。
- 如果不知道，就写 `待确认`，不能假装已经确定。

`当前设计` 推荐结构：

1. `运行入口`：HTTP API、前端路由、命令入口、后台任务或脚本入口。
2. `主要组件`：逐项写负责 / 不负责。
3. `主流程`：按步骤写清调用链，每一步标代码路径或 API。
4. `状态与不变量`：写状态枚举、触发条件、退出条件和禁止状态。
5. `数据与副作用`：写表、缓存、文件、容器、registry、通知、事件等副作用及失败处理。
6. `外部依赖`：写调用方、失败处理、是否重试、超时或幂等策略。
7. `错误与回退`：写失败点、回滚、补偿、清理和用户可见结果。
8. `兼容与历史例外`：写仍保留的旧路径、allowlist 和已废弃设计。
9. `已知限制`：写未实现、只本地验证、依赖人工操作或待确认内容。

`当前设计` 质量检查：

- `scripts/check-docs-consistency.py` 会检查本次改动过的 `docs/architecture/**/*.md`，以及已经写了 `## 当前设计` 的架构文档。
- 非索引、非页面参考稿的架构文档一旦被修改，必须包含 `## 当前设计`；否则说明它不应该作为当前架构事实放在 `docs/architecture/`。
- `## 当前设计` 必须同时出现 `负责` 与 `不负责`，并且至少包含代码路径、API、契约文档、测试或 Guardrail 之一作为依据。
- `统一处理`、`自动同步`、`平台负责`、`系统管理` 这类词如果出现在 `## 当前设计`，同段必须给出代码路径、API、数据结构或测试依据。
- `支持` 这类能力声明如果出现在 `## 当前设计`，同段必须说明入口、数据结构、状态变化和测试；无法确认时写 `待确认`。
- `待落地`、`TODO`、`后续计划`、`计划支持` 不能写成当前事实；确实未完成时放入 `已知限制` 或写 `待确认`。
- 这层检查是防弱化的底线，不替代人工 review；人工 review 仍要检查外部依赖、失败处理、重试、回滚、兼容和测试是否真的完整。

### 架构文档规范化流程

适用：把存量 `docs/architecture/` 文档、索引、页面参考稿或设计系统资料改到当前规范。

默认做法：

1. 先改入口，不先大批量改正文。
   - 优先处理 `docs/architecture/README.md`、`docs/architecture/frontend/` 入口、`docs/architecture/backend/` 入口、`docs/architecture/features/专题架构索引.md`、`docs/design/README.md`。
   - 入口必须说明哪些是当前事实、哪些是参考稿、哪些是历史草稿、哪些已被替代。
2. 对存量文档先标状态，再移动或删除。
   - `Current`：当前事实，保留并按“架构事实”结构补齐。
   - `Draft`：仍在推演，进入 `docs/design/`。
   - `Superseded`：已被替代，文首标注 `Superseded by ...`。
   - `Archive`：只保留历史价值，不作为活动入口。
   - `Delete candidate`：无引用、过期且内容已被当前事实源覆盖，才能进入删除候选。
3. 不做“一次性格式化全部架构文档”的大提交。
   - 按主题切片，例如前端架构、题包 / Registry、后端模块边界、业务专题架构。
   - 每个提交只解决一个可 review 的事实源问题。
4. 稳定结论必须回收到事实源。
   - 架构结论回 `docs/architecture/`。
   - 契约结论回 `docs/contracts/`。
   - 过程方案留在 `docs/plan/impl-plan/`。
   - review 结论留在 `docs/reviews/`，不直接覆盖事实源。
5. 修改入口或删除旧文档时，必须同步机械检查。
   - 检查 README / 索引不引用已删除路径。
   - 检查专题索引中列出的文件真实存在。
   - 检查设计索引中列出的活动设计稿真实存在。
   - 能写进 `scripts/check-consistency.sh` 的约束，不只写在提示词里。

推荐提交切片：

- `文档(架构): 规范架构文档入口`
- `文档(前端): 规范前端架构事实源`
- `文档(题包): 对齐题包导入架构文档`
- `工程(harness): 增加架构文档一致性检查`

### 架构图生成规范

适用：把当前架构文档、契约文档、代码和测试转成 Mermaid、PlantUML、C4、draw.io 或图片前的结构化图表说明。

图表存放位置：

- 单篇文档专属图表源码优先与文档同目录同名前缀存放，例如 `题包Registry交付架构.diagram.mmd`。
- 跨专题图表源码放到 owning 目录的 `diagrams/` 子目录，例如 `docs/architecture/features/diagrams/`。
- 渲染图片只作为辅助产物；必须同时保留 Mermaid / PlantUML / C4-PlantUML 等可 review 文本源。
- 图片不能单独作为事实源。事实变更必须先更新对应架构文档和图表源码，再重新渲染图片。

默认输出：

- 优先使用 Mermaid。
  - `flowchart`：系统边界、容器关系、组件依赖、数据流。
  - `sequenceDiagram`：题包导入、实例启动、Flag 提交、通知推送、后台任务等流程。
  - `stateDiagram-v2`：实例生命周期、导入状态、构建任务状态。
  - `classDiagram` 只用于稳定领域模型，不用于临时代码结构。
- 需要 C4 分层表达时，先写 Mermaid 或 C4-PlantUML 源码，再渲染图片；不要只提交不可 review 的图片。
- 复杂图可以配一段“图表输入说明”，但图片本身不得替代事实源文档。

图表类型选择：

- 系统上下文图：用户、CTF 平台、外部服务、学校身份系统、镜像仓库、Docker runtime 等系统级关系。
- 容器图：前端、后端、PostgreSQL、Redis、registry、runtime engine、checker runner 等运行单元。
- 组件图：单个模块内部的 handler / application service / repository / adapter / background task。
- 时序图：跨模块调用顺序、事务边界、异步任务、回调和错误分支。
- 数据流图：事件、缓存、读模型、报表聚合、通知分发、导入产物。
- 部署图：本地开发、测试、生产、AWD runtime 或 registry 拓扑。

事实来源优先级：

1. 当前代码、测试、配置和脚本。
2. `docs/architecture/` 与 `docs/contracts/` 的当前事实。
3. `docs/operations/` 中已执行过的运行说明。
4. `docs/reviews/` 只能作为历史证据，不能覆盖当前事实。
5. `docs/design/` 只能作为 Draft 来源，图中必须标注 `待落地` 或 `Draft`，不得画成已落地事实。

图中每个节点必须满足至少一项：

- 能对应到一个代码模块、组件、表、配置、脚本、外部服务或文档事实。
- 能对应到一个明确业务角色或系统边界。
- 能对应到一个稳定状态，例如 `pending / building / ready / failed`。

图中每条边必须满足至少一项：

- 函数调用、HTTP API、WebSocket、数据库读写、缓存读写、事件、文件产物、容器挂载、网络访问或部署依赖。
- 文档中已明确的依赖方向、数据流或生命周期转换。
- 测试中已经覆盖的调用链或边界约束。

禁止事项：

- 不得把实现计划画成当前架构事实。
- 不得把 review finding、历史草稿或旧路径画成当前主链路。
- 不得为了图好看新增不存在的队列、网关、服务、缓存、领域层或中间件。
- 不得把一个函数级细节图伪装成系统架构图。
- 不得只提交 PNG / JPG；除非用户明确要求，必须同时保留可 review 的文本源。
- 不得在图表源码或外发给模型的输入里包含真实密钥、token、cookie、生产密码、内部访问凭据或可直接复用的敏感 URL。
- 内网地址、生产域名、registry 凭据和数据库连接串默认用占位符，例如 `<registry-host>`、`<database-url>`、`<token>`。

重要架构文档推荐补充 `图表视角` 小节：

```md
## 图表视角

推荐图表：

- 容器图
- 时序图

必须包含：

- ChallengeImportService
- ImageBuildService
- Registry

禁止包含：

- 未落地的异步队列方案

主要边：

- PreviewImport -> 保存预览目录
- CommitImport -> 持久化 build source
- ImageBuildService -> 创建 build job
```

生成后的检查清单：

- Mermaid / PlantUML 源码能渲染。
- 图中节点能回到事实源、代码、配置或外部服务。
- 图中边有调用、数据流、依赖、状态转换或部署依据。
- 没有引用已删除路径。
- 没有把 `docs/design/` 的 Draft 画成 Current。
- 没有遗漏文档中已经声明的 Guardrail。
- 图的粒度和标题一致：系统图不塞函数细节，组件图不替代部署图。
- 不包含真实敏感字段；必要时用占位符替代。

### 代码路径映射规范

适用：文档、图表、prompt 输入包和 review 记录中引用代码、配置、脚本和文档路径。

默认使用仓库根相对路径：

- 前端代码：`code/frontend/src/...`
- 后端代码：`code/backend/internal/...`
- 后端命令入口：`code/backend/cmd/...`
- 脚本：`scripts/...`
- 题包：`challenges/...`
- 文档：`docs/...`
- Harness 资料：`feedback/`、`prompts/`、`references/`、`works/`、`practice/`、`thinking/`、`concepts/`

约束：

- 不再新增 `frontend/src/...`、`backend/internal/...`、`api/admin.ts` 这类缺少仓库根上下文的活动引用。
- 同一文档内路径风格要一致；旧路径迁移时优先同步父级 README 或专题索引。
- 图表节点的依据字段应引用仓库根相对路径，便于 review 和脚本检查。

### 实现方案

适用：结构性改动、跨模块改动、职责拆分、迁移、已知结构债收口。

必需结构：

- `目标`
- `非目标`
- `输入与现状`
- `阶段切片`
- `兼容性与回退`
- `验证计划`
- `Review 重点`
- `完成判定`

实现方案不覆盖架构事实。方案落地后，稳定结论要回收到 `docs/architecture/` 或 `docs/contracts/`。

### Review 记录

适用：代码、架构、UI、流程的审查结论。

必需结构：

- `Review 对象`
- `结论`
- `Findings`
- `必须重跑的验证`
- `残余风险`

Review 是评审证据，不是当前设计入口。若 review 改变架构结论，必须同步更新事实源。

### 外部参考

适用：GitHub 仓库、文章、工具、论文、官方文档。

必需结构：

- 链接
- 一句话说明
- 与本项目的关联
- 已采用的项目规则或明确未采用的原因

外部资料只做索引和摘要，不复制长篇正文。

### 反馈记录

适用：agent 工作流、review、prompt、policy、协作方式的踩坑与修正规则。

必需结构：

- `问题描述`
- `原因分析`
- `解决方案`
- `收获`

业务待办不写入 `feedback/`。

## 命名规则

- 架构、设计和项目规范文档默认使用中文文件名。
- 计划、review、反馈记录默认使用日期前缀：`YYYY-MM-DD-简述.md`。
- 外部契约、工具配置或必须与英文生态对齐的文档可以使用英文文件名。
- 禁止新增 `misc.md`、`notes.md`、`new.md`、`temp.md` 这类无法表达归属的文件名。
- 同一主题只能有一个活动入口；拆分子文档时，父级 README 必须给出读取顺序。

## 写作风格

- 默认中文，代码、命令、路径、协议字段、报错保持原样。
- 先给结论，再给依据和关键细节。
- 用短句和列表表达规则；只有严格对齐明显提升可读性时才使用表格。
- 每个标题下都要有实际信息，避免空标题。
- 文档中的链接优先使用相对路径，方便本地阅读和分支内跳转。
- 可见 UI 文案只写给终端用户看的内容；实现说明、设计意图、布局介绍不得写进页面文案。

## 更新流程

1. 判断文档类型和目录归属。
2. 检查同主题是否已有活动入口。
3. 更新具体文档。
4. 如果新增入口或替代旧入口，同步更新父级 README 或 AGENTS 导航。
5. 如果形成可机械化约束，同步更新 `scripts/check-consistency.sh` 或对应架构检查。
6. 运行最小充分验证。

## 验证要求

文档规范、入口、harness 或外部参考变更后，至少运行：

```bash
bash scripts/check-consistency.sh
git diff --check -- <touched-files>
```

如果改动会影响架构边界，还要运行：

```bash
bash scripts/check-architecture.sh --full
```

如果只修改独立说明文档，且不改变入口、规则或 guardrail，可只运行 `git diff --check`。
