## 基本信息

| 字段 | 说明 |

## 需求概述

构建一个基于 Docker 的校园级 CTF 网络攻防靶场平台，为信息安全专业提供教学实训和竞赛环境。平台核心价值：**快速生成隔离的漏洞靶机环境，支撑日常训练和校赛选拔，同时通过数据驱动的技能评估辅助教学改进**。

关键技术特征：Docker 容器化靶机、网络隔离、实时计分、自动化环境生命周期管理。

### 模块一：靶场管理

| 编号 | 功能点 | 优先级 | 描述 | 验收标准 |
|------|--------|--------|------|----------|
| F1.1 | 靶机镜像管理 | Must | 管理员上传/导入 Docker 镜像，支持从 Dockerfile 构建或从 Registry 拉取 | 镜像上传后可在列表中查看，支持版本标签管理 |
| F1.2 | 漏洞环境配置 | Must | 配置漏洞类型（SQL 注入、XSS、权限绕过、文件上传、命令注入等），绑定到具体靶机镜像 | 每个靶机至少关联一个漏洞类型标签，支持多标签 |
| F1.3 | 难度分级 | Must | 靶机按难度分级（入门/简单/中等/困难/地狱），影响计分权重 | 每个靶机有且仅有一个难度等级 |
| F1.4 | 自定义网络拓扑 | Should | 管理员可编排多容器组成的网络拓扑（如 Web→DB→内网渗透链路） | 支持至少 3 层网络拓扑编排，容器间网络可达性可配置 |
| F1.5 | Flag 管理 | Must | 为每个靶机配置静态 Flag 或动态 Flag 生成规则（每个用户实例唯一） | 动态 Flag 在容器启动时注入，不同用户实例的 Flag 不同 |
| F1.6 | 靶机分类与标签 | Should | 按漏洞类型、技术栈、知识点对靶机分类，支持标签筛选 | 支持多维度筛选和搜索 |
| F1.7 | 环境模板 | Could | 将常用的多容器拓扑保存为模板，一键复用 | 模板可创建、编辑、删除，从模板创建靶场 ≤ 30s |
| F1.8 | 靶机资源限制配置 | Must | 为每个靶机配置 CPU、内存、磁盘、网络带宽上限 | 容器实际资源使用不超过配置上限 |
| F1.9 | Writeup 管理 | Should | 管理员为靶机配置官方解题思路（Writeup），可设置赛后公开 | Writeup 在竞赛期间不可见，赛后可按配置自动公开 |

## 非功能性需求

### 性能

| 指标 | 要求 | 说明 |
|------|------|------|
| 容器启动时间 | ≤ 30s | 从用户点击到靶机可访问 |
| 并发容器数 | ≥ 200 | 校赛高峰期预估（按 200 人同时在线，每人 1 个实例） |
| Flag 提交响应 | ≤ 500ms | 提交到返回结果 |
| 排行榜刷新 | ≤ 5s | 得分变更到排行榜更新 |
| 页面加载 | ≤ 2s | 首屏加载时间（非容器启动） |
| 报告导出 | ≤ 30s（单人）/ 异步（班级） | 大批量导出走异步任务 |

### 安全（重点）

CTF 靶场平台的安全需求比一般 Web 应用更高——平台本身承载的就是漏洞环境，必须确保靶机漏洞不会影响宿主机和平台本身。

| 维度 | 要求 |
|------|------|
| 容器隔离 | 靶机容器必须运行在隔离的 Docker 网络中，禁止访问宿主机网络和管理接口 |
| 容器逃逸防护 | 禁止特权容器（`--privileged`），限制 Linux Capabilities，启用 seccomp/AppArmor |
| 网络隔离 | 每个用户实例独立网络命名空间，不同用户的靶机实例间网络不可达 |
| 资源限制 | 每个容器强制 CPU/内存/磁盘/PID 数限制，防止资源耗尽攻击（Fork Bomb 等） |
| Flag 安全 | 动态 Flag 不可通过容器间通信获取；Flag 生成算法不可被逆向推导 |
| 平台认证 | 对接学校统一身份认证（CAS）；用户密码加密存储（bcrypt），支持登录失败锁定，Session/JWT 安全管理 |
| 网络访问控制 | 平台仅限校园网内访问（白名单限制），校外网络不可达；校园网对接暂不实现，先占位预留接口 |
| 输入校验 | 平台自身接口做严格输入校验，防止 SQL 注入、XSS（靶机内部的漏洞不算） |
| 审计日志 | 关键操作（登录、Flag 提交、竞赛配置变更、容器创建/销毁）全量记录 |

### 可用性

|------|------|
| 降级策略 | Docker 资源不足时，排队等待而非直接报错；非核心功能（推荐、画像）不可用不影响核心解题流程 |

## 约束与依赖

### 技术约束

| 约束 | 说明 | 影响 |
|------|------|------|
| Docker 单机部署 | 学校物理机部署（台数待确认），非 K8s 集群 | 并发容器数受单机资源限制，需做好资源规划 |
| 无 CI/CD 基础设施 | 目前没有 CI/CD 流水线，需从零搭建或手动部署 | 初期可手动部署，后续考虑引入轻量 CI/CD（如 Drone、Gitea Actions） |
| 网络隔离复杂度 | Docker 网络模型对多层拓扑支持有限 | 复杂拓扑可能需要 docker-compose + 自定义网桥，增加编排复杂度 |
| 镜像存储空间 | 漏洞靶机镜像体积大（单个 200MB~2GB） | 需规划镜像仓库存储容量，考虑镜像分层复用 |
| 容器启动延迟 | 冷启动（首次拉取镜像）耗时较长 | 需预拉取常用镜像，或使用镜像缓存策略 |
| AWD 模式网络要求 | 团队间需要可控的网络互通 | 需要精细的 iptables/网桥规则，实现"可攻击但隔离"的网络拓扑 |

### 外部依赖

| 依赖 | 说明 | 风险 |
|------|------|------|
| Docker Engine | 容器运行时，核心基础设施 | 版本兼容性、安全补丁需持续跟进 |
| 漏洞靶机镜像 | 需从零构建，目前无现成镜像；可参考社区开源资源（DVWA、Vulhub 等）但需自行适配 | 镜像构建工作量较大，建议优先构建常见漏洞类型（SQL 注入、XSS、文件上传等），逐步扩充 |
| 校园网络环境 | 学员通过校园网访问平台；校园网有白名单限制，校外网络无法访问；校园网对接暂不实现，先占位 | 后续对接时需与学校网络中心协调端口放行和白名单配置 |
| 服务器资源 | 学校提供的物理机（台数和配置待确认） | 资源上限决定并发容器数，需提前确认配置 |
| 学校统一身份认证 | CAS 认证；一期先占位预留接口，暂不实现对接 | 后续对接时需获取 CAS 接口文档和测试环境 |

## 风险评估

### 🔴 高风险

| 编号 | 风险描述 | 影响范围 | 缓解措施 |
|------|----------|----------|----------|
| R-H1 | **容器逃逸** — 学员通过靶机漏洞逃逸到宿主机 | 整个平台和服务器安全 | 禁用 `--privileged`；Drop 所有非必要 Capabilities；启用 seccomp/AppArmor profile；考虑 gVisor/Kata 等沙箱运行时 |
| R-H2 | **资源耗尽攻击** — 学员在靶机内执行 Fork Bomb、挖矿、大量写磁盘 | 影响同服务器其他用户的靶机和平台本身 | 强制 cgroup 限制（CPU/内存/PID/磁盘 IO）；设置容器最大存活时间；监控异常资源使用并自动 kill |
| R-H3 | **网络隔离失败** — 不同用户靶机实例间网络可达 | 竞赛公平性、Flag 泄露 | 每个用户实例使用独立 Docker Network；AWD 模式下精细控制 iptables 规则；定期审计网络隔离策略 |
| R-H4 | **Flag 泄露/作弊** — 学员共享 Flag 或通过非预期方式获取 Flag | 竞赛公平性 | 动态 Flag（每个实例唯一）；Flag 提交时间/IP 关联分析；短时间内相同 Flag 跨队提交告警 |

### 🟡 中风险

| 编号 | 风险描述 | 影响范围 | 缓解措施 |
|------|----------|----------|----------|
| R-M1 | **竞赛高峰并发** — 开赛瞬间大量用户同时启动容器 | 容器启动延迟、部分用户无法开始 | 预热机制（赛前预创建容器池）；排队系统；限制单用户并发实例数 |
| R-M2 | **靶机镜像质量** — 社区镜像存在后门或非预期漏洞 | 平台安全、教学质量 | 镜像入库前安全扫描（Trivy/Clair）；管理员审核机制；隔离测试环境 |
| R-M3 | **AWD 模式复杂度** — 攻防对抗的网络编排和计分逻辑复杂 | 开发周期、竞赛稳定性 | AWD 作为第二期功能，优先实现 Jeopardy 模式；充分测试后再上线 |
| R-M4 | **数据一致性** — 并发 Flag 提交导致计分错误 | 竞赛公平性 | Flag 验证和计分操作加分布式锁或数据库乐观锁；幂等设计 |

## 建议的开发拆分

### 第一期：核心基础（MVP）

1. **基础框架搭建** — 项目骨架、数据库设计、用户认证（RBAC）、基础 API 框架
2. **Docker 容器管理引擎** — 容器生命周期管理（创建/启动/停止/销毁）、资源限制、网络隔离
3. **靶场管理（基础）** — 镜像管理、靶机 CRUD、难度分级、Flag 配置（静态+动态）
4. **攻防演练（核心）** — 靶场浏览、实例启动、Flag 提交验证、实时计分
5. **竞赛管理（Jeopardy 模式）** — 竞赛创建、组队、排行榜、动态计分、榜单冻结

### 第二期：进阶功能

6. **靶场管理（进阶）** — 自定义网络拓扑编排、环境模板、Writeup 管理、靶机分类标签
7. **攻防演练（增强）** — 提示系统（Hint）、攻击过程记录、解题进度追踪
8. **竞赛管理（AWD 模式）** — 团队攻防对抗、攻击流量监控、防守检测与计分、作弊检测
9. **竞赛辅助** — 竞赛公告推送、赛后数据导出

### 第三期：数据驱动

10. **技能评估（核心）** — 能力画像生成（雷达图）、薄弱项识别、靶场推荐
11. **技能评估（报告）** — 个人/班级实训报告导出、进步趋势追踪
12. **管理增强** — 系统仪表盘、操作审计日志、通知系统、学习路径规划

### 拆分原则

- 第一期交付后即可支撑日常训练和 Jeopardy 模式校赛，具备独立使用价值
- 第二期补齐 AWD 对抗和进阶体验，适合在首次校赛反馈后迭代
- 第三期聚焦数据价值，需要积累足够的实训数据后才有意义
- 每期内部的任务可进一步拆分为独立可提交的最小单元，由 leader 细化
