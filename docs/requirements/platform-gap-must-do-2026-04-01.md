# 校内 CTF 平台对标主流平台必须补齐项

## 背景

当前平台已经具备题目导入、靶机实例、比赛与队伍、AWD、通知、能力画像、推荐与报告等基础能力，且在教学化闭环上已经超过很多通用 CTF 平台。

下一阶段不再追求“再补几个基础页面”，而是聚焦补齐两类核心短板：

1. 平台级赛事标准能力
2. 攻防教学证据链与复盘闭环

## 必须做

### 1. 攻防证据链与教学复盘闭环

- 目标：把当前零散的 `challenge_detail_view`、`instance_access`、`instance_proxy_request`、`submit`、`hint_unlock` 等事件，升级为教师可查看、可筛选、可复盘的攻击证据链。
- 当前基础：
  - 后端已记录实例访问与平台代理请求审计。
  - 学员时间线已展示“利用轨迹”。
  - 教师侧已有学员分析页与学员时间线接口。
- 必须补齐：
  - 证据链聚合视图
  - 攻击会话/阶段化摘要
  - 教师侧复盘入口与证据面板
  - 事件元数据标准化

### 2. 选手 Writeup 提交、评阅与归档

- 目标：补齐“选手提交解题报告 -> 教师评阅 -> 纳入教学复盘”的闭环。
- 当前基础：
  - 平台已有管理员/题目作者维护题解的能力。
  - 平台已有报告、能力画像、教师学员分析页。
- 必须补齐：
  - 学员提交题目 writeup
  - 教师/管理员评阅与状态流转
  - 与题目、比赛、学员分析关联
  - 后续教学复盘和优秀案例沉淀入口

### 3. 判题模式补强

- 目标：把当前以 `static` / `dynamic` 为主的判题模型，补到能覆盖毕设与主流平台常见场景的层级。
- 当前基础：
  - 文档与前端契约里已有 `regex`
  - 实际主流程主要支持 `static` / `dynamic`
- 必须补齐：
  - `regex` 判题完整落地
  - 一个非纯 Flag 文本型模式，优先 `manual_review`
  - 题目导入、后台配置、自检、查询侧一致性

### 4. 赛事资产导出与归档能力

- 目标：补齐平台级运营能力，至少让赛事结果、班级报告、关键审计/复盘数据可导出归档。
- 当前基础：
  - 已有比赛、队伍、公告、冻结榜、AWD。
  - 已有个人/班级报告导出能力。
- 必须补齐：
  - 比赛结果导出
  - 复盘证据或 writeup 归档导出
  - 导出结构稳定，可作为后续备份/迁移基础

## 实施顺序

1. 攻防证据链与教学复盘闭环
2. 选手 Writeup 提交、评阅与归档
3. 判题模式补强
4. 赛事运营补强

## 范围控制

- 不做插件市场级 challenge type framework
- 不做大规模容器编排/多集群调度
- 不做完整 CTFtime 双向同步
- 不做重度游戏化叙事系统
