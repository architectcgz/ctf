## 1. 背景与问题

教师复盘需要看到“学生何时开始实操、发起了哪些关键请求、最终是否形成有效提交或后续复盘材料”。原始事实分散在 `audit_logs`、`submissions`、`submission_writeups` 和 AWD 相关表里，直接把这些表暴露给教师端会带来两个问题：

- 教师页面需要自己拼接多张表，前端展示口径很难稳定。
- 证据字段若不先做结构化与脱敏，后续复盘、归档和导出都会反复写兜底逻辑。

当前需要收口的是：教师侧攻击证据的最终事实源不是某张新表，也不是原始审计日志，而是统一的证据链读模型。

## 2. 架构结论

- 教师侧事件级证据事实源统一为 `GET /api/v1/teacher/students/:id/evidence`。
- 证据事件由 `teaching_query` 在查询时聚合生成，不新增独立攻击证据持久化表。
- 事件语义、`detail` 生成和 `meta` 脱敏规则统一归属 `internal/teaching/evidence`。
- 教师分析页和学生复盘归档都使用同一套证据事件语义，但可以通过不同读接口组织展示。
- 当前证据链只保留安全截断后的结构化元数据，不承担完整流量包、完整请求体或录屏存储。

## 3. 模块边界与职责

### 3.1 模块清单

- `internal/teaching/evidence`
  - 负责：统一事件类型、阶段、来源、详情文案和安全元数据格式
  - 不负责：直接做权限控制或页面展示

- `internal/module/teaching_query/infrastructure`
  - 负责：从 `audit_logs`、`submissions`、`submission_writeups`、`awd_attack_logs`、`awd_traffic_events` 读取原始事实并映射为证据事件
  - 不负责：把原始表结构直接暴露给 HTTP 或前端

- `internal/module/teaching_query/application/queries`
  - 负责：教师/管理员访问控制、查询过滤、分页和 `summary` 汇总
  - 不负责：发明新的底层攻击事实

- `internal/module/assessment/infrastructure/report_repository`
  - 负责：在复盘归档与导出场景下复用同一套证据事件语义
  - 不负责：替代教师读模型接口成为新的事件 owner

- `frontend/src/features/teacher-student-analysis` 与 `frontend/src/widgets/teacher-student-review-workspace`
  - 负责：按题目筛选、展示事件列表、构造教师观察点
  - 不负责：自行拼装原始审计日志

### 3.2 事实源与所有权

- 原始业务事实源：
  - `audit_logs`
  - `submissions`
  - `submission_writeups`
  - `awd_attack_logs`
  - `awd_traffic_events`
- 教师事件级证据事实源：`TeacherEvidenceResp`
- 事件字段和脱敏口径事实源：`internal/teaching/evidence`

## 4. 关键模型与不变量

### 4.1 核心查询与响应

- `TeacherEvidenceQuery`
  - `challenge_id`
  - `contest_id`
  - `round_id`
  - `event_type`
  - `from`
  - `to`
  - `limit`
  - `offset`

- `TeacherEvidenceResp`
  - `summary.total_events`
  - `summary.proxy_request_count`
  - `summary.submit_count`
  - `summary.success_count`
  - `summary.challenge_id`
  - `events[]`

### 4.2 事件类型

当前统一事件类型包括：

- `instance_access`
- `instance_proxy_request`
- `challenge_submission`
- `manual_review`
- `writeup`
- `awd_attack_submission`
- `awd_traffic`

### 4.3 不变量

- 证据事件按时间升序组织，过滤与分页都发生在读路径上。
- `summary.submit_count` 只统计 `challenge_submission` 与 `awd_attack_submission`。
- `summary.success_count` 只根据 `meta.is_correct` 或 `meta.is_success` 推导。
- `instance_proxy_request`、`awd_traffic` 等事件的 `meta` 只能保存安全截断后的方法、路径、状态码、预览信息，不能回流完整敏感载荷。
- 新增证据能力时，应优先扩展统一事件语义，而不是让前端直接读取新的原始事实表。

## 5. 关键链路

### 5.1 教师分析页读链路

1. 教师学生分析页通过 `getStudentEvidence` 请求 `/api/v1/teacher/students/:id/evidence`。
2. `teaching_query` handler 绑定筛选参数并解析当前用户身份。
3. query service 校验教师是否可以访问该学员，再构造标准化 `TeacherEvidenceQuery`。
4. repository 从多张事实表读取记录，并通过 `internal/teaching/evidence` 生成统一事件。
5. query service 计算 `summary`，前端再基于 `events[]` 构造题目筛选项与观察条。

### 5.2 复盘归档链路

1. 学生复盘归档读接口或导出接口进入 `assessment.ReportService`。
2. report repository 复用同一套事件语义读取 `timeline/evidence/writeups/manual_reviews`。
3. 归档页面和导出文件据此生成 `teacher_observations`、证据面板和反思区块。

## 6. 接口与契约

### 6.1 对外接口

- `GET /api/v1/teacher/students/:id/evidence`

支持的主要查询参数：

- `challenge_id`
- `contest_id`
- `round_id`
- `event_type`
- `from`
- `to`
- `limit`
- `offset`

### 6.2 事件元数据契约

按事件类型返回结构化 `meta`，当前稳定口径包括：

- 代理请求事件：`method`、`target_path`、`status_code`、`payload_preview`
- 提交事件：`is_correct`、`points`
- 人工审核事件：`review_status`、`score`
- 题解事件：`submission_status`、`visibility_status`、`is_recommended`
- AWD 事件：`contest_id`、`round_id`、`service_id`、`victim_team_id`、`is_success`

## 7. 兼容与迁移

- 学生时间线仍然保留，但它是叙事型时间线，不再承担教师复盘的主事实源职责。
- 攻击会话接口 `attack-sessions` 是基于证据事件派生出的阅读视图，不是并列的底层事实源。
- 若后续需要更重的流量取证能力，应在统一事件语义之外增加外部证据引用，而不是把完整敏感内容写回 `meta`。

## 9. 验证标准

- 教师/管理员可以通过 `/api/v1/teacher/students/:id/evidence` 读取可访问学员的证据链。
- 证据链能稳定聚合访问、代理请求、提交、人工审核、题解和 AWD 相关事件。
- 前端按题目筛选时，`summary` 与 `events[]` 口径保持一致。
- 事件 `meta` 只暴露结构化预览，不泄漏完整敏感载荷。
