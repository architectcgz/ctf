# 专题架构索引

## 定位

`docs/architecture/features/` 只收录面向产品能力或业务专题的最终架构事实。

- 这里的文档应回答边界、主身份、事实源、关键链路、兼容约束和验证标准。
- 如果某个专题已经形成稳定结论，应直接写成“当前系统现在是什么”；不要额外保留“怎么从旧方案迁过来”的收口稿。
- 这里不再长期保存大段 `方案比较`、`页面与交互设计`、`视觉说明`、`实施顺序` 或任务 checklist。
- 如果专题同时涉及页面稿、中间设计或实施过程，应拆到对应目录，`features/` 只保留最终架构结论。
- 单题题包设计、题面设计、解法设计不属于本目录；已落地题目以题包目录自身为事实源，仍在推演的单题方案进入 `docs/design/`。

## 文档结构要求

专题文档默认按以下结构组织：

1. 文档元信息
2. 背景与问题
3. 架构结论
4. 模块边界与职责
5. 关键模型与不变量
6. 关键链路
7. 接口与契约
8. 兼容与迁移
9. 代码落点
10. 验证标准

允许按专题特点裁掉不适用的小节，但不能回到“设计过程 + 页面稿 + 实施步骤”混写。

## 拆分规则

- 页面结构、交互形态、视觉方向：进入 `docs/architecture/frontend/pages/` 或 `docs/design/`
- 单题题包、题面、解法与提示设计：回到对应 `challenges/...` 目录；若仍属中间方案，则进入 `docs/design/`
- OpenAPI、事件、字段结构、导入导出格式：进入 `docs/contracts/`
- 实施阶段、任务切片、验证命令：进入 `docs/plan/impl-plan/`
- review 结论与 findings：进入 `docs/reviews/`
- 历史过程追溯：进入 `practice/` 或通过 Git 历史查看
