# 竞赛题目编排工作台设计

## 目标

把当前“只能创建竞赛、不能完整编排竞赛题目”的后台流程，升级成一套可直接用于普通赛和 AWD 赛的竞赛工作台。

本轮重点解决的问题：

- `ContestEdit` 目前只有基础设置和轻量挂题，管理员无法完成一场比赛从选题到可开赛的完整编排
- AWD 相关能力已经存在，但入口分散在“题目编排”和 “AWD 运维” 两套视图里，流程断裂
- 管理员无法在同一页面判断“哪些题已经入赛”“哪些题已完成 AWD 配置”“哪些阻塞当前赛事开赛”
- 当前挂题只能维护 `challenge_id / points / order / is_visible`，不足以支撑对抗赛

## 非目标

- 不改现有 `contest_challenges` 的核心数据模型
- 不新增第二个“AWD 专用题目池”
- 不把本轮设计扩展成通用赛事模板系统
- 不重写现有 AWD 轮次、巡检、攻击日志和态势能力
- 不在本轮处理赛题模板跨赛事复用、批量导入包、复杂拖拽交互的全部细节

## 当前背景

现有代码已经具备这些基础：

- `ContestEdit.vue` 已经提供竞赛基础信息编辑页
- `ContestChallengeOrchestrationPanel.vue` 已经支持基础挂题：
  - 关联题目
  - 顺序
  - 分值
  - 可见性
- `contest_challenges` 后端 DTO 已经支持 AWD 字段：
  - `awd_checker_type`
  - `awd_checker_config`
  - `awd_sla_score`
  - `awd_defense_score`
  - `awd_checker_preview_token`
- `AWDOperationsPanel.vue` 已经具备运行态和服务配置能力：
  - `轮次态势`
  - `题目配置`
- `AWDReadiness` 已经具备 readiness 汇总、阻塞判断和强制放行机制

问题不在于“后端没有 AWD 能力”，而在于这些能力没有被组织成一条完整的竞赛编排链路。

## 竞品参考结论

参考常见 CTF 平台的管理方式，可以抽象出两个稳定规律：

- 普通 Jeopardy 平台通常都会在竞赛工作区里完成 challenge assignment，不会让管理员只停在“创建比赛”
- AWD / Attack-Defense 平台在挂题之后，通常还会继续补一层 service config、checker、赛前检查和轮次运行，不会把这些能力留成互相跳转的孤岛

因此，这里真正缺的是“竞赛工作台”，而不是“再加一个简单挂题弹窗”。

## 方案比较

### 方案 A：单工作台，按运营阶段拆成 5 段

做法：

- 在 `ContestEdit` 内形成完整工作台
- 按管理员真实操作链路组织为：
  - `基础信息`
  - `题目池`
  - `AWD 配置`
  - `赛前检查`
  - `轮次运行`
- 普通赛只展示前 2 到 3 段，AWD 赛事展示完整链路

优点：

- 最符合管理员的真实流程
- 最容易复用当前 `ContestEdit / AWDOperationsPanel / readiness gate`
- 能把“挂题”和“可开赛”之间的缺口补完整
- 数据中心仍然是现有 `contest_challenges`

缺点：

- 页面结构会明显变复杂，需要控制信息密度
- 需要重新梳理 `ContestEdit` 和 `ContestManage -> AWD 运维` 的职责边界

推荐采用。

### 方案 B：左侧题目目录，右侧单题配置

做法：

- 以题目为中心搭一个高密度配置台
- 左侧始终是竞赛题目列表
- 右侧承载单题基础信息和 AWD 配置详情

优点：

- 适合高频逐题配置
- 单题编辑效率高

缺点：

- 不擅长表达“这场比赛现在整体是否能开”
- 赛前检查和运行态承载不自然
- 对首次挂题、批量管理和整体准备度不够直观

本轮不采用。

### 方案 C：看板式状态推进

做法：

- 按“待关联 / 已编排 / 待补 AWD / 已就绪”做状态看板

优点：

- 进度感强
- 阻塞项一眼可见

缺点：

- 不适合复杂 Checker 表单
- 与现有 `AWDOperationsPanel` 的复用关系差
- 对运行态支撑弱

本轮不采用。

## 决策

采用方案 A。

竞赛编排工作台按单工作台组织，不再新增第二个 AWD 题目池，而是使用“统一竞赛题目池 + AWD 配置视图 + 赛前检查 + 轮次运行”的链路完成整场比赛的管理。

## 核心设计

### 1. 工作台结构

`ContestEdit` 升级为完整的竞赛工作台，阶段结构如下：

1. `基础信息`
2. `题目池`
3. `AWD 配置`
4. `赛前检查`
5. `轮次运行`

模式差异：

- `mode=jeopardy`
  - 默认展示：
    - `基础信息`
    - `题目池`
  - 若后续引入更完整的普通赛 readiness，可再扩展 `赛前检查`
- `mode=awd`
  - 展示完整 5 段

约束：

- `AWD 配置`、`赛前检查`、`轮次运行` 只在 AWD 赛事出现
- 工作台顶部需要有统一状态摘要，明确当前比赛处于：
  - 编排中
  - 待开赛
  - 运行中
  - 已结束

### 2. 统一题目池

竞赛只有一个题目池，它负责定义“这场比赛到底用了哪些题”。

题目池承载两类信息：

- 基础编排字段：
  - `title`
  - `category`
  - `difficulty`
  - `points`
  - `order`
  - `is_visible`
- AWD 摘要字段（仅 AWD 赛事显示）：
  - `awd_checker_type`
  - `awd_checker_validation_state`
  - `awd_sla_score`
  - `awd_defense_score`
  - `awd_checker_last_preview_at`

题目池不直接承载复杂 Checker 表单，只承担：

- 关联入赛
- 顺序与分值调整
- 可见性管理
- 准备度筛选
- 批量编排操作

建议提供的筛选与视图：

- `全部题目`
- `未配置 AWD`
- `预检失败`
- `待重新验证`
- `已就绪`

建议提供的批量操作：

- 批量关联
- 批量移除
- 批量设置可见性
- 批量调整分值

### 3. 不新增第二个 AWD 题目池

不采用“竞赛题目池 + AWD 专用题目池”双池模型。

原因：

- `contest_challenges` 已经同时承载基础编排和 AWD 字段，双池会和现有数据模型冲突
- 同一题在两个池中维护，容易出现顺序、分值、可见性和配置状态漂移
- 管理员会混淆“已入赛”和“已进入 AWD 池”两个概念

正确的关系应当是：

- `题目池` 负责“题目属于这场比赛”
- `AWD 配置` 负责“这道入赛题是否已经成为可运行的服务题”

如果未来需要跨赛事复用 AWD 服务方案，应设计“服务模板库”，而不是第二个题目池。

### 4. AWD 配置层

`AWD 配置` 是题目池的深度编辑层，不负责再次选题。

职责：

- 对已入赛题目补齐 `checker + service score + preview`
- 提供逐题的完整配置与验证
- 让管理员连续处理“待补配置”和“预检失败”的题

推荐交互：

- 主列表沿用题目池里的竞赛题清单
- 点击某题后打开抽屉或详情区
- 在详情区编辑：
  - `awd_checker_type`
  - `awd_checker_config`
  - `awd_sla_score`
  - `awd_defense_score`
  - `awd_checker_preview_token`
- 操作按钮至少包含：
  - `保存`
  - `保存并试跑`
  - `上一题`
  - `下一题`

边界：

- `points / order / is_visible` 仍由题目池维护
- AWD 配置页不重复编辑这些字段，避免两个地方都能改同一件事

### 5. 赛前检查

`赛前检查` 是正式开赛门禁，不只是一个提示页。

职责：

- 汇总整场比赛 readiness
- 告诉管理员当前是否可以开赛
- 把所有阻塞项映射回具体题目和具体动作
- 承接现有的 override 强制开赛能力

页面建议分 3 块：

- `总体准备度`
  - `可开赛 / 可强制开赛 / 不可开赛`
  - 题目总数
  - 已通过数
  - 未配置数
  - 预检失败数
  - 待重新验证数
- `阻塞项列表`
  - 题目名
  - 阻塞原因
  - 建议动作
  - 快捷入口
- `开赛动作区`
  - `修复后开赛`
  - `填写原因强制开赛`
  - `返回题目池继续编排`

关键要求：

- 所有阻塞项都能直接跳回对应题目的 AWD 配置
- 赛事从 `draft/registering` 切到 `running` 前，优先进入此阶段
- 仍然复用现有 readiness gate 与 override 审计链路

### 6. 轮次运行

`轮次运行` 作为工作台最后一段，承接现有 `AWDOperationsPanel` 的运行态能力。

设计原则：

- 开赛前可见但弱化，只展示“尚未进入运行阶段”
- 开赛后成为默认焦点，承接：
  - 轮次态势
  - 即时巡检
  - 服务状态
  - 攻击日志
  - 榜单与冻结态

约束：

- 未开赛时，影响赛时数据的操作默认禁用
- 已开赛后，仍可查看前面阶段，但会影响赛时行为的字段应谨慎加锁或提示风险

## 组件与代码映射

### 现有组件复用建议

- `ContestEdit.vue`
  - 升级成完整工作台容器
- `ContestChallengeOrchestrationPanel.vue`
  - 演进成新的 `题目池` 视图
- `AWDChallengeConfigPanel.vue`
  - 演进成工作台里的 `AWD 配置` 视图
- `AWDReadinessSummary.vue`
  - 演进成 `赛前检查` 的摘要区
- `AWDReadinessOverrideDialog.vue`
  - 继续承接强制开赛原因输入
- `AWDOperationsPanel.vue`
  - 作为 `轮次运行` 段的主体

### 状态中心

不新增新的题目实体，继续以 `contest challenge` 为中心。

前端状态拆分建议：

- 竞赛基础信息
- 题目池列表与筛选
- 当前选中题目的 AWD 配置上下文
- readiness 摘要与阻塞项
- 运行态数据

避免把所有状态继续塞进单一 `useAdminContestAWD` 里，建议按工作台阶段拆分 composable。

## 数据流

### 编排阶段

1. 管理员在 `题目池` 关联题目
2. 前端调用 `createAdminContestChallenge`
3. 列表刷新后展示基础编排状态
4. AWD 赛事额外展示 readiness 摘要字段

### 配置阶段

1. 管理员从题目池或 AWD 配置列表选中某题
2. 前端读取该 `contest challenge` 的 AWD 字段
3. 保存时调用 `updateAdminContestChallenge`
4. 保存并试跑后刷新校验状态与最近一次结果

### 开赛阶段

1. 管理员进入 `赛前检查`
2. 前端读取 readiness 摘要
3. 若存在阻塞项：
  - 默认阻止开赛
  - 允许填写原因后强制放行
4. 开赛成功后，把工作台默认焦点切到 `轮次运行`

## 错误处理

### 题目池

- 关联失败时保留当前筛选与滚动位置
- 批量操作失败时需要返回逐项结果或最少返回失败原因摘要

### AWD 配置

- 保存失败不关闭详情面板
- 试跑失败与保存成功要分开展示，避免用户误以为“保存失败”

### 赛前检查

- readiness 请求失败时，要阻止开赛入口继续乐观执行
- 强制开赛失败时，保留管理员输入的 override reason

### 运行态

- 未开赛时，不展示误导性的“空运行结果”
- 切换到运行段但数据未加载完成时，明确区分“尚未开始”和“加载失败”

## 测试建议

### 组件测试

- `ContestEdit` 在普通赛和 AWD 赛下展示不同阶段
- 题目池正确显示基础字段与 AWD 摘要字段
- 从题目池进入 AWD 配置后，不会出现重复编辑字段
- 赛前检查能正确显示阻塞项并跳转到目标题
- 开赛后默认进入 `轮次运行`

### 集成测试

- 从创建 AWD 赛事到挂题、配置 Checker、通过 readiness、切到运行态的完整链路
- readiness 不通过时，开赛动作被阻止
- 使用 override reason 后允许强制开赛
- 已开赛赛事下，关键编排字段受限或给出风险提示

### 回归重点

- Jeopardy 赛事不能被 AWD 特有视图污染
- 现有 `ContestManage -> AWD 运维` 功能迁移后不丢能力
- 现有 `create/update/listAdminContestChallenges` 契约保持兼容

## 分期建议

### Phase 1：工作台结构重组

- `ContestEdit` 扩展为 5 段结构
- 把现有轻量挂题和 AWD 运维入口迁入工作台

### Phase 2：统一题目池

- 让题目池显示 AWD 摘要状态
- 增加筛选与批量操作

### Phase 3：AWD 配置深度编辑

- 把现有 `AWDChallengeConfigDialog` 接入工作台上下文
- 支持连续逐题保存与试跑

### Phase 4：赛前检查闭环

- 将 readiness 摘要升级成可操作的阻塞清单
- 把开赛动作统一收敛到这一段

### Phase 5：运行态承接

- 用工作台最后一段承接现有 `AWDOperationsPanel`
- 调整未开赛与已开赛的交互边界

## 推荐落地顺序

推荐从“结构重组 -> 统一题目池 -> 赛前检查 -> AWD 配置交互增强 -> 运行态承接”推进。

原因：

- 先统一工作台骨架，管理员才能感知“编排链路已经连上”
- 题目池和赛前检查最直接解决“为什么现在还不能开赛”
- AWD 配置与运行态可以在已有能力上渐进替换，不需要一开始大改

## 最小交付

第一轮最小可交付标准：

- `ContestEdit` 形成单工作台结构
- AWD 赛事可在同一工作台里完成：
  - 挂题
  - 查看题目级 AWD 摘要
  - 进入单题 AWD 配置
  - 查看赛前检查
  - 开赛后进入运行态
- 不新增第二个 AWD 题目池
- 现有 readiness gate 和 AWD 运维能力可继续复用

## 实施回链（2026-04-15）

本轮前端已经落地工作台的 5 个阶段：

- `基础信息`：保留赛事基础编辑，并把 AWD 开赛前的 readiness gate 接回保存链路
- `题目池`：升级为统一题目池，已支持 AWD 摘要列、筛选和跳转到下一阶段补配置
- `AWD 配置`：已接入独立阶段，支持按当前聚焦题连续编辑赛事题目的 checker 与分值配置
- `赛前检查`：已接入 readiness 摘要、阻塞清单、回跳 AWD 配置和强制开赛入口
- `轮次运行`：已承接现有 AWD 运维面板，未开赛时弱化显示，开赛后作为默认焦点

保留到后续 phase 的交互：

- 题目池已支持 AWD 创建时一次关联多个题目；复杂批量编辑和拖拽编排仍未接入
- `mode=jeopardy` 的 `赛前检查` 扩展仍未实现，当前完整 5 段仅对 AWD 赛事开放
- 已开赛后对前置阶段高风险字段的更强限制或风险提示仍待后续补齐
