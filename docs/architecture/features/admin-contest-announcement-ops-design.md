# 管理端竞赛公告运营设计

## 目标

为管理员补齐“按竞赛发布和删除公告”的前端运营能力，并和学生侧竞赛公告消费页对齐。

本轮设计要解决的问题：

- 学生侧 `/contests/:id?panel=announcements` 已经有公告只读展示，但管理端没有对应的竞赛公告运营能力
- 当前现有后台结构里，公告运营不应该塞回 `ContestEdit.vue` 的编辑流程
- 普通竞赛和 AWD 竞赛都应该支持竞赛公告
- 管理员需要两个入口：
  - 竞赛列表里快速发布
  - 单场竞赛工作台顶部进入完整公告管理页

## 非目标

- 不改学生侧公告页的阅读布局和信息结构
- 不给教师增加发布或删除竞赛公告的能力
- 不把竞赛公告和全站通知系统合并成一套抽象
- 不在本轮做公告置顶、已读状态、筛选标签或富文本编辑
- 不重做现有 AWD 运维页的信息架构

## 当前背景

当前系统里已经有三块相关能力，但没有真正闭环：

1. 学生侧竞赛公告消费
   - `ContestDetail.vue` 的 `announcements` tab 已支持公告只读展示
   - `useContestDetailPage.ts` 已负责拉取公告列表
   - `ContestAnnouncementRealtimeBridge.vue` 已支持公告实时刷新

2. 后端竞赛公告接口
   - 普通查询：`GET /api/v1/contests/:id/announcements`
   - 管理员查询：`GET /api/v1/admin/contests/:id/announcements`
   - 管理员发布：`POST /api/v1/admin/contests/:id/announcements`
   - 管理员删除：`DELETE /api/v1/admin/contests/:id/announcements/:aid`

3. 后台竞赛相关页面
   - `ContestManage.vue` / `ContestOrchestrationPage.vue` 承担竞赛目录
   - `ContestEdit.vue` 承担单场竞赛配置型工作台
   - `ContestOperations.vue` 是现有的单场 AWD 运维页

因此，本轮问题不是“缺少后端能力”，而是“缺少合理的管理端前端运营入口和交互闭环”。

## 决策摘要

采用以下结构：

- 学生侧 `/contests/:id?panel=announcements` 保持只读，不增加管理入口
- 只有管理员可以发布和删除竞赛公告
- 教师无发布、删除权限
- 普通竞赛和 AWD 竞赛都支持公告运营
- 已结束竞赛只允许查看历史公告，不允许发布或删除
- 管理端提供两个入口：
  - 竞赛列表“更多 -> 发布通知”打开右侧抽屉
  - 单场竞赛公告管理页用于完整运营

## 方案比较

### 方案 A：把公告运营放回 `ContestEdit.vue`

做法：

- 在 `ContestEdit.vue` 新增一个公告 stage
- 在编辑型工作台中直接发布和删除公告

优点：

- 路由最少
- 可直接复用单场竞赛上下文

缺点：

- “配置赛事”和“运营赛事”语义混在一起
- 比赛进行中管理员不应回到编辑页做公告操作
- 不符合当前已经明确的用户操作心智

不采用。

### 方案 B：把现有 `ContestOperations.vue` 升级成通用竞赛运维页

做法：

- 在当前 AWD 运维页内部加入公告面板
- 同时扩展普通竞赛进入该页

优点：

- 比赛进行中时，公告确实位于运维上下文

缺点：

- 会把当前 AWD 专用运维页强行扩成“所有竞赛通用”的大页
- 为了公告管理引入了不必要的页面职责重构
- 改动面大，和当前代码结构不匹配

本轮不采用。

### 方案 C：新增独立的管理员竞赛公告管理页，并保留竞赛列表快速抽屉

做法：

- 竞赛列表里提供“更多 -> 发布通知”，打开右侧抽屉
- 新增单场竞赛公告管理页
- 从单场竞赛工作台顶部跳转到该管理页

优点：

- 同时满足“快发”和“完整运营”
- 不破坏现有 AWD 运维页结构
- 不把运营动作塞进编辑页
- 普通竞赛和 AWD 竞赛都能统一复用

缺点：

- 需要增加一个新路由和一组管理组件

推荐采用。

## 信息架构

### 1. 学生侧

保留现状：

- `/contests/:id?panel=announcements`
- 只读展示竞赛公告
- 继续通过公告 websocket 刷新

学生侧不显示任何发布、删除、编辑入口。

### 2. 管理员侧入口一：竞赛列表快速发布

在竞赛目录列表每行的“更多”菜单下新增：

- `发布通知`

点击后行为：

- 在当前页右侧打开公告运营抽屉
- 抽屉上下文绑定当前竞赛
- 不跳离列表页

此入口定位为“快速发公告”。

### 3. 管理员侧入口二：单场竞赛公告管理页

新增路由：

- `/platform/contests/:id/announcements`

入口来源：

- `ContestEdit.vue` 顶部增加“公告”跳转按钮
- 竞赛列表抽屉中也可附带“进入完整管理页”入口

此页定位为“完整公告运营页”。

## 页面与交互设计

### 1. 竞赛列表右侧抽屉

抽屉分为两块：

- 发布表单区
- 最近公告列表区

#### 抽屉头部

显示：

- 竞赛标题
- 当前竞赛状态
- 一句状态说明

目的：

- 避免管理员在列表上下文里对错比赛发布公告

#### 表单字段

- 标题
  - 必填
  - 最大 200 字
- 正文
  - 选填
  - 最大 5000 字

按钮：

- `发布公告`
- `取消`

#### 最近公告列表

- 按创建时间倒序
- 每条显示标题、时间、正文摘要
- 可操作状态下，每条显示 `删除`
- 已结束竞赛时只读展示，不显示删除

#### 快速发布流程

- 点击 `发布公告`
- 按钮进入 loading，禁止重复提交
- 成功后：
  - 清空表单
  - 刷新列表
  - 新公告置顶
  - toast 提示“公告已发布”
- 失败后：
  - 保留输入内容
  - 在抽屉局部显示错误
  - 不进入全局 500 页面

#### 删除流程

- 点击删除先弹确认
- 确认文案：
  - `删除后学生端将不再显示该公告，此操作不可恢复。`
- 成功后从当前列表移除并 toast
- 失败后保留原列表并在局部提示错误

### 2. 单场竞赛公告管理页

该页不是抽屉，而是完整页面。

页面结构：

- 顶部：竞赛标题、状态信息、返回入口
- 中部：发布表单区
- 下部：完整公告列表

交互规则与抽屉一致，但适合连续运营：

- 发布成功后表单清空
- 列表立即刷新
- 删除前需要确认
- 错误在页面局部消费

### 3. 已结束竞赛只读态

状态为 `ended` 时：

- 竞赛列表“更多”里不显示 `发布通知`
- 管理员如果通过路由直接进入单场公告页，只显示历史公告
- 页面显示只读提示，例如：
  - `该竞赛已结束，公告已封存，不可再发布或删除。`
- 发布表单不渲染
- 删除按钮不渲染

## 权限与状态规则

### 1. 角色规则

- `admin`
  - 可进入公告管理抽屉和公告管理页
  - 可在允许状态下发布和删除公告
- `teacher`
  - 不显示发布、删除入口
- `student`
  - 仅学生侧消费页可见，只读

### 2. 竞赛状态规则

前端展示规则统一为：

- `draft / registering / running / frozen`
  - 可发布
  - 可删除
- `ended`
  - 只读查看
  - 不可发布
  - 不可删除

当前实现的已结束只读限制主要在前端 `useContestAnnouncementManagement` 中完成；后端路由仍由管理员权限兜底，尚未按竞赛状态拒绝已结束赛事的发布和删除请求。

## API 设计

管理端新增或补齐以下前端 API 封装：

- `getAdminContestAnnouncements(contestId)`
- `createAdminContestAnnouncement(contestId, payload)`
- `deleteAdminContestAnnouncement(contestId, announcementId)`

对应后端接口：

- `GET /api/v1/admin/contests/:id/announcements`
- `POST /api/v1/admin/contests/:id/announcements`
- `DELETE /api/v1/admin/contests/:id/announcements/:aid`

学生端现有：

- `getAnnouncements(contestId)`

继续只用于学生侧消费，不和管理员公告管理 API 混用。

### 1. 路由

新增管理员路由：

- `/platform/contests/:id/announcements`

元信息：

- 仅 `admin`
- 标题可命名为 `竞赛公告`
- 保持当前后台页面风格

### 2. 组件

当前已落地：

- `ContestAnnouncementManageDrawer.vue`
  - 用于竞赛列表快速发布
- `ContestAnnouncements.vue`
  - 用于单场完整公告管理
- `ContestAnnouncementsTopbarPanel.vue`
  - 用于单场公告页顶部竞赛状态和返回入口
- `ContestAnnouncementsWorkspacePanel.vue`
  - 用于单场公告页发布表单、只读提示和公告列表

### 3. Composable

当前已落地：

- `useContestAnnouncementManagement.ts`

统一管理：

- 公告列表加载
- 局部错误状态
- 发布表单状态
- 发布 loading
- 删除 loading
- ended 只读判断
- 刷新逻辑

### 4. 现有页面改动边界

最小改动面控制为：

  - 增加“发布通知”入口

  - 顶部增加“公告”跳转按钮

- 全站通知系统结构

## 数据流

### 1. 抽屉数据流

- 打开抽屉
- 读取当前竞赛信息与公告列表
- 发布成功后刷新当前竞赛公告列表
- 删除成功后刷新当前竞赛公告列表

### 2. 单场页数据流

- 进入路由后加载竞赛详情和管理员公告列表
- 发布和删除都使用同一套 composable 行为
- 页面只负责展示和触发，不重复持有业务规则

## 文案原则

页面对管理员只展示真实 UI 文案，不展示实现说明。

建议统一使用“公告”作为用户可见词，不混用“通知”“消息”“运营播报”等多套表述。

操作文案建议：

- 按钮：`发布公告`
- 菜单项：`发布通知`
- 页标题：`竞赛公告`
- 只读提示：`该竞赛已结束，公告已封存，不可再发布或删除。`

## 验收标准

完成后应满足：

- 管理员在竞赛列表“更多”菜单中可看到 `发布通知`
- 点击后右侧打开当前竞赛的公告运营抽屉
- 管理员可以发布竞赛公告，成功后立即在列表中看到新公告
- 管理员可以删除竞赛公告，成功后列表立即同步
- 普通竞赛和 AWD 竞赛都支持上述能力
- 已结束竞赛不显示快速发布入口
- 已结束竞赛的单场公告页为只读态
- 教师看不到公告管理入口
- 学生侧公告页仍保持只读消费
- 发布、删除、加载失败都不会跳转全局 500 页面

建议覆盖：

- 竞赛列表菜单项显示逻辑
  - `admin + 非 ended` 显示
  - `admin + ended` 不显示
  - `teacher` 不显示
- 抽屉打开后加载公告列表
- 发布成功后表单清空、列表刷新
- 发布失败后保留输入并显示局部错误
- 删除确认与删除成功后的列表更新
- ended 只读态不显示发布和删除操作
- 单场公告管理页的只读态与可操作态切换

### 2. 回归测试

建议补充：

- 学生侧 `ContestDetail.vue` 公告 tab 继续只读展示
- 公告 websocket 刷新不受本轮改动影响
- 竞赛列表现有“编辑 / 进入工作台 / AWD 赛区”操作不回归

## 风险与注意事项

### 1. ended 状态只做前端隐藏不够

如果后端没有同样规则，管理员仍可能绕过前端误改已结束竞赛公告。

因此需要后端最终兜底：

- ended 禁止发布
- ended 禁止删除

### 2. 抽屉与单场页共用一套逻辑时要避免双份状态机

建议把发布和删除都收口进同一个 composable，避免一边能 catch 错误、一边把异常冒到全局。

### 3. 列表页抽屉不能破坏原有筛选与分页状态

抽屉应是附加层，不应该影响竞赛列表当前的筛选、搜索或分页状态。

### 4. 删除后优先以服务端状态为准

删除成功后建议重新对齐当前竞赛公告列表，而不是只做局部 splice，避免并发时前端状态漂移。

## 推荐实施顺序

1. 增加管理员竞赛公告 API 封装
2. 实现 `useContestAnnouncementManagement`
3. 接入竞赛列表快速发布抽屉
4. 新增单场竞赛公告管理页
5. 在 `ContestEdit.vue` 顶部增加“公告”跳转按钮
6. 补齐角色、状态与异常回归测试
