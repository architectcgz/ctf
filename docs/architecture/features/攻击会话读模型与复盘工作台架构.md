# 攻击会话读模型与复盘工作台架构

## 文档元信息

- 状态：`implemented`
- 事实源级别：`final`
- 适用范围：`backend`、`frontend`、`contracts`
- 关联模块：
  - `internal/module/teaching_readmodel`
  - `internal/teaching/evidence`
  - `frontend/src/features/teacher-student-analysis`
  - `frontend/src/widgets/teacher-student-review-workspace`
- 最后更新：`2026-05-07`

## 1. 背景与问题

事件级证据链已经能回答“发生了什么”，但教师复盘还需要回答“这一轮尝试是怎样连续推进的”。如果继续让教师直接看散落事件，会遇到两个问题：

- 同一题目的多次尝试会混在一起，教师难以区分哪一段是一次独立攻击过程。
- 事件列表适合作为事实源，不适合作为复盘主阅读界面。

当前需要收口的是：攻击会话不是新的底层事实表，而是基于统一证据事件派生出的会话化读模型。

## 2. 架构结论

- 教师侧会话化复盘事实源统一为 `GET /api/v1/teacher/students/:id/attack-sessions`。
- `attack-sessions` 由 `teaching_readmodel` 在查询时从证据事件派生生成，不新增会话持久化表。
- 普通训练、Jeopardy、AWD 采用不同的分组键，但都共享同一套事件来源与事件语义。
- `with_events` 控制是否返回会话内事件明细；教师复盘工作台默认请求明细。
- DTO 中保留了 `capture_available` 与 `capture_ref` 字段，但当前实现统一返回 `capture_available=false`，尚未落地完整流量附件。

## 3. 模块边界与职责

### 3.1 模块清单

- `internal/module/teaching_readmodel/application/queries`
  - 负责：从证据事件派生会话、按模式/结果筛选、分页并生成 `summary`
  - 不负责：写入或缓存会话快照

- `internal/module/teaching_readmodel/infrastructure`
  - 负责：复用证据链读取逻辑，为会话构建提供原始事件序列
  - 不负责：定义会话规则

- `frontend/src/features/teacher-student-analysis`
  - 负责：维护题目/模式/结果筛选状态，协调 `evidence` 与 `attack-sessions` 双接口加载
  - 不负责：在前端自己推导会话切分

- `frontend/src/widgets/teacher-student-review-workspace`
  - 负责：把会话作为主阅读单位展示，并基于事件/会话生成摘要和教师观察点
  - 不负责：重新定义攻击结果判定逻辑

### 3.2 事实源与所有权

- 会话派生输入事实源：`TeacherEvidenceEvent`
- 会话读接口事实源：`TeacherAttackSessionResp`
- 会话切分、结果判定和摘要统计 owner：`teaching_readmodel.QueryService`

## 4. 关键模型与不变量

### 4.1 核心查询与响应

- `TeacherAttackSessionQuery`
  - `mode`
  - `challenge_id`
  - `contest_id`
  - `round_id`
  - `result`
  - `with_events`
  - `limit`
  - `offset`

- `TeacherAttackSessionResp`
  - `summary.total_sessions`
  - `summary.success_count`
  - `summary.failed_count`
  - `summary.in_progress_count`
  - `summary.unknown_count`
  - `summary.event_count`
  - `summary.capture_available_count`
  - `sessions[]`

### 4.2 会话切分规则

- 普通训练：按 `challenge_id` 聚合
- Jeopardy：按 `contest_id + challenge_id` 聚合
- AWD：按 `team_id + contest_id + service_id + victim_team_id` 聚合
- 同一分组内事件间隔超过 `1h` 时拆成新的 session

### 4.3 结果判定规则

- 存在成功 `challenge_submission` 或成功 `awd_attack_submission`：`success`
- 存在失败提交但没有成功事件：`failed`
- 存在访问、代理请求或 AWD 流量行为但没有提交结果：`in_progress`
- 其余情况：`unknown`

### 4.4 不变量

- 会话本身不入库，只在读路径派生。
- 会话列表按 `started_at` 倒序返回；会话内事件按时间升序返回。
- `capture_count` 当前恒为 `0`，`capture_available_count` 当前恒为 `0`，这两个字段仅作为后续扩展预留。
- `attack-sessions` 不绕过证据链接口单独定义事件格式，事件 `stage/source/meta` 仍继承统一证据语义。

## 5. 关键链路

### 5.1 会话读链路

1. 教师复盘工作台请求 `/api/v1/teacher/students/:id/attack-sessions`。
2. handler 绑定 `mode/result/challenge_id/contest_id/round_id/with_events`。
3. query service 先构造内部 `TeacherEvidenceQuery`，统一读取证据事件。
4. query service 再按模式过滤、分组、切分、判定结果并生成会话摘要。
5. 若 `with_events=false`，仅返回会话级摘要，不返回嵌套事件明细。

### 5.2 前端复盘工作台链路

1. 页面先读取 `evidence`，用于生成题目筛选项和观察条。
2. 再读取 `attack-sessions`，以会话为主组织展示访问、请求、提交和复盘输出。
3. 切换题目/模式/结果时，仅更新筛选参数并重新请求会话接口。

## 6. 接口与契约

### 6.1 对外接口

- `GET /api/v1/teacher/students/:id/attack-sessions`

主要查询参数：

- `mode=practice | jeopardy | awd`
- `challenge_id`
- `contest_id`
- `round_id`
- `result=success | failed | in_progress | unknown`
- `with_events`
- `limit`
- `offset`

### 6.2 会话与事件契约

- 会话字段包含：
  - `id`
  - `mode`
  - `student_id`
  - `team_id`
  - `challenge_id`
  - `contest_id`
  - `round_id`
  - `service_id`
  - `victim_team_id`
  - `title`
  - `started_at`
  - `ended_at`
  - `result`
  - `event_count`
  - `capture_count`

- 会话内事件字段包含：
  - `type`
  - `stage`
  - `source`
  - `occurred_at`
  - `actor`
  - `target`
  - `summary`
  - `meta`
  - `capture_available`
  - `capture_ref`

## 7. 兼容与迁移

- `GET /api/v1/teacher/students/:id/evidence` 继续保留，并作为会话派生的底层事实源。
- 当前没有 `AttackSession` 数据表，也没有 `TrafficCapture` 数据表；后续扩展完整流量附件时，应在现有 DTO 上补充 `capture_ref`，而不是重写当前会话接口。
- 教师复盘工作台应继续以会话为主、事件为辅，不应回退到直接展示散乱原始事件。

## 9. 验证标准

- 教师/管理员可以通过 `/api/v1/teacher/students/:id/attack-sessions` 读取会话化复盘结果。
- 相同目标下间隔超过 `1h` 的尝试会被拆成多个独立 session。
- `success / failed / in_progress / unknown` 结果判定与事件序列一致。
- `with_events=false` 时会话列表仍可用，但不返回会话内事件明细。

## 详细实施计划

### Phase 1：事件级会话还原落地

后端先在 `teaching_readmodel` 内完成读模型聚合，不新增写入链路和数据库表：

- 在 `TeacherEvidenceEvent` 的内部记录结构中补充 `source`、`stage`、`student_id`、`team_id`、`contest_id`、`round_id`、`service_id`、`victim_team_id` 等归属字段。
- 扩展教师证据仓库读取范围，纳入 `submissions`、`submission_writeups`、人工评审提交、`awd_attack_logs` 和 `awd_traffic_events`。
- 新增 `GET /api/v1/teacher/students/:id/attack-sessions`，复用证据事件构建结果，按目标和时间间隔聚合为会话。
- 会话结果只从明确事件推导：正确提交或成功 AWD 攻击为 `success`，失败提交或失败 AWD 攻击为 `failed`，仅访问/请求为 `in_progress`，其余为 `unknown`。
- DTO 中预留 `capture_available` 和 `capture_ref`，第一阶段固定无完整流量证据。

### Phase 2：教师前端会话视图

前端在学生分析页证据区域接入会话接口：

- 页面加载学生详情时并行拉取 `evidence` 和 `attack-sessions`。
- 复盘区默认展示会话摘要、结果、模式、目标和关键事件路径。
- 会话内按时间线展示事件明细，并针对代理请求、AWD 攻击、AWD 流量、Writeup、人工评审提供字段化摘要。
- 对未知事件类型保留通用标签和 summary 展示，不让后续事件扩展破坏页面。

### Phase 3：筛选与归档对齐

在会话读模型稳定后继续收敛复盘体验：

- 为证据接口和会话接口补齐 `contest_id`、`round_id`、`event_type`、`result`、时间范围和分页筛选。
- 将复盘归档的证据构建逻辑迁移到同一事件构建器，确保实时页面和导出归档口径一致。
- 增加教师观察点，例如只访问未提交、连续错误提交、成功后缺少 Writeup、AWD 成功但缺少复盘材料。

### Phase 4：全流量证据包

当 runtime proxy 或对象存储证据能力准备好后，再引入 `traffic_captures`：

- 数据库只保存证据包索引、hash、大小、脱敏状态和对象存储引用。
- `AttackEvent` 只暴露 `capture_ref`，不把完整请求响应写入事件 `meta`。
- 查看和下载完整证据需要教师/管理员权限，并写入审计日志。

### v1：事件级攻击过程还原

- 聚合 `audit_logs`、`submissions`、`awd_attack_logs`、`awd_traffic_events`。
- 返回统一 `AttackEvent` 和 `AttackSession`。
- 教师页面按时间线展示主要攻击过程。
- 不保存完整请求响应和 pcap。

### v2：统一事件构建器

- 把证据接口和攻击会话接口共用的事件转换逻辑收敛到教学读模型内部。
- 对 `meta` 字段做稳定 schema 和脱敏处理。
- 增加未知事件类型兜底展示。

### v3：会话切分增强

- 支持按时间间隔拆分同一题的多次尝试。
- 支持按 AWD 轮次、目标队伍、服务实例聚合。
- 支持按结果、题目、竞赛和时间范围筛选。

### v4：流量证据包

- 新增 `traffic_captures` 索引表。
- runtime proxy 可选保存脱敏 HTTP exchange 或 HAR。
- 对象存储保存大对象，数据库只保存引用和校验信息。

### v5：回放与行为识别

- 前端提供请求响应查看、HAR 下载或回放入口。
- 后端增加行为标签识别，例如 SQL 注入尝试、文件上传、疑似命令执行。
- 报告和复盘归档可引用会话和证据包。

## 不做项

第一阶段明确不做：

- 全流量 pcap 抓取。
- 命令级终端录制。
- 浏览器录屏。
- 完整 HTTP body 长期持久化。
- 自动判断漏洞类型和攻击成功原因。

这些能力可以在 `TrafficCapture` 和行为标签体系成熟后继续扩展。

## 最小验收标准

- 教师可以打开某个学生的攻击会话列表。
- 每个会话能展示访问目标、平台代理请求、Flag 提交、AWD 攻击结果等关键事件。
- 普通训练、Jeopardy 和 AWD 场景至少都有清晰的数据归属字段。
- 事件中不暴露完整敏感凭据和完整 Flag。
- `capture_ref` 字段已预留，但没有全流量证据时不会影响页面展示。
