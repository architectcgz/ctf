## 1. 背景与问题

教学复盘工作台、攻击证据链、能力画像和推荐练习已经落地，但当前“建议层”仍然偏粗：

- 班级复盘建议主要依赖活跃率、已解题数和最低画像维度
- 个人复盘观察项数量少，且存在无事实支撑的固定文案
- 推荐题理由主要是维度标签回显，解释性不足

这会带来两个直接问题：

1. 系统难以稳定说明“为什么给出这个建议”
2. 论文里难以把现有实现论证成“基于实训数据生成教学建议”

当前需要补的不是新的页面或新的事实表，而是一层位于“证据事实”和“建议输出”之间的统一建议内核。

## 当前设计

- `code/backend/internal/teaching/advice/advice.go`
  - 负责：作为教学建议共享规则层，统一生成班级复盘建议、个人复盘观察和推荐理由，并在 `BuildClassReview(...)` 中决定哪些 class item 只能给动作、哪些 class item 可以附题；个人侧 `submission_stability` 与 `low_activity` 的阈值也在这里统一收口。
  - 不负责：直接查询题目、直接读取数据库、绕过 `assessment` 或 `teaching_query` 自行装配事实。
- `code/backend/internal/module/teaching_query/application/queries/class_insight_service.go`
  - 负责：把 `BuildClassReview(...)` 产出的 class item 装配成 `TeacherClassReviewResp`，并且只在语义匹配时附推荐题。
  - 不负责：重算班级风险规则、把流程型建议重新解释成题目推荐。
- `code/backend/internal/module/assessment/application/commands/report_service.go`
  - 负责：在个人复盘归档链路里，把 `timeline / evidence / writeups` 装配成个人教学事实快照，并补齐近 7 天活跃事实，供共享规则层生成 `low_activity` 等观察项。
  - 不负责：单独维护另一套个人观察规则，或绕过 `internal/teaching/advice` 直接写教师结论。
- `code/backend/internal/module/assessment/application/queries/recommendation_service.go`
  - 负责：把 challenge 查询返回的“实际命中维度”继续传给共享规则层，保证推荐理由解释的是命中的推荐维度，而不是盲用题目 `category`。
  - 不负责：重新发明弱项识别规则，或根据展示文案回写推荐候选排序。
- `code/backend/internal/module/assessment/infrastructure/repository.go`
  - 负责：作为个人 recommendation teaching snapshot owner，在 `GetStudentTeachingFactSnapshot(...)` 中把个人 AWD 正向成功证据合并进 `DimensionFact`，统一补齐 `success_count`、`evidence_count`、`solved_difficulty_counts`，并用 `max(skill_profiles.score, awd_success_coverage_ratio)` 抬升 `profile_score`。
  - 不负责：改写 `skill_profiles` 持久化公式、为 `AWDChallenge` 发明新的 points 口径，或把队伍级防守结果均摊回个人画像。
- `code/backend/internal/module/teaching_query/infrastructure/repository.go`
  - 负责：作为班级 teaching snapshot owner，在 `ListClassTeachingFactSnapshots(...)` 中沿用与个人 recommendation snapshot 相同的 AWD 维度合并口径，保证 class review 和个人 recommendation 看到的是同一套维度事实。
  - 不负责：维护独立于 `assessment` 之外的 AWD 维度统计分支，或把 AWD 事实只停留在 `AWDSuccessCount` 的展示层。
- 当前班级建议口径分两类：
  - 维度补强型建议：当前只有 `weak_dimension_cluster`，允许附推荐题；附题时必须和 class item `dimension` 匹配，依据见 `code/backend/internal/module/teaching_query/application/queries/class_insight_service.go` 与测试 `code/backend/internal/module/teaching_query/application/queries/class_insight_service_test.go`。
  - 流程干预型建议：`activity_risk`、`training_closure_gap`、`retry_cost_high` 默认只输出 `summary / evidence / action`，不再挂某个学生的第一条个人推荐题；这类建议的 owner 仍是 `code/backend/internal/teaching/advice/advice.go`。
- 当前 `activity_risk` 不是“只要出现低活跃学生就 warning”，而是同时看整体 `active_rate` 和低活跃学生占比：
  - `danger`：整体活跃明显下滑，或低活跃学生占比已经达到班级级风险
  - `warning`：班级节奏开始走低，需要班级层面补练
  - `attention`：整体训练仍健康，但有少数学生节奏偏慢
  - `good`：整体稳定且没有明显掉队点
- 当前个人推荐仍保持“证据优先”的弱项识别，不因为本次班级策略调整而改成画像最低分优先；对应实现仍在 `code/backend/internal/teaching/advice/advice.go` 与 `code/backend/internal/module/assessment/application/queries/recommendation_service.go`。
- 当前个人推荐遵循“两段式”路径：
  - 先看“已有证据支撑的弱项”或“仍需补样本的 attention 维度”
  - 如果没有补弱/补样本目标，再看是否存在 `progression-ready` 维度，并把推荐带宽提升到更高一档
- 当前个人复盘里的 `dimension_focus` 只在“已有证据支撑的弱项”或“仍需补样本的 attention 维度”上输出；`progression-ready` 只影响推荐题链路，不会被渲染成问题项。
- `recommended_difficulty_band` 仍表示“当前更适合的训练带宽”；但推荐理由里的 `summary` 必须绑定实际候选题 `challenge.difficulty`，如果两者不一致，要明确说明这是当前最接近的候选，而不是把 `easy` 题说成 `beginner`。
- 当前 `recommended_difficulty_band` 不再只停留在 `beginner / easy / medium`。当某个维度已经形成稳定基础，且同维度里已经完成更低一档的已解覆盖时，可以提升到 `hard / insane` 作为进阶推荐带宽。
- 当前个人复盘观察里，`submission_stability` 不再因为“只有 1 次错误提交且没有连续错误”就直接升成 `warning`；`low_activity` 也不再凭空读取 0 值，而是基于 review archive 装配出的近 7 天活跃事实决定 `attention / warning`。
- 当前个人弱项判定不会再把“1 次成功、少量样本”的维度直接放大成高置信度弱项；明确弱项至少要同时满足低画像、足够样本和没有稳定成功基础。
- 当前 review archive 装配会显式区分“题目成功数”“归档成功事件数”“归档失败事件数”和 `AWD success`，观察证据统一引用这套口径，不再用 `TotalAttempts - CorrectSubmissionCount` 反推失败数。
- 当前个人观察文案按训练状态分支：
  - `dimension_focus` 区分“高置信度薄弱项”和“已有成功样本但覆盖不足”
  - `submission_stability` 区分“已经有结果但试错偏高”和“当前还没有形成稳定命中”
  - `hands_on_depth` 区分“只有过程证据”“过程充分”和“过程充分且已有 AWD 结果”
  - `training_closure` 区分“还没有复盘输出”“已经开始补输出”和“闭环已经稳定”
- 当前 `weak_dimension_cluster` 只有在同一维度至少形成 2 名学生、且达到班级聚类阈值时才输出，避免把单个学生的弱项误报成班级共性问题。
- Guardrail：
  - `code/backend/internal/teaching/advice/advice_test.go`
  - `code/backend/internal/module/assessment/application/commands/report_service_test.go`
  - `code/backend/internal/module/assessment/application/queries/recommendation_service_test.go`
  - `code/backend/internal/module/challenge/infrastructure/repository_test.go`
  - `code/backend/internal/module/teaching_query/application/queries/class_insight_service_test.go`

## 2. 架构结论

- 教学复盘建议统一由共享的 `internal/teaching/advice` 规则层生成，不再分别散落在 `teaching_query`、`assessment` 和 `recommendation` 服务中。
- 建议生成继续采用规则型、可解释路径，不引入 LLM 或外部推理服务。
- 建议层的输入不是原始数据库表，而是由各 owning 模块装配后的原始教学事实快照。
- 建议层统一产出三类结果：
  - 班级复盘建议
  - 个人复盘观察
  - 推荐题理由
- 班级建议里的题目推荐只属于“维度补强型建议”，流程干预型建议默认不给题。
- 弱项维度识别、排序和证据充足性判断统一归属 `internal/teaching/advice`，`RecommendationService` 不再保留单独 owner。
- 弱项维度判断必须同时考虑画像得分与证据置信度，不能再把“最低分”直接等同于“明确弱项”。
- 没有 evidence-backed risk / coverage gap 时，个人推荐允许为空；系统不能为了“始终给题”而把健康维度伪装成补强目标。
- 没有 evidence-backed risk / coverage gap 时，如果某个维度已经满足 progression-ready 条件，个人推荐可以转成进阶题；只有补弱、补样本和进阶都不成立时才返回空。
- `activity_risk` 必须同时结合整体活跃率和低活跃学生占比，不能因为个别学生掉队就直接把全班提升为 `warning`。
- 本次设计不新增事实表，不改 `solved / score / rank` 统计口径，不重写现有证据链和攻击会话读模型。

## 3. 模块边界与职责

### 3.1 模块清单

- `internal/teaching/advice`
  - 负责：定义教学事实评估 contract、弱项维度识别、弱项置信度计算、统一严重度语义、班级建议规则、个人观察规则、推荐理由规则
  - 不负责：直接查询数据库、做权限控制、决定推荐候选题集合

- `internal/module/teaching_query`
  - 负责：读取教师实时分析所需事实、装配班级和学生信号、组织教师读接口
  - 不负责：长期维护独立于共享规则层之外的班级建议逻辑

- `internal/module/assessment`
  - 负责：读取画像、复盘归档和报告事实，装配个人观察所需信号
  - 不负责：单独维护另一套个人观察规则

- `internal/module/challenge`
  - 负责：按维度、标签、发布状态和已解集合返回推荐候选题
  - 不负责：解释为什么推荐给当前学生

- 能力覆盖事实：
  - `skill_profiles`
  - `submissions`

  - `audit_logs`
  - `submission_writeups`
  - `awd_attack_logs`
  - `awd_traffic_events`
- 班级统计事实：
  - `users`

- 建议层共享 owner：
  - `internal/teaching/advice`

### 4.2 不变量

- 建议层只消费原始教学事实快照，不直接消费原始表结构。
- `assessment` 与 `teaching_query` 的 teaching fact snapshot 会统一吸收个人 AWD 正向成功证据；合并口径是“已发布 `awd_challenge` 总量”对比“当前学生去重成功的 `awd_challenge_id` 覆盖”，并以 `max(skill_profiles.score, awd_success_coverage_ratio)` 计算 snapshot 层 `profile_score`。
- `AWDSuccessCount` 只统计 `submitted_by_user_id = 当前学生`、`source = submission`、`is_success = true`、`score_gained > 0` 的个人攻击成功事件；零分重复命中、手工补录和队伍级结果不算个人正向成功证据。
- `success_count`、`evidence_count` 和 `solved_difficulty_counts` 在 teaching fact snapshot 层会额外合并 AWD challenge 的个人去重成功覆盖；同一 AWD 题打多个受害队只记一次。
- 每条建议都必须能追溯到至少一个明确指标和一个明确事实来源。
- `confidence`、`is_weak`、`severity` 只能由 `internal/teaching/advice` 计算，不能分别散落在 `teaching_query`、`assessment` 和 `recommendation` 中派生。
- 弱项维度必须由“得分偏低 + 证据充足 + 没有稳定成功基础”共同判定；证据不足或刚出现少量成功样本时，只能给出“待补样本 / 待建立基础”，不能伪装成明确弱项。
- `RecommendationService` 只能消费 `advice` 给出的推荐目标结果，不能继续保留独立的弱项识别或 progression 判断逻辑。
- 推荐候选题集合仍由现有 challenge 推荐查询产生；建议层只负责排序补充与理由解释。
- 进阶推荐必须同时满足：当前维度没有明显风险、基础证据稳定、并且同维度里已经完成更低一档的已解覆盖；不能只因为画像高就直接跳到 `hard / insane`。
- challenge 推荐查询可以通过 `category` 或 knowledge tag 命中候选题；一旦通过 knowledge tag 命中，后续推荐理由必须继续沿用实际命中的推荐维度，不能再退回题目原始 `category` 猜测。
- 推荐理由允许保留“目标训练带宽”字段，但 `summary` 不能伪装实际题目难度；当 `difficulty_band` 与 `challenge.difficulty` 不一致时，必须显式说明是“最接近候选”。
- `progression-ready` 只属于推荐链路语义，不属于复盘问题项；个人复盘观察不能把“准备好进阶”渲染成新的风险提示。
- 个人观察与班级建议共用同一套规则码和严重度语义，避免出现两套不一致的判断口径。
- review archive 的失败数在 evidence 可用时必须来自显式 tracked submission / attack 事件，而不是从总尝试数反推。
- 班级维度聚类必须满足最小 cluster 门槛，不能因为单个学生的明确弱项就直接上升为班级共性问题。
- `severity` 是唯一严重度语义字段，不再额外维护 `Accent`、`Level` 这类展示型平行字段。
- 当前设计不新增数据库持久化表，所有建议结果继续在查询时计算。

## 5. 关键链路

### 5.1 班级复盘建议链路

1. 教师请求班级复盘。
2. `teaching_query` 读取班级学生目录、活跃事实、`skill_profiles`、练习侧维度统计，以及 `awd_attack_logs` / `awd_challenges` 中可归因到个人的 AWD 正向成功覆盖。
3. `teaching_query` 为每个学生构造 `StudentTeachingFactSnapshot`，并在 snapshot owner 层把 AWD 维度成功覆盖合并进 `profile_score`、`success_count`、`evidence_count` 和 `solved_difficulty_counts`。
4. `internal/teaching/advice` 计算：
   - 高置信度薄弱维度
   - 低活跃风险
   - 闭环不足学生
   - 高试错成本学生
5. Query service 组装 `TeacherClassReviewResp`：
   - `weak_dimension_cluster` 才允许附推荐题
   - 附题时只接受和 class item `dimension` 匹配的推荐结果
   - `activity_risk`、`training_closure_gap`、`retry_cost_high` 只保留动作建议，不挂题

### 5.2 个人复盘观察链路

1. `assessment.ReportService` 构造个人复盘归档时，继续读取 `timeline`、`evidence`、`writeups`、`manualReviews` 和 `skillProfile`。
2. `assessment` 将这些事实装配为个人原始教学事实快照，并补齐近 7 天活跃天数、事件数、题目成功数、归档成功 / 失败事件数和 AWD 成功数。
3. `internal/teaching/advice` 生成个人观察项：
   - 训练闭环
   - 提交稳定性
   - 近期活跃度
   - 实操深度
   - 维度聚焦（仅在存在 evidence-backed weak / attention target 时输出）
   - AWD 实战参与
4. `submission_stability` 只在连续错误或错误成本达到重复风险时升级为 `warning / danger`，不再把单次错误误报成高试错；`dimension_focus` 对“已有成功样本但覆盖不足”的维度只给 `attention` 级补样本结论。
5. 归档页和导出继续消费结构化观察项，不单独维护旧规则；观察 `evidence` 必须和 `solved / attempts / AWD success` 的统计语义可同时成立。

### 5.3 推荐题解释链路

1. `assessment.RecommendationService` 读取由 `assessment.Repository.GetStudentTeachingFactSnapshot(...)` 组装的 teaching fact snapshot，以及当前普通练习已解题集合。
2. `internal/teaching/advice` 先输出弱项维度识别、证据置信度、进阶准备状态和推荐难度带。
3. `RecommendationService` 再据此向 `challenge` 查询未解候选题，并继续保留候选题实际命中的推荐维度。
4. `internal/teaching/advice` 对候选题生成推荐理由和严重度解释。
5. 如果当前没有 evidence-backed weak / attention target，但存在 progression-ready 维度，推荐链路可以继续返回进阶题；只有两类目标都不存在时才返回空结果。

## 6. 接口与契约

### 6.1 对外接口影响

当前设计不新增新的教师入口，继续复用现有接口：

- `GET /api/v1/teacher/classes/:class_name/review`
- 学生画像与推荐相关接口
- 学生复盘归档接口

接口变化以“直接收口为统一 advice 契约”为主，不要求改路由，但允许直接替换旧字段。

### 6.2 稳定契约方向

当前设计直接将相关 DTO 收口为统一 advice 契约：

- `TeacherClassReviewItem`
  - 使用 `severity`、`summary`、`evidence`、`action`、`reason_codes`
  - 移除 `Accent`
- `TeacherRecommendationItem`
  - 使用 `dimension`、`difficulty_band`、`severity`、`reason_codes`、`summary`、`evidence`
  - `difficulty_band` 表示目标训练带宽，`summary` 需要同时对齐实际推荐题难度
  - `difficulty_band` 在补弱场景可以是 `beginner / easy / medium`，在 progression-ready 场景可以提升到 `hard / insane`
- `ReviewArchiveObservation`
  - 使用 `severity`、`code`、`dimension`、`summary`、`evidence`、`action`
  - 移除 `Level`

## 7. 兼容与迁移

- 现有 `skill_profiles` 继续保留；recommendation / class review 读取的 teaching snapshot 可以在查询时用 AWD success coverage 抬升 `profile_score`，但不会回写或重算 `skill_profiles` 表。
- 现有证据链、攻击会话和归档页面继续保留；本专题只升级建议生成，不重做教师工作台。
- 当前 `teaching_query.QueryService.GetClassReview`、`assessment.ReportService.buildReviewArchiveObservations`、`RecommendationService` 中的硬编码规则，在实现阶段必须直接收口到共享规则层，不保留兼容双轨。
- 当前无事实支撑的 `hint_usage` 固定文案不再作为建议层长期保留项；若后续重新引入提示解锁事实，应以新增信号方式接入，而不是恢复占位文案。
- 当前仍处于开发期，相关 DTO、前端消费和归档展示允许一次性直接重构，不做渐进迁移。

## 9. 验证标准

- 对同一批固定样本数据，班级建议、个人观察和推荐题理由应稳定输出，不依赖维度字典序偶然命中。
- 系统能区分“维度薄弱”“训练节奏下滑”“试错成本过高”“闭环不足”这几类不同问题。
- 同一学生在班级建议、个人观察和推荐题理由三条链路里，证据不足维度不能被任何一条链路判成明确弱项。
- 推荐题理由必须同时说明推荐维度和推荐原因，不能只回显维度标签。
- 健康、证据稳定的学生不应再被强行附上推荐题或 `dimension_focus` 观察。
- 班级流程型建议不能再附和建议主题无关的个人推荐题。
- 当班级整体活跃健康、只是少数学生掉队时，`activity_risk` 至少降为 `attention`，而不是直接 `warning`。
- 推荐理由不能再把目标训练带宽直接回写成题目实际难度。
- `AWDSuccessCount` 必须忽略 `score_gained = 0` 的重复命中和 `source != submission` 的手工补录。
- 个人复盘里只有单次错误提交时，`submission_stability` 不能再误报成 `warning`。
- 个人复盘里的 `low_activity` 必须基于真实近 7 天活跃事实，而不是默认 0 值。
- 通过 knowledge tag 命中的推荐题，理由解释维度不能再退回题目原始 `category`。
- 个人复盘观察项应能显著区分高闭环学生、连续错误提交学生、低活跃学生和 AWD 参与学生。
- 个人复盘里“少量成功样本”的维度不能再直接被判成 `danger` / `warning` 弱项。
- 当普通 `skill_profile` 仍偏低，但同一维度已经完成 `easy + medium` 的 AWD 成功覆盖时，推荐链路应能切到 progression-ready，并优先返回更高一档的候选题。
- review archive 的观察证据不能再出现“摘要 solved 很少，但观察里正确提交很多次却没有解释来源”的口径冲突。
- 共享规则层成为唯一建议 owner 后，不再在 `teaching_query` 与 `assessment` 中保留两套长期并行规则。
