# 判题模式扩展架构

## 文档元信息

- 状态：`implemented`
- 事实源级别：`final`
- 适用范围：`backend`、`frontend`、`contracts`
- 关联模块：
  - `internal/module/challenge`
  - `internal/module/practice`
  - `internal/module/contest`
  - `frontend/src/features/platform-challenge-detail`
  - `frontend/src/features/challenge-detail`
  - `frontend/src/components/teacher/student-insight`
- 最后更新：`2026-05-07`

## 1. 背景与问题

平台最初主要围绕 `static` 和 `dynamic` Flag 设计，但教学场景很快出现两类稳定需求：

- `regex`：需要接受一类符合规则的答案，而不是单个固定字符串。
- `manual_review`：需要教师或管理员对开放题、过程题、题解式答案做人工判定。

当前需要收口的是：判题模式扩展仍沿用现有 `challenge + submission` 主模型，不引入新的插件化判题框架或平行提交表。

## 2. 架构结论

- 平台当前统一支持四种判题模式：`static`、`dynamic`、`regex`、`manual_review`。
- 判题模式主轴仍是 `challenges.flag_type`，不是独立策略表。
- `regex` 的规则事实源是 `challenges.flag_regex`。
- `manual_review` 的审核状态事实源是 `submissions.review_status / reviewed_by / reviewed_at / review_comment / score`。
- 题包导入、后台配置、自检、训练提交、教师审核和前端展示都共享同一套模式枚举。
- `manual_review` 当前只在练习链路形成完整闭环；竞赛链路显式拒绝人工审核题提交，不把它误记为错误提交。

## 3. 模块边界与职责

### 3.1 模块清单

- `internal/module/challenge/domain` 与 `application/commands`
  - 负责：题包解析、后台配置、自检和导入时的模式落库
  - 不负责：保存教师审核结果

- `internal/module/challenge/application/queries/flag_service`
  - 负责：统一提供静态、动态、正则三类自动校验语义
  - 不负责：对 `manual_review` 做自动通过判断

- `internal/module/practice/application/commands/submission_service`
  - 负责：练习提交流程、`pending_review` 状态返回、自动判题和实例解题副作用
  - 不负责：教师审核决策

- `internal/module/practice/application/commands/manual_review_service`
  - 负责：教师/管理员审核队列、审核详情、通过/驳回、审核后的计分副作用
  - 不负责：扩展新的题目配置来源

- `internal/module/contest/application/commands`
  - 负责：竞赛提交流程
  - 不负责：当前阶段的人工审核计分

- `frontend/src/features/platform-challenge-detail`
  - 负责：管理员配置 `flag_type / flag_regex / flag_prefix`
  - 不负责：前端自行实现判题规则

- `frontend/src/features/challenge-detail`
  - 负责：学员侧展示 `correct / incorrect / pending_review`
  - 不负责：自行推导审核状态

- `frontend/src/components/teacher/student-insight/StudentInsightManualReviewSection.vue`
  - 负责：教师查看人工审核列表、详情和审核操作
  - 不负责：改变后端审核规则

### 3.2 事实源与所有权

- 判题模式事实源：`challenges.flag_type`
- 正则规则事实源：`challenges.flag_regex`
- 人工审核状态事实源：`submissions.*review*`
- 学员提交结果展示事实源：`SubmissionResp.status`

## 4. 关键模型与不变量

### 4.1 题目侧模型

- `Challenge.FlagType`
- `Challenge.FlagRegex`
- `Challenge.FlagPrefix`

支持值：

- `static`
- `dynamic`
- `regex`
- `manual_review`

### 4.2 提交侧模型

- `Submission.ReviewStatus`
- `Submission.ReviewedBy`
- `Submission.ReviewedAt`
- `Submission.ReviewComment`
- `Submission.Score`
- `Submission.IsCorrect`

当前审核状态包括：

- `not_required`
- `pending`
- `approved`
- `rejected`

### 4.3 不变量

- `regex` 题必须能编译通过后才允许导入或保存配置。
- `manual_review` 题提交时保存学员原答案到 `submissions.flag`，审核前 `is_correct=false`。
- `manual_review` 审核通过后，提交记录被更新为 `is_correct=true`、`score=challenge.points`，并触发进度缓存失效和计分事件。
- `manual_review` 审核驳回后，提交仍保留答案与审核意见，但 `score=0`。
- 题目自检对 `manual_review` 只验证配置存在，不做自动 Flag 命中校验。
- 竞赛链路当前对 `manual_review` 直接返回“不支持竞赛提交”，而不是写入一条错误提交。

## 5. 关键链路

### 5.1 `regex` 配置与提交链路

1. 管理员在后台题目详情页选择 `regex`。
2. 后端通过 `ConfigureFlagReq` 或题包导入写入 `flag_type=regex`、`flag_regex=<compiled pattern>`。
3. 自检阶段编译并校验正则表达式。
4. 学员在训练或竞赛中提交答案时，统一走 `regexp.MatchString(challenge.FlagRegex, input)`。
5. 前端按自动判题结果展示 `correct` 或 `incorrect`。

### 5.2 `manual_review` 练习链路

1. 管理员在后台题目详情页或题包中选择 `manual_review`。
2. 学员在练习侧提交答案后，`submission_service` 创建一条 `review_status=pending` 的提交记录。
3. 学员端收到 `SubmissionResp.status = pending_review`。
4. 教师或管理员通过 `/api/v1/teacher/manual-review-submissions*` 查看队列与详情。
5. 审核通过后更新提交记录、分数、进度缓存和计分事件；驳回则记录审核意见并保持未通过。

### 5.3 竞赛安全兜底链路

1. 竞赛提交先进入 `validateContestSubmission`。
2. 若题目 `flag_type=manual_review`，直接返回参数错误。
3. 不写入竞赛错误提交，也不把人工审核题误计入自动判题统计。

## 6. 接口与契约

### 6.1 配置契约

- `ConfigureFlagReq`
  - `flag_type`
  - `flag`
  - `flag_regex`
  - `flag_prefix`

- 题包 `challenge.yml`
  - `flag.type`
  - `flag.value`
  - `flag.prefix`

### 6.2 提交与审核契约

- 学员提交响应 `SubmissionResp.status` 当前支持：
  - `correct`
  - `incorrect`
  - `pending_review`

- 教师审核接口：
  - `GET /api/v1/teacher/manual-review-submissions`
  - `GET /api/v1/teacher/manual-review-submissions/:id`
  - `PUT /api/v1/teacher/manual-review-submissions/:id/review`

## 7. 兼容与迁移

- 原有 `static` / `dynamic` 题不需要迁移到新的模型，仍沿用现有字段和自动判题链路。
- `regex` 和 `manual_review` 只是对现有 `flag_type` 轴的扩展，不新增平行表或插件注册机制。
- 若后续需要支持竞赛侧人工审核，必须继续复用 `submissions` 审核字段和计分副作用，而不是再新造一套竞赛专用审核记录。

## 9. 验证标准

- `regex` 题可以从后台配置或题包导入后完成自动判题。
- `manual_review` 题在练习侧提交后返回 `pending_review`，并进入教师审核队列。
- 审核通过会更新得分与通过状态；驳回会保留审核意见但不计分。
- 竞赛侧提交人工审核题不会被误记为自动判错。
