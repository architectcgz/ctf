## 1. 背景与问题

教师学生分析页当前已经不是单纯的“若干列表”。代码里已经落成了一套围绕证据事件、攻击会话和复盘归档的工作台，需要把真实结构写清楚：

- 实时教学复盘使用平铺证据流和攻击会话两种读模型
- 会话不是单独存表，而是由 evidence event 在线归并得到
- 完整归档页与实时分析页共享同类证据事实，但职责不同

## 2. 架构结论

- `TeacherStudentAnalysis.vue` 是教师实时教学复盘的主入口，承载画像、推荐、题解、人工审核、证据链和训练记录。
- `GetStudentEvidence` 提供平铺证据流，`GetStudentAttackSessions` 提供过程化会话视图，二者共用同一批 evidence 事实。
- 攻击会话当前按 `practice / jeopardy / awd` 三种模式归类，并以 1 小时空窗切分连续会话。
- 前端 `useTeacherReviewWorkspace` 统一管理证据、会话和筛选条件，不让页面组件各自直接拼查询。
- `TeacherStudentReviewArchive` 仍然保留为稳定归档页，与实时分析页并行，不混成单页多职责大组件。
- 个人 / 班级成绩口径当前仍按练习 solved / score / rank 统计；AWD 个人事件进入 `total_attempts`、证据链、攻击会话和复盘归档，但不直接改写这些练习型排名指标。

## 3. 模块边界与职责

### 3.1 模块清单

- `teaching_readmodel.QueryService`
  - 负责：权限校验、证据查询、攻击会话构建
  - 不负责：个人报告导出

- `buildAttackSessions`
  - 负责：把 evidence event 归并成教师可读的训练 / 攻击过程
  - 不负责：持久化 session

- `useTeacherReviewWorkspace`
  - 负责：前端加载 `evidence`、`attackSessions`、筛选状态和 challenge 选项
  - 不负责：学生基础画像与推荐任务

- `StudentAnalysisPage`
  - 负责：把不同读模型组织成多标签工作台
  - 不负责：直接请求后端

- `TeacherReviewArchiveWorkspace`
  - 负责：展示导出的稳定归档快照
  - 不负责：实时筛选和会话重算

### 3.2 事实源与所有权

- 实时证据事实源：`GetStudentEvidence`
- 会话化事实源：`GetStudentAttackSessions(buildAttackSessions(events))`
- 归档事实源：`ReportService.GetStudentReviewArchive`

## 4. 关键模型与不变量

### 4.1 核心实体

- `TeacherEvidenceEvent`
  - `type`
  - `challenge_id`
  - `title`
  - `detail`
  - `timestamp`
  - `meta`

- `TeacherAttackSession`
  - `mode`
  - `challenge_id`
  - `contest_id`
  - `round_id`
  - `service_id`
  - `victim_team_id`
  - `started_at`
  - `ended_at`
  - `result`
  - `events`

### 4.2 不变量

- 会话构建不新建数据库表，完全由 evidence 事件在线分组生成。
- `attackSessionGap = 1h` 是当前固定切分窗口，超过 1 小时会拆成新会话。
- AWD 会话按 `team + contest + service + victim_team` 分组；非 AWD 会话按 `mode + challenge + contest` 分组。
- `TeacherAttackSessionQuery.with_events = false` 时后端会返回 session 摘要但裁掉事件列表。
- 前端 challenge 筛选选项来自 evidence 和 session 的合并结果，不单独维护静态字典。
- AWD 个人攻击事件会进入教学复盘读模型，但不会把个人 / 班级报告里的 `solved / score / rank` 直接改写成比赛口径。

## 5. 关键链路

### 5.1 实时教学复盘链路

1. 教师进入 `TeacherStudentAnalysis.vue`。
2. `useTeacherStudentAnalysisPage` 加载画像、推荐、时间线等基础数据。
3. `useTeacherReviewWorkspace` 先调用 `getStudentEvidence`，再调用 `getStudentAttackSessions`。
4. `StudentAnalysisPage` 在“证据链”和“训练记录”标签中分别展示平铺事件与攻击会话。

### 5.2 会话构建链路

1. `GetStudentAttackSessions` 先用 `buildEvidenceQuery` 生成 evidence 查询条件。
2. 后端读取证据事件后，按 mode / challenge / contest / round 等条件过滤。
3. `buildAttackSessions` 将事件按分组键聚合，并在超过 1 小时间隔时拆段。
4. 每段生成 `TeacherAttackSession`，同时计算 `success / failed / in_progress / unknown` 摘要。

### 5.3 归档链路

1. 教师可从分析页跳转到完整复盘页，或直接导出 review archive。
2. 归档页 `TeacherStudentReviewArchive` 读取 `ReportService.GetStudentReviewArchive` 的稳定快照。
3. 实时页与归档页共享证据类型，但归档页不再重新计算实时筛选状态。

## 6. 接口与契约

### 6.1 教师复盘接口

- `GET /teacher/students/:id/evidence`
- `GET /teacher/students/:id/attack-sessions`

当前会话查询支持：

- `mode`
- `challenge_id`
- `contest_id`
- `round_id`
- `result`
- `with_events`
- `limit`
- `offset`

### 6.2 前端工作台契约

`StudentAnalysisPage` 当前稳定承载以下标签：

- `学员画像`
- `推荐任务`
- `发布的题解`
- `人工审核`
- `证据链`
- `训练记录`

## 7. 兼容与迁移

- 证据流仍然保留平铺结构，便于审计与归档；会话只是读取层派生视图。
- AWD 复盘页和学生分析页仍分别服务“赛事态势”和“个人教学复盘”，不合并为同一条路由。
- 若后续要调整 session 切分窗口或分组键，必须同步更新前后端查询约束与测试基线。

## 9. 验证标准

- 教师实时分析页能同时看到平铺证据和按过程组织的攻击会话。
- 会话筛选条件变化后，evidence 与 sessions 会按同一 challenge / contest / round 条件重载。
- AWD 事件会进入 `attackSessions`，且按当前分组键和 1 小时间隔正确拆段。
- 归档页继续输出稳定快照，不依赖实时分析页的本地状态。
