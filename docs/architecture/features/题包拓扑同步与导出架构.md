## 1. 背景与问题

题包拓扑需要在导入、平台内编辑和导出之间保持同一条来源链路，同时不能把平台内部拓扑结构反向强加给题包作者。

需要收口的核心问题是：

- 题包如何表达原生拓扑输入
- 平台如何保存来源、基线和漂移状态
- 导出时如何基于当前平台状态重建完整题包

## 2. 架构结论

- 标准题包通过 `challenge.yml` 中的 `extensions.topology.source` 指向题包内 `docker/topology.yml`。
- `docker/topology.yml` 是 package-native 输入，平台导入后再映射成内部 `TopologySpec`。
- 平台保存题包来源、导入基线、源码树和修订记录，不丢失 provenance。
- 导出以当前平台状态为准，重写 `challenge.yml` 和 `docker/topology.yml` 后生成新的 exported revision。

## 3. 模块边界与职责

### 3.1 模块清单

- package parser / topology parser
  - 负责：解析 `challenge.yml` 与 package-native topology
  - 不负责：直接把外部 YAML 当成平台内部 JSONB

- topology service
  - 负责：保存当前平台拓扑、来源信息、漂移状态和导出关联
  - 不负责：丢弃导入基线或源码树

- challenge package revision service
  - 负责：保存导入修订与导出修订的完整题包上下文
  - 不负责：只记录元数据而不保留可追溯来源

- 拓扑工作台
  - 负责：展示来源路径、同步状态、修订历史和导出入口
  - 不负责：重新定义题包原生输入格式

### 3.2 事实源与所有权

- 题包原生拓扑输入事实源：`challenge.yml -> extensions.topology.source`
- 当前平台拓扑事实源：`challenge_topologies`
- 题包修订事实源：`challenge_package_revisions`

## 4. 关键模型与不变量

### 4.1 核心实体

- `extensions.topology.source`
  - 语义：题包内部拓扑文件相对路径
  - 当前标准值：`docker/topology.yml`

- `challenge_topologies`
  - 关键字段：`source_type`、`source_path`、`package_revision_id`、`package_baseline_spec`、`sync_status`、`last_export_revision_id`

- `challenge_package_revisions`
  - 关键字段：`revision_no`、`source_type`、`parent_revision_id`、`package_slug`、`archive_path`、`source_dir`、`manifest_snapshot`、`topology_source_path`、`topology_snapshot`

### 4.2 不变量

- `source` 必须是题包内部相对路径，不能越过题包根目录。
- `drifted` 只表示平台当前拓扑偏离最近导入基线，不表示错误状态。
- 导出必须重写 manifest 和 topology，不能把原始 zip 原样返回。

## 5. 关键链路

### 5.1 导入链路

1. 上传题包。
2. 解析 `challenge.yml`。
3. 若 `extensions.topology.enabled = true`，读取 `extensions.topology.source`。
4. 解析 package-native topology。
5. 返回导入预览，包含拓扑摘要和题包文件树。
6. 提交导入后创建或更新 challenge。
7. 保存 `challenge_topologies` 与 `challenge_package_revisions`。

### 5.2 编辑与漂移链路

1. 拓扑工作台读取导入基线和当前平台拓扑。
2. 平台内编辑保存当前拓扑。
3. 系统比较当前拓扑与导入基线，更新 `sync_status = clean | drifted`。

### 5.3 导出链路

1. 读取 challenge、writeup、hint、runtime 和当前 topology。
2. 以最近 package revision 的源码树为基础。
3. 重写 `challenge.yml` 与 `docker/topology.yml`。
4. 生成新的 exported revision，并更新 `last_export_revision_id`。
5. 返回下载地址。

## 6. 接口与契约

### 6.1 题包契约

```yaml
extensions:
  topology:
    enabled: true
    source: docker/topology.yml
```

第一版 package-native topology 支持：

- `api_version: v1`
- `kind: topology`
- `entry_node_key`
- `networks`
- `nodes`
- `links`
- `policies`

### 6.2 导出接口

- `POST /api/v1/authoring/challenges/:id/package-export`
- `GET /api/v1/authoring/challenges/:id/package-export/download?revision_id=:rid`

## 7. 兼容与迁移

- 平台内部 `TopologySpec` 只是内部表示，不反向要求题包作者按内部 JSONB 书写。
- 若没有显式网络，平台补默认网络。
- 若没有节点标记 `inject_flag`，平台把入口节点作为默认 flag 注入点。
