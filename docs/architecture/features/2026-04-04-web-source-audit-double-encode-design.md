# Web 入门题「源码审计：双层伪装」设计

## 目标

新增一道本地 Web 入门题，改编自用户提供的源码审计练习，保留以下核心解题路径：

- 打开题目页面
- 查看页面源码
- 在一批 `_0x...` 风格变量中找到真正可疑的目标变量
- 对该变量做两次 Base64 解码
- 得到当前实例运行时注入的动态 flag

这道题应作为标准 `challenge-pack-v1` 容器题交付，并可在本地仓库中作为独立题目包维护。

## 非目标

- 不复刻远端站点的原始题面、原始假线索文本或原始 flag 值
- 不做多步骤交互、提交表单、接口抓包或浏览器调试型玩法
- 不引入数据库、鉴权、前端框架或多页面结构
- 不实现题目导入、发布或平台判题逻辑改造

## 当前背景

用户给出了原题的核心信息：

- 题型是源码审计 Level 1
- 页面源码中存在多条假 flag 注释
- 存在大量 `_0x...` 命名变量作为干扰
- 真正目标变量经过双层 Base64 编码
- 原题实际解出结果为一个静态值

本地改编需要保留“发现可疑变量并递归解码”的训练点，但必须改造成仓库内可运行、可维护、可动态注入 flag 的题目包，而不是直接镜像原题。

## 方案选择

### 方案 A：静态 HTML 直接写死双层编码结果

优点：

- 实现最简单
- 页面可完全静态托管

缺点：

- 无法接入平台动态 flag
- 题目答案会被固定在仓库里
- 与当前容器题交付方式不一致

### 方案 B：最小 Flask 服务端渲染单页源码审计题

优点：

- 可在运行时读取动态 flag 并双重编码
- 仍保持单页、低复杂度和稳定解法
- 与现有 `challenge-pack-v1` 容器题示例一致

缺点：

- 比纯静态页多一层服务端封装

### 方案 C：前端应用动态生成变量和提示

优点：

- 外观可以更复杂
- 可做更多干扰手法

缺点：

- 超出入门题需要
- 容易把题目做成交互题
- 会显著增加实现和维护成本

## 决策

采用方案 B。

题目将实现为一个最小 Flask 服务，访问首页时返回单页 HTML。服务端读取注入的 `FLAG` 环境变量，生成若干干扰变量和一个真实变量，将真实变量的值设为 `base64(base64(FLAG))`。选手唯一需要完成的动作是查看源码并完成双层解码。

## 题目定位

- 建议 slug：`web-source-audit-double-wrap-01`
- 建议标题：`Web-01 源码审计：双层伪装`
- 分类：`web`
- 难度：`easy`
- 分值：`100`
- flag 类型：`dynamic`
- 运行方式：`container`

该题面向首次接触 Web 源码审计的新手，训练点是：

- 识别源码中真假线索
- 对可疑编码内容继续递归分析，而不是只解一层
- 理解“查看源码”也是 Web 题的重要入口

## 题目包边界

建议按 `challenge-pack-v1` 落在独立题目包目录，结构最小化：

```text
docs/challenges/packs/web-source-audit-double-wrap-01/
  challenge.yml
  statement.md
  docker/
    Dockerfile
    requirements.txt
    app.py
```

不包含附件、不包含 writeup、不包含多节点拓扑。

## 页面与源码设计

### 页面外观

页面保持极简，只展示一张说明卡片和少量正文，例如“欢迎来到源码审计训练场，请从页面表面开始观察”。页面不提供输入框、按钮、下载链接或额外交互，避免选手误判题型方向。

### HTML 注释层

在页面源码中放置 2 到 3 条明显的假 flag 注释，以及 1 条提示继续深入查看的注释。例如：

- 假 flag 格式看起来可信，但内容明确错误
- 提示语暗示“真正内容更深”

这些注释负责把选手引导到“继续看源码，而不是停留在第一层注释”。

### JavaScript 干扰层

脚本中生成一批 `_0x...` 风格变量，数量建议控制在 6 到 12 个之间，既形成搜索感，又不至于让入门用户被噪音淹没。大多数变量值为单层 Base64，解开后仅得到干扰字符串，例如：

- `NOT_THE_FLAG_XX`
- `KEEP_LOOKING`
- `ALMOST_THERE`

这样选手会自然注意到某个变量的值更像“还需要继续处理”的编码串。

### 真线索层

保留一个真正目标变量，命名继续使用 `_0x...` 风格，但内容不沿用原题。其值由服务端在请求时生成：

1. 读取运行时 `FLAG`
2. 做第一次 Base64 编码
3. 对第一次结果再做第二次 Base64 编码
4. 将最终字符串放入目标变量

脚本尾部再加一条简短提示注释，可使用单层 Base64 包装，解出后告诉选手“仔细检查可疑变量”之类的信息。

## 改编策略

为确保这是本地改编题而不是直接照搬，必须做以下变化：

- 改写标题、题面文案和页面提示语
- 重写所有假 flag、假注释和干扰变量内容
- 不使用用户提供的静态最终值
- 不复用原题整段 HTML 结构和变量集合
- 保留同类型机制，但让本地题拥有独立的文本、噪音分布和变量布局

允许保留“`_0x...` 命名 + 双层 Base64”这一核心机制，因为它正是本题训练目标。

## 服务端行为

Flask 服务仅提供一个首页路由：

- `GET /`：返回题目页面 HTML

服务端职责：

- 从环境变量读取 `FLAG`
- 若 `FLAG` 缺失，使用仅供本地开发启动的安全占位值，避免服务直接崩溃
- 在模板或字符串渲染阶段构造干扰变量与真实变量
- 输出完整 HTML 页面

不提供额外 API，不做状态写入，不引入 session 或数据库。

该占位值只用于本地未注入环境的开发启动，不作为动态 flag 验收输入。

## 题面说明

`statement.md` 应传达的信息只有三类：

- 这是源码审计入门题
- 目标是从页面源码中找到隐藏 flag
- 提示可疑变量和编码可能不止一层

提示不能写到“对某个变量做两次 Base64 解码”这么直接，否则题目会退化成机械操作。

## 运行时与镜像要求

`challenge.yml` 应符合当前 `challenge-pack-v1` 规范：

- `api_version: v1`
- `kind: challenge`
- `meta.slug: web-source-audit-double-wrap-01`
- `meta.title: Web-01 源码审计：双层伪装`
- `meta.category: web`
- `meta.difficulty: easy`
- `meta.points: 100`
- `content.statement: statement.md`
- `flag.type: dynamic`
- `runtime.type: container`
- `runtime.image.ref` 指向本题构建后的镜像引用

Docker 镜像使用最小 Python Web 运行时即可，不需要额外系统依赖。

`docker/` 目录在本题中作为作者侧源码交付和审计复现材料保留；平台运行时仍以 `runtime.image.ref` 指向的最终镜像为准。

## 验收标准

实现完成后至少满足以下条件：

1. 题目容器可以在本地正常启动并访问首页
2. 页面表面仅表现为静态说明页，不暴露直接答案
3. 查看页面源码能稳定看到假线索、干扰变量和真实目标变量
4. 对真实目标变量连续做两次 Base64 解码后，结果等于当前实例注入的动态 flag
5. 不存在其他更简单的非预期解法，例如直接在页面正文、接口返回或错误日志中泄露明文 flag

## 风险与控制

- 如果提示写得过直白，题目会失去源码审计训练价值，因此题面和源码提示只给方向，不点名具体变量
- 如果干扰变量过多，入门用户会被噪音淹没，因此变量数量应保持克制
- 如果把真实 flag 静态写入仓库，将违反动态题要求，因此真实值只能来自运行时环境变量
- 如果缺少本地开发占位值，未注入 `FLAG` 时服务会直接失效，因此需要安全兜底，但占位值必须显式标注仅供开发测试

## 计划输入约束

后续实现计划应只覆盖这一道单页容器题的题目包创建，不扩展到题库批量改造、导入器增强或其它 Web 入门题系列化工作。
