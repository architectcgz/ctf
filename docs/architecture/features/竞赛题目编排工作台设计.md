## 1. 背景与问题

旧稿把竞赛编排写成了一个“五阶段单页运营台”的方案，但当前代码已经把“赛事编辑”与“运行态运维”拆成两条链。这里需要说明的不是理想形态，而是当前真实工作台如何分层：

- `ContestEdit` 负责赛事配置和开赛前收口
- `AWDOperationsPanel` 负责运行态巡检和实例编排
- AWD 题目编排在 UI 上复用目录式题目工作台，但底层事实源已经从 `contest_challenges` 分离到 `contest_awd_services`

## 2. 架构结论

- 当前竞赛编辑工作台入口仍是 `ContestEdit.vue`。
- `ContestEdit` 当前只包含四个 stage：
  - `basics`
  - `pool`
  - `awd-config`
  - `preflight`
- 旧稿中的“轮次运行”并没有并入 `ContestEdit`；当前运行态能力仍在独立的 `AWDOperationsPanel` 中承载。
- `pool` stage 负责目录式题目编排；普通赛使用 `contest_challenges`，AWD 赛在显示层复用同一目录 UI，但数据来自 `contest_awd_services`。
- `preflight` 对应 readiness 审计，不等于运行态控制面。
- 当前没有第二个“AWD 专用题目池”；AWD 服务编排是在赛事内的独立 service 配置层，不是另一套 challenge relation 表。

## 3. 模块边界与职责

### 3.1 模块清单

- `useContestWorkbench`
  - 负责：根据赛事模式决定可见 stage、默认 stage 和摘要信息
  - 不负责：加载 AWD service 数据

- `ContestEdit.vue`
  - 负责：承接赛事编辑主页面与 stage 切换
  - 不负责：运行态实例运维

- `useContestEditAwdWorkspace`
  - 负责：并行加载 `listContestAWDServices` 与 `getContestAWDReadiness`，管理 AWD 配置对话框和 preflight 跳转
  - 不负责：轮次态势或实例编排

- `ContestChallengeOrchestrationPanel`
  - 负责：统一渲染题目目录、筛选和单项编排入口
  - 不负责：区分底层 relation 表

- `mapPlatformContestAwdServicesToChallengeLinks`
  - 负责：把 `contest_awd_services` 映射成共享目录面板可消费的 `challengeLinks`
  - 不负责：保存 service 配置

- `AWDOperationsPanel` / `useContestOperationsPage`
  - 负责：运行态 readiness、轮次态势和实例编排
  - 不负责：赛事基础信息或题目目录编辑

### 3.2 事实源与所有权

- 普通赛题目关系事实源：`contest_challenges`
- AWD 服务编排事实源：`contest_awd_services`
- 赛前 readiness 事实源：`GET /admin/contests/:id/awd/readiness`
- 运行态巡检与实例编排事实源：`AWDOperationsPanel` 对应的运行态接口

## 4. 关键模型与不变量

### 4.1 核心实体

- `ContestWorkbenchStageKey`
  - 当前只允许：
    - `basics`
    - `pool`
    - `awd-config`
    - `preflight`

- `ContestChallenge`
  - 表示普通竞赛与题目的关系层

- `ContestAWDService`
  - 关键字段：`contest_id`、`awd_challenge_id`、`display_name`、`order`、`is_visible`、`runtime_config`、`validation_state`

### 4.2 不变量

- AWD 赛事在编辑期会同时出现 `pool / awd-config / preflight` 三段；普通赛只显示基础信息和题目编排。
- `ContestEdit` 当前不承载运行态“轮次运行”页面。
- AWD 目录展示可以复用 `ContestChallengeOrchestrationPanel`，但其 challenge links 不是直接读 `contest_challenges`。
- preflight 负责 readiness 审计与阻塞项定位，不直接替代 `AWDOperationsPanel` 的运行态视图。
- 当前不新增第二个 AWD 题目池。

## 5. 关键链路

### 5.1 赛事编辑链路

1. 管理员进入 `ContestEdit`。
2. `useContestWorkbench` 根据赛事模式计算可见 stage。
3. 若为 AWD 赛事，默认会暴露 `awd-config` 与 `preflight`。
4. `useContestEditAwdWorkspace.refreshAwdWorkbenchData` 并行请求：
   - `GET /admin/contests/:id/awd/services`
   - `GET /admin/contests/:id/awd/readiness`
5. 前端把 service 列表映射成共享目录项，再在 `awd-config` 和 `preflight` 中分别消费。

### 5.2 运行态链路

1. 管理员进入独立的 AWD 运行工作台。
2. `useContestOperationsPage` 按 runtime stage ready 与否决定面板内容。
3. `AWDOperationsPanel` 当前有两个操作面：
   - `轮次态势`
   - `实例编排`
4. readiness 未 ready 时，面板只保留 readiness / inspector 的受限视图，不开放完整实例编排。

## 6. 接口与契约

### 6.1 编辑期接口

- `GET /admin/contests/:id/awd/readiness`
- `GET /admin/contests/:id/awd/services`
- `POST /admin/contests/:id/awd/services`
- `PUT /admin/contests/:id/awd/services/:sid`
- `DELETE /admin/contests/:id/awd/services/:sid`

### 6.2 运行期接口

- `GET /admin/contests/:id/awd/instances`
- `POST /admin/contests/:id/awd/instances`
- `POST /admin/contests/:id/awd/instances/prewarm`

当前语义约定：

- `POST /admin/contests/:id/awd/instances`
  - 负责：单个队伍 × 服务实例启动与赛前失败单元重试
  - 不负责：整队或整场批量预热
- `POST /admin/contests/:id/awd/instances/prewarm`
  - 负责：`registration` 阶段的单队 / 整场批量预热
  - 不负责：比赛中的全量重启或运行中批量恢复

## 7. 兼容与迁移

- 旧稿里的“五段完整运营台”不是当前实现事实。
- 当前产品真实结构是“一个竞赛编辑工作台 + 一个独立 AWD 运行工作台”，不是单页贯通所有运营动作。
- 题目编排层继续保留共享目录式 UI，避免为 AWD 再复制一套完全不同的题目目录。
- 若未来真的要把运行态并回 `ContestEdit`，应新开专题，而不是把旧稿方案直接写回本文。
