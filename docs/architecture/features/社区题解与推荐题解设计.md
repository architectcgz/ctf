## 1. 背景与问题

旧稿仍把这个专题写成“从官方题解 + 学生提交 + 教师评阅演进到社区模型”的方案，但当前代码已经完成模型切换。现在需要明确的，是当前题解区到底围绕什么对象运转：

- 学生当前看到的是“推荐题解 + 社区题解 + 我的题解”
- 教师和管理员承担的是运营和筛选角色，不再承担题解评阅角色
- 练习判题里的 `manual_review_submission` 仍然是另一条业务链，不能和社区题解混写

## 2. 架构结论

- 挑战详情页当前正式题解模型由三部分组成：
  - 推荐题解
  - 社区题解
  - 我的题解
- 题解展示入口在挑战详情页的 `题解` tab，学生个人投稿入口在独立的 `编写题解` tab。
- 推荐题解和社区题解正文只对已解出该题的学生开放；后端统一通过 `ensureSolvedChallengeVisible` 校验。
- 学生每题当前只有一篇 `submission_writeup`，唯一键是 `(user_id, challenge_id)`。
- 学生题解当前真实状态只有：
  - `draft`
  - `published`
- `submitted` 仍作为兼容别名保留，但当前实际等同于 `published`。
- 教师 / 管理员可以对社区题解执行：
  - 推荐
  - 取消推荐
  - 隐藏
  - 恢复
- 官方题解 `challenge_writeup` 仍然存在，并且也可以进入“推荐题解”集合。

## 3. 模块边界与职责

### 3.1 模块清单

- `useChallengeDetailPage`
  - 负责：驱动挑战详情页的 `题解` / `编写题解` tab，加载推荐题解、社区题解和我的题解
  - 不负责：决定解题可见性

- `ChallengeSolutionsPanel`
  - 负责：展示推荐题解和社区题解列表，以及“未解题不可查看”的前端提示
  - 不负责：绕过后端权限校验

- `challenge/application/queries/WriteupService`
  - 负责：查询推荐题解、社区题解、教师题解投稿目录，并统一做 solved gate
  - 不负责：写入投稿或运营状态

- `challenge/application/commands/WriteupService`
  - 负责：学生投稿 upsert、官方题解推荐、社区题解推荐/隐藏/恢复
  - 不负责：练习判题中的人工评审

- `ChallengeWriteupManagePanel`
  - 负责：教师 / 管理员查看当前题目的官方题解与学员题解目录
  - 不负责：学生侧题解阅读体验

### 3.2 事实源与所有权

- 学生投稿事实源：`submission_writeups`
- 官方题解事实源：`challenge_writeup`
- 学生题解列表与权限事实源：`WriteupService` query layer
- 运营动作事实源：`WriteupService` command layer

## 4. 关键模型与不变量

### 4.1 核心实体

- `SubmissionWriteup`
  - 关键字段：`user_id`、`challenge_id`、`title`、`content`、`submission_status`、`visibility_status`、`is_recommended`、`published_at`

- `RecommendedChallengeSolutionResp`
  - 当前可来源于：
    - 官方题解
    - 社区题解

- `CommunityChallengeSolutionResp`
  - 当前只收录已发布且可见的社区题解

### 4.2 不变量

- 每个用户对同一题目当前只能维护一篇投稿。
- `published` 投稿必须在用户已解出题目后才能创建或更新为公开态。
- 隐藏的社区题解不能再被设为推荐。
- 题解区正文读取必须先通过 solved gate；前端提示只是辅助手段，不是唯一权限边界。
- `manual_review_submissions` 仍独立存在，当前没有被题解社区模型替代。

## 5. 关键链路

### 5.1 学生阅读链路

1. 学生进入挑战详情页。
2. `useChallengeDetailPage` 提供 `题解` 和 `编写题解` 两个 workspace tab。
3. 学生进入 `题解` tab 时，请求：
   - `GET /challenges/:id/solutions/recommended`
   - `GET /challenges/:id/solutions/community`
4. 若学生未解题，后端拒绝正文查询，前端显示“解出题目后可查看推荐题解与社区题解”。

### 5.2 学生投稿链路

1. 学生在 `编写题解` tab 中编辑自己的 title / content。
2. 前端调用 `POST /challenges/:id/writeup-submissions`。
3. 后端按 `(user_id, challenge_id)` upsert 投稿。
4. 投稿状态为 `draft` 时仅保存草稿；为 `published` 时还会校验用户已解题。

### 5.3 教师运营链路

1. 教师或管理员查看题目管理页中的 `ChallengeWriteupManagePanel`，或教师题解投稿目录 `/teacher/writeup-submissions`。
2. 后端通过社区题解运营接口执行推荐、取消推荐、隐藏、恢复。
3. 运营结果会回写 `submission_writeups` 的 `visibility_status / is_recommended / recommended_at / recommended_by`。

## 6. 接口与契约

### 6.1 学生接口

- `GET /challenges/:id/solutions/recommended`
- `GET /challenges/:id/solutions/community`
- `POST /challenges/:id/writeup-submissions`
- `GET /challenges/:id/writeup-submissions/me`

### 6.2 官方题解运营接口

- `GET /admin/challenges/:id/writeup`
- `PUT /admin/challenges/:id/writeup`
- `POST /admin/challenges/:id/writeup/recommend`
- `DELETE /admin/challenges/:id/writeup/recommend`

### 6.3 社区题解运营接口

- `POST /teacher/community-writeups/:id/recommend`
- `DELETE /teacher/community-writeups/:id/recommend`
- `POST /teacher/community-writeups/:id/hide`
- `POST /teacher/community-writeups/:id/restore`
- `GET /teacher/writeup-submissions`
- `GET /teacher/writeup-submissions/:id`

## 7. 兼容与迁移

- 旧的“教师评阅学生题解”口径不再是当前题解社区主模型。
- `submitted`、`reviewed` 等旧别名仍在模型常量里保留兼容，但当前正式状态和可见性已经收敛为 `draft / published` 与 `visible / hidden`。
- 官方题解没有消失，而是从“单独一块内容”变成了“推荐题解”来源之一。
- `manual_review_submission` 继续用于练习人工判题，不能被写成“题解社区替代了人工评审”。

## 9. 验证标准

- 已解题用户可以查看推荐题解和社区题解；未解题用户只能看到受限提示。
- 学生每题只有一篇投稿，重复保存走 upsert，不会生成多篇并行记录。
- 学生题解状态当前只表现为 `draft / published`。
- 社区题解支持推荐、取消推荐、隐藏和恢复；隐藏题解不能继续推荐。
- 练习人工评审接口 `manual-review-submissions` 仍作为独立链路保留。
