# 社区题解与推荐题解设计

## 目标

把当前“官方题解 + 学生提交题解 + 教师评阅”的工作流，重构为“官方题解 + 社区题解 + 推荐题解”的内容社区模型。

首版目标是让挑战详情页具备类似 LeetCode 的题解区域，但保持 CTF 场景约束：

- 只有解出题目的学生才能查看推荐题解和社区题解正文
- 学生每题只能维护 1 篇自己的题解
- 学生题解默认可公开进入社区
- 教师/管理员不再评阅学生题解
- 教师/管理员可以隐藏、恢复、推荐、取消推荐题解
- 推荐题解既可以来自官方题解，也可以来自学生社区题解

## 背景与现状

当前系统已经存在两类相关内容：

- `challenge_writeup`
  - 代表官方题解，由管理员在后台编辑并控制可见性
- `submission_writeups`
  - 代表学生题解提交，当前语义是草稿/提交 + 教师评阅

现有问题：

- 挑战页中“官方题解”和“个人 writeup”是两套彼此割裂的 UI
- 学生题解被设计成教学评阅材料，而不是社区内容
- 教师分析页和接口围绕“评语/评阅状态”组织，与社区题解目标不一致
- 没有“推荐题解”概念，无法把优秀学生题解或官方题解置顶展示

## 替代关系

本设计替代旧的“选手 Writeup 提交与评阅设计”中“教师评阅学生 writeup”的部分。旧设计文件已移除，避免继续把教师评阅流当成当前产品方向。

具体替代内容：

- 移除教师评阅流
- 移除 `review_comment`、`reviewed_at`
- 不再保留“reviewed / excellent / needs_revision”一类评阅状态
- 教师角色从“批改者”改为“内容运营者”

## 设计范围

### 包含

- 挑战页新增 LeetCode 风格题解区域
- 官方题解与社区题解统一进入“题解区”展示
- 学生每题维护 1 篇社区题解
- 社区题解支持草稿、发布、撤回、编辑
- 教师/管理员对官方题解和社区题解执行推荐管理
- 教师/管理员对社区题解执行隐藏/恢复
- 仅在学生解题成功后开放题解正文
- 后端接口、数据迁移、权限与测试更新

### 不包含

- 点赞
- 评论
- 收藏
- 关注作者
- 多篇题解版本树
- 富文本协同编辑

## 产品设计

### 1. 挑战页题解区

挑战详情页在题目正文之后显示独立的“题解”模块。

未解题用户：

- 可看到题解区入口
- 可看到摘要信息，例如“推荐 2 篇 · 社区 8 篇”
- 不可看到任何题解正文、列表细节
- 显示提示：“解出题目后可查看推荐题解与社区题解”

已解题用户：

- 可看到完整题解区
- 题解区采用 3 个页签：
  - `推荐题解`
  - `社区题解`
  - `我的题解`

### 2. 推荐题解

推荐题解区展示被教师/管理员手动标记为推荐的内容。

规则：

- 来源可以是官方题解，也可以是学生社区题解
- 推荐内容按推荐时间倒序展示
- 首版建议最多展示 3 篇，超出时仍允许列表浏览，但页面默认突出前 1 到 3 篇

推荐卡片字段：

- 标题
- 作者名
- 来源标识：`官方` 或 `社区`
- 更新时间
- 摘要

展开后展示完整正文。

### 3. 社区题解

社区题解区展示所有“已发布 + 未隐藏”的学生题解。

规则：

- 只有已解题用户可见
- 支持按标题或作者搜索
- 支持排序：
  - `最新`
  - `最早`

首版不做热度排序，因为没有点赞模型。

列表字段：

- 标题
- 作者
- 发布时间
- 摘要
- 是否推荐

### 4. 我的题解

每个学生在每道题下最多维护 1 篇自己的题解。

状态：

- `draft`
- `published`
- `hidden`

允许操作：

- 保存草稿
- 发布
- 更新
- 撤回为草稿

`hidden` 语义：

- 表示题解已被教师/管理员隐藏
- 学生仍能看到自己的内容，但不能让其他学生看到
- 页面需明确提示“当前题解已被隐藏”

### 5. 教师/管理员后台

后台操作从“评阅”改成“内容运营”。

管理员题解页保留官方题解编辑能力，并新增社区题解管理区：

- 官方题解编辑
- 社区题解管理

社区题解管理列表字段：

- 标题
- 作者
- 发布时间
- 当前状态
- 是否推荐
- 来源

操作：

- `推荐`
- `取消推荐`
- `隐藏`
- `恢复`

教师分析页不再显示：

- 待评阅
- 评语
- excellent / needs_revision 一类状态

教师分析页中的 writeup 区块调整为学生社区题解概览，只显示：

- 是否发布
- 是否隐藏
- 是否推荐
- 最近更新时间

## 数据模型设计

### 1. 官方题解

保留现有官方题解模型，并新增推荐字段：

- `is_recommended`
- `recommended_at`
- `recommended_by`

现有可见性仍保留：

- `private`
- `public`
- `scheduled`

是否在前台可见仍由当前发布逻辑控制。

### 2. 社区题解

复用 `submission_writeups` 表，但把语义从“评阅对象”改为“社区题解”。

建议字段调整如下：

- 保留
  - `id`
  - `user_id`
  - `challenge_id`
  - `contest_id`
  - `title`
  - `content`
  - `created_at`
  - `updated_at`
- 修改
  - `submission_status`：从 `draft/submitted` 调整为 `draft/published`
- 新增或重命名为社区语义
  - `visibility_status`
    - `visible`
    - `hidden`
  - `is_recommended`
  - `recommended_at`
  - `recommended_by`
  - `published_at`
- 删除
  - `review_status`
  - `reviewed_by`
  - `reviewed_at`
  - `review_comment`

备注：

- 若当前实现不便直接重命名 `review_status`，可先通过迁移替换成新字段，再更新代码，避免保留误导性命名
- `contest_id` 继续保留为可空字段，不作为本次产品主流程的一部分

### 3. 推荐聚合视图

前台推荐题解区需要统一展示官方题解与社区题解，因此查询层应输出统一 DTO，而不是让前端拼接两种不同结构。

建议新增统一响应结构：

- `id`
- `source_type`
  - `official`
  - `community`
- `source_id`
- `challenge_id`
- `title`
- `content`
- `author_name`
- `is_recommended`
- `recommended_at`
- `updated_at`

## 权限与可见性

### 学生

- 可创建、编辑自己的社区题解
- 可把自己的题解保存为草稿
- 可发布自己的题解
- 可把自己的已发布题解撤回为草稿
- 可查看自己被隐藏的题解内容
- 只有在自己已解出题目后，才能查看推荐题解和社区题解正文

### 教师/管理员

- 可查看社区题解列表
- 可隐藏/恢复社区题解
- 可推荐/取消推荐社区题解
- 可推荐/取消推荐官方题解
- 不再对学生题解写评语或标记评阅结果

### 未解题学生

- 不可查看推荐题解和社区题解正文
- 可编辑自己的草稿题解
- 是否允许未解题前发布自己的社区题解：
  - 首版建议允许保存草稿
  - 仅在解题成功后允许正式发布

这样可避免学生在未完成题目时提前公开剧透内容。

## API 设计

### 保留并调整

- `GET /api/v1/challenges/:id/writeup`
  - 获取官方题解
  - 首版可保留，供统一题解区内部调用
- `GET /api/v1/challenges/:id/writeup-submissions/me`
  - 获取我的社区题解
- `POST /api/v1/challenges/:id/writeup-submissions`
  - 保存或更新我的社区题解
  - 请求体状态改为 `draft` / `published`

### 新增

- `GET /api/v1/challenges/:id/solutions/recommended`
  - 获取当前题目的推荐题解列表
- `GET /api/v1/challenges/:id/solutions/community`
  - 获取当前题目的社区题解列表
  - 支持 `q`, `sort`, `page`, `page_size`
- `POST /api/v1/authoring/challenges/:id/writeup/recommend`
  - 官方题解设为推荐
- `DELETE /api/v1/authoring/challenges/:id/writeup/recommend`
  - 取消官方题解推荐
- `POST /api/v1/teacher/community-writeups/:id/recommend`
  - 学生社区题解设为推荐
- `DELETE /api/v1/teacher/community-writeups/:id/recommend`
  - 取消推荐
- `POST /api/v1/teacher/community-writeups/:id/hide`
  - 隐藏社区题解
- `POST /api/v1/teacher/community-writeups/:id/restore`
  - 恢复社区题解

备注：

- 如果团队更偏好 REST 统一风格，也可把推荐/隐藏改成 `PATCH` 更新状态；首版重点是语义清晰，不强制绑定某种接口风格

## 数据迁移策略

需要新增数据库迁移，将旧评阅模型转换为社区模型。

迁移要求：

- 删除 `review_comment`
- 删除 `reviewed_at`
- 删除 `reviewed_by`
- 删除旧 `review_status`
- 把 `submission_status = submitted` 的记录迁移为 `published`
- 把 `submission_status = draft` 的记录保留为 `draft`
- 为所有历史记录补齐：
  - `visibility_status = visible`
  - `is_recommended = false`

关于历史“优秀题解”的处理：

- 如果数据库中存在 `excellent` 历史状态且业务认为仍有价值，可在迁移中映射为 `is_recommended = true`
- 如果不想自动推荐历史内容，则统一不迁推荐位，由教师/管理员重新挑选

首版建议默认不自动推荐，避免旧教学评价直接变成前台推荐内容。

## 错误处理

### 学生侧

- 未解题访问推荐题解或社区题解正文时，返回权限错误或空内容，并显示明确提示
- 发布空标题或空内容时，继续做表单校验
- 学生尝试修改他人题解时，返回权限错误

### 教师/管理员侧

- 推荐不存在的内容时返回 404
- 对已隐藏内容重复隐藏时，接口应幂等
- 对已推荐内容重复推荐时，接口应幂等

## 风险与取舍

### 1. 复用旧表而不是新建新表

优点：

- 迁移成本低
- 现有接口与测试可局部重用

风险：

- 旧命名容易残留误导

处理方式：

- 尽量在数据库迁移和 DTO 层完成字段重命名，避免代码里继续保留 review 语义

### 2. 未解题前是否允许发布

首版建议：

- 允许写草稿
- 不允许发布

原因：

- 这更符合 CTF 场景
- 能显著降低剧透风险

### 3. 推荐题解是否自动继承历史 excellent

首版建议不自动继承，由教师/管理员重新挑选。

原因：

- “教学上优秀”不等于“适合作为公共推荐题解”
- 避免旧数据直接污染前台体验

## 验收标准

- 挑战页存在统一题解区，支持推荐题解、社区题解、我的题解
- 学生只有在解题成功后才能查看推荐题解与社区题解正文
- 学生每题只维护 1 篇题解，支持草稿与发布
- 教师/管理员可以推荐或隐藏题解
- 官方题解与社区题解都可以成为推荐题解
- 系统中不再存在学生题解评语与评阅状态流程
- 相关后端、前端、迁移、测试全部更新
