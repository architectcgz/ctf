# CTF 前端试探型彩蛋设计

## 目标

在不影响 CTF 平台主功能、不改变真实业务反馈的前提下，为前端补一组面向“试探型用户”的轻量彩蛋。

本轮设计要解决的问题：

- 当前登录页已经有控制台提示，但只停留在单点梗，没有形成全站一致的彩蛋语言
- 有些学生会主动试探登录页、错误页、通知页和题目页的边缘区域，前端可以对这种行为做轻反馈
- 彩蛋需要有一点“系统正在看着你”的调性，但不能伪装成真实漏洞、真实风控或真实系统故障
- 这组能力应尽量复用现有 Vue 页面结构，不引入大而重的“隐藏玩法系统”

## 非目标

- 不改真实登录、权限、路由守卫、接口返回和错误状态
- 不实现 ARG 式长链谜题，不把彩蛋做成新的功能入口
- 不依赖后端计数、后端开关或真实鉴权失败次数
- 不在主操作区、主按钮和关键工作流上挂彩蛋
- 不把任何设计说明文案直接渲染成用户必须阅读的正式页面内容

## 当前背景

当前前端已经具备几个适合承接彩蛋的现成落点：

1. 登录页已有控制台提示
   - `LoginView.vue` 已在 `onMounted` 输出控制台文案
   - 当前调性是半认真、半告诫，已经接近本轮目标

2. 错误页有统一壳组件
   - `ErrorStatusShell.vue` 已统一承接 404/403/429/5xx 页
   - 这类页面天然适合接住“路径试探”用户

3. 通知页已有元信息展示
   - `NotificationList.vue` 有显式刷新动作
   - `NotificationDetail.vue` 有独立 `ID` 卡片和元信息区

4. 题目详情页具备非关键交互区
   - `ChallengeDetail.vue` 中的分值侧栏、标签区、提示区都可以做轻触发
   - 这些区域不属于 flag 提交、附件下载或主 tab 切换等关键链路

5. 会话状态由轻量 composable 承接
   - 当前实现使用 `useProbeEasterEggs.ts` 读写 `sessionStorage`
   - 页面各自接入触发和局部反馈，没有在 `AppLayout.vue` 中做全局挂载

## 决策摘要

采用“分布式、低信号、前端本地触发”的方案：

- 彩蛋散布在登录页、错误页、通知页、题目详情页和全局壳
- 每个彩蛋都只对主动试探的人产生反馈
- 所有状态只存浏览器会话本地，优先使用 `sessionStorage`
- 反馈形式限定为控制台输出、轻 toast、局部附注、小条幅或卡片短暂翻面
- 任何彩蛋触发都不改变当前页面的真实业务行为

## 方案比较

### 方案 A：静态散点型

做法：

- 只在控制台、注释、轻微文案细节里埋梗
- 不做点击序列或前端本地状态

优点：

- 风险最低
- 不需要额外状态模型

缺点：

- 互动性太弱
- 很难形成“系统在回应你的试探”这层体验

不采用。

### 方案 B：分布式互动型

做法：

- 在全站 5 到 6 个落点挂极轻的本地触发
- 用会话内状态记录是否触发过某类彩蛋
- 在少数页面给出后续呼应

优点：

- 最贴合“试探型用户”的行为路径
- 互动性够用，但不会像独立玩法
- 可以严格控制风险边界

缺点：

- 需要补一个很薄的本地状态 composable

推荐采用。

### 方案 C：轻 ARG 串线型

做法：

- 把多个页面的线索串成一条隐藏支线
- 触发点和反馈跨页强关联

优点：

- 可玩性最高

缺点：

- 复杂度和维护成本明显更高
- 容易被误读为正式功能或真实隐藏入口

本轮不采用。

## 彩蛋语言与体验原则

### 1. 语气

统一采用“半认真半调侃”的系统口吻：

- 不幼稚
- 不嘲讽学生
- 更像系统管理员、值守脚本或日志注释在做轻回应

### 2. 可见性

彩蛋应该是低信号反馈：

- 正常使用流程下基本看不到
- 有明确试探行为时才会出现
- 出现后很快消失，或只在局部停留

### 3. 安全边界

以下内容明确禁止：

- 模拟账号锁定、风控命中、权限升级成功、数据库错误等假状态
- 修改登录成功失败文案
- 让用户误以为发现了真实漏洞
- 让错误页、通知页或题目页表现得像系统异常

## 彩蛋清单

### 1. 登录页控制台扩展

触发方式：

- 用户打开登录页并查看控制台即可看到

反馈形式：

- 在现有控制台输出后追加 1 到 2 条审计口吻提示

设计目的：

- 给会先看控制台的人一个明确回应
- 延续当前登录页已有调性，不额外造风格

### 2. 登录页品牌区连点

触发方式：

- 用户连续点击登录页左侧品牌标题区或装饰条

反馈形式：

- 登录面板边缘出现一条 2 到 3 秒后自动消失的小提示

设计目的：

- 回应“视觉区是不是藏了入口”的试探行为
- 保证提示不遮挡表单、不干扰提交

### 3. 错误页路径试探回应

触发方式：

- 用户进入 404 页后，对状态码或标题区域做短点击序列

反馈形式：

- 描述文案下方临时出现一条浅色附注，或 kicker 短暂切换

设计目的：

- 接住改地址、猜路由、撞路径的人
- 保持错误页仍然是一张正常的错误页

### 4. 通知详情 ID 区块连击

触发方式：

- 用户连续点击通知详情页右侧 `ID` 区块

反馈形式：

- ID 卡片短暂翻成“值守备注”样式，再恢复原状

设计目的：

- 营造“从系统元信息里撬出一角后台口风”的感觉
- 不修改实际通知内容

### 5. 通知列表高频刷新吐槽

触发方式：

- 用户在短时间内连续点刷新按钮数次

反馈形式：

- toast 或标题下方短提示

设计目的：

- 回应“疯狂刷新能不能刷出隐藏内容”的试探行为
- 不改变真实刷新逻辑

### 6. 题目详情非关键区试探反馈

触发方式：

- 用户在分值侧栏、标签区或提示标题区域做轻量组合点击

反馈形式：

- 局部出现一句短提示，自动消失

设计目的：

- 把彩蛋放在做题用户会注意、但不是做题关键动作的位置
- 明确避开附件下载、题解区、主 tab 和提交链路

## 信息架构与页面落点

### 1. 登录页

保留现有主结构：

- `AuthEntryShell.vue` 继续承担品牌区和表单区
- `LoginView.vue` 继续承担控制台输出和提交逻辑

新增内容：

- 品牌区轻点击触发
- 面板边缘的轻提示槽位
- 扩展控制台提示

这里不引入任何和登录失败次数绑定的彩蛋，避免和真实鉴权反馈混淆。

### 2. 错误页

保留现有结构：

- `NotFoundView.vue` 等错误页仍只负责传入文案和动作
- `ErrorStatusShell.vue` 作为统一承载层

新增内容：

- 对标题区或状态码区的点击序列监听
- 一条很轻的附注槽位

错误页的返回、刷新和回到主页逻辑不受影响。

### 3. 通知列表与详情

列表页：

- 只在现有刷新按钮上记录本地频率
- 超过阈值后展示一句轻提示

详情页：

- 只在右侧 ID 卡片区域做连击触发
- 反馈在卡片本体内部完成，不扩散到正文

### 4. 题目详情页

放置原则：

- 只放在非关键区
- 只做局部提示
- 绝不接入题目数据或提交流程

推荐触发区：

- 分值侧栏
- 标签区
- 提示区标题

不推荐触发区：

- 附件下载按钮
- 题解分类 tab
- 题目主 tab
- 任何提交入口

### 5. 会话状态

当前实现没有在 `AppLayout.vue` 中加入主动逻辑。跨页状态由 `useProbeEasterEggs.ts` 使用 `sessionStorage` 保存，每个页面只接入自己的触发器和局部反馈。

## 前端结构设计

### 1. 状态模型

新增一个轻量 composable，例如：

- `useProbeEasterEggs.ts`

职责只包括：

- 记录每类彩蛋的本地计数
- 判断是否达到触发阈值
- 生成一次性可消费信号
- 记录本会话是否已经展示过某条反馈

不负责：

- 管业务数据
- 调接口
- 接管页面真实交互

### 2. 存储策略

优先使用 `sessionStorage`：

- 同一轮试探可跨页留痕
- 关闭标签页后自然清空
- 不给用户留下长期“被记录”的感觉

若 `sessionStorage` 不可用：

- 直接静默降级为内存态或完全禁用彩蛋
- 不影响页面主功能

### 3. 页面接入方式

每个页面只接自己的触发器和展示：

- 登录页：品牌区连点、控制台扩展
- 错误页：点击序列
- 通知列表：高频刷新计数
- 通知详情：ID 连击
- 题目详情：非关键区组合点击

如果后续发现多个页面都需要同款局部提示，再补一个极薄的通用展示组件；否则直接写在页面里，保持最小 diff。

## 交互与反馈设计

### 1. 反馈形式优先级

优先使用以下轻反馈形式：

1. 控制台输出
2. 局部附注
3. 小条幅
4. 卡片短暂翻面
5. toast

原因：

- 这些形式对布局侵入最小
- 不会和业务成功/失败状态混淆

### 2. 展示时长

除控制台和彩蛋卡片翻面外，其余提示建议自动消失：

- 短提示：约 2 到 3 秒
- 翻面提示：约 1.5 到 2 秒后恢复

### 3. 文案要求

文案必须满足：

- 短句
- 不解释设计意图
- 不带“点击这里继续”“解锁下一步”这类功能暗示
- 不声称真实监控、真实封禁或真实漏洞存在

## 失败与降级策略

必须保证彩蛋逻辑出错时，页面仍按正常路径运行：

- 本地状态读写失败时，直接静默跳过
- 点击监听异常时，不阻断原事件
- 提示展示失败时，不影响真实按钮和真实路由
- 彩蛋状态永远不能成为页面渲染所必需的数据

## 测试设计

### 1. composable 单测

至少覆盖：

- 计数与阈值判断
- 一次性信号消费
- `sessionStorage` 读写
- 存储不可用时的降级行为

### 2. 页面测试

登录页：

- 触发品牌区彩蛋后仍可正常提交登录

错误页：

- 触发彩蛋后，返回上一页、刷新页面、回到首页仍正常

通知列表：

- 高频刷新提示不影响真实刷新流程

通知详情：

- ID 区块彩蛋不影响详情展示和返回列表

题目详情：

- 彩蛋不影响附件下载、tab 切换、题解展示和其它主流程

### 3. 手工验证

- 不触发彩蛋时，全站行为与当前版本一致
- 触发彩蛋后只出现轻反馈，不出现布局抖动、焦点丢失或主按钮状态异常
- 刷新页面或关闭标签页后，会话内状态符合设计预期

## 风险与约束

- 如果彩蛋挂在主操作区，误触会明显上升，因此必须严格限制落点
- 如果文案过于像真实告警，用户会误判系统状态，因此必须克制
- 如果把多页逻辑全塞进全局壳，会导致维护边界变差，因此只保留最薄的跨页状态

## 最小交付

第一版最小交付为：

- 登录页控制台扩展
- 登录页品牌区连点提示
- 404/错误页点击序列提示
- 通知详情 ID 连击反馈
- 通知列表高频刷新提示
- 题目详情非关键区轻反馈
- 一个会话内本地状态 composable

这版交付完成后，平台会形成一组统一但低存在感的“试探型彩蛋”，既能回应主动试探的学生，也不会干扰任何正常业务流程。
