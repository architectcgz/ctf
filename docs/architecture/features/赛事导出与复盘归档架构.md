## 1. 背景与问题

赛事运营和教学复盘需要稳定导出两类资产：

- 面向班级教学复盘的报告导出，供教师汇报、留档和继续分析使用。
- 面向赛事结果的结构化导出，供答辩展示、留档或后续分析使用。
- 面向单个学生的复盘归档，供教师回看时间线、证据、题解与人工审核记录。

这类能力如果每个场景都单独造一套导出表和轮询机制，会很快失控。当前需要收口的是：赛事导出和学生复盘归档都复用统一 `reports` 异步报告链路，但保持各自的数据构建边界。

## 2. 架构结论

- 班级报告、赛事导出与学生复盘归档统一复用 `reports` 表和报告下载链路。
- 当前新增的报告类型为：
  - `class`
  - `contest_export`
  - `review_archive`
- `class` 报告支持 `pdf / excel`；`contest_export` 与 `review_archive` 当前固定输出 `json`，不在本专题内扩展为完整备份系统。
- 班级报告预览与导出复用同一组 `from_date / to_date` contract，不再在班级详情页和导出弹窗各自维护第二套“近 7 天”口径。
- 学生复盘归档除了导出接口，还提供同步读接口 `GET /api/v1/teacher/students/:id/review-archive`，供页面直接渲染。
- 导出文件和同步读接口使用同一套归档构建逻辑，避免页面内容和导出结果分叉。

## 3. 模块边界与职责

### 3.1 模块清单

- `internal/module/assessment/application/commands/ReportService`
  - 负责：创建报告任务、权限校验、构建 class / contest export / review archive 数据、标记 `reports` 生命周期
  - 不负责：在前端轮询状态

- `internal/module/assessment/infrastructure/report_repository`
  - 负责：读取班级快照指标、分布统计、赛事迁移摘要、学生时间线、证据、题解、人工审核以及赛事导出所需的底层数据
  - 不负责：定义最终导出文件的任务状态

- `internal/module/teaching_query/infrastructure`
  - 负责：作为班级 `summary / trend / review` 的查询 owner，为班级报告导出提供当前窗口数据
  - 不负责：生成报告文件或维护 `reports` 状态

- `internal/module/contest`
  - 负责：提供比赛基础信息、榜单、题目和队伍相关事实
  - 不负责：维护报告下载生命周期

- `frontend/src/features/contest-workbench/model/useContestExportFlow.ts`
  - 负责：发起赛事导出、轮询报告状态、准备下载
  - 不负责：拼装导出数据

- `frontend/src/features/teacher-student-review-archive`
  - 负责：读取学生复盘归档、发起导出并轮询结果
  - 不负责：在前端重新聚合时间线与证据

- `frontend/src/widgets/teacher-review-archive`
  - 负责：展示归档的摘要、证据、题解与教师观察
  - 不负责：决定导出文件结构

### 3.2 事实源与所有权

- 报告任务事实源：`reports`
- 班级报告内容 owner：`ReportService.buildClassReportData`
- 赛事导出内容 owner：`ReportService.buildContestExportData`
- 学生复盘归档内容 owner：`ReportService.buildStudentReviewArchiveData`

## 4. 关键模型与不变量

### 4.1 报告生命周期模型

- `reports.type`
  - `class`
  - `contest_export`
  - `review_archive`
- `reports.format`
  - `pdf`
  - `excel`
  - `json`
- `reports.status`
  - `processing`
  - `ready`
  - `failed`

### 4.2 班级报告导出结构

- `class_name`
- `window`
- `total_students`
- `average_score`
- `dimension_averages`
- `top_students`
- `summary`
- `trend`
- `review`
- `category_distribution`
- `difficulty_distribution`
- `contest_migration`

其中：

- `summary / trend / review` 严格按当前时间窗统计
- `average_score / dimension_averages / top_students / category_distribution / difficulty_distribution / contest_migration` 仍表示当前班级能力快照，不回溯重算历史窗口

### 4.3 赛事导出 JSON 结构

- `generated_at`
- `contest`
- `scoreboard`
- `challenges`
- `teams`

### 4.4 学生复盘归档结构

- `generated_at`
- `student`
- `summary`
- `skill_profile`
- `timeline`
- `evidence`
- `writeups`
- `manual_reviews`
- `teacher_observations`

### 4.5 不变量

- 班级报告由教师或管理员发起；教师默认落到自己班级，管理员可以显式指定班级。
- 赛事导出只有管理员可以发起。
- 学生复盘归档由教师或管理员发起；教师只能访问自己班级学生。
- `POST /reports/class` 与 `GET /teacher/classes/:name/{summary,trend,review}` 共享同一组 `from_date / to_date` contract，不能出现页面预览与导出口径分叉。
- `GET /review-archive` 与 `POST /review-archive/export` 使用同一套归档构建逻辑，不能出现页面和导出内容口径不一致。
- `teacher_observations` 为派生字段，在读/导出时根据 `summary + evidence + writeups + manual_reviews` 计算，不单独持久化。
- 当前专题只覆盖结构化导出，不承担全量备份恢复语义。

## 5. 关键链路

### 5.1 赛事导出链路

1. 管理员通过 `POST /api/v1/admin/contests/:id/export` 发起导出。
2. `ReportService` 创建一条 `reports.type=contest_export`、`status=processing` 的记录。
3. 后台任务读取赛事基础信息、榜单、题目和队伍数据，生成 JSON 文件。
4. 报告状态更新为 `ready` 或 `failed`。
5. 前端通过统一报告轮询与下载接口拿到文件。

### 5.2 学生复盘归档读链路

1. 教师或管理员请求 `GET /api/v1/teacher/students/:id/review-archive`。
2. `ReportService` 做用户与班级权限校验。
3. `buildStudentReviewArchiveData` 读取：
   - 训练摘要
   - 技能画像
   - 时间线
   - 证据事件
   - 题解
   - 人工审核记录
4. 服务端派生 `summary` 和 `teacher_observations`，前端据此渲染归档页。

### 5.3 学生复盘归档导出链路

1. 教师或管理员请求 `POST /api/v1/teacher/students/:id/review-archive/export`。
2. `ReportService` 创建一条 `reports.type=review_archive` 的异步任务。
3. 后台任务复用 `buildStudentReviewArchiveData` 生成 JSON 文件。
4. 前端轮询 `GET /api/v1/reports/:id`，完成后调用 `GET /api/v1/reports/:id/download` 下载。

### 5.4 班级报告导出链路

1. 教师或管理员请求 `POST /api/v1/reports/class`，可选携带 `from_date / to_date`。
2. `classwindow.Parse(...)` 先归一时间窗：
   - 两个都不传：默认最近 `7` 个自然日
   - 只传一个：直接返回校验错误
   - 两个都传：按自然日闭区间统计，最大跨度 `31` 天
3. `ReportService` 创建一条 `reports.type=class` 的任务，并同时读取：
   - 班级能力快照指标
   - 当前窗口 `summary / trend / review`
   - 分类分布、难度分布和赛事迁移摘要
4. 导出层复用 `classreview` 返回的结构化 review items，不再额外维护第二套班级复盘拼装逻辑。
5. 前端预览与导出按钮共享同一时间窗；导出完成后继续通过统一 `GET /api/v1/reports/:id` 和下载接口取文件。

## 6. 接口与契约

### 6.1 对外接口

- `POST /api/v1/reports/class`
- `GET /api/v1/teacher/classes/:name/summary`
- `GET /api/v1/teacher/classes/:name/trend`
- `GET /api/v1/teacher/classes/:name/review`
- `POST /api/v1/admin/contests/:id/export`
- `GET /api/v1/teacher/students/:id/review-archive`
- `POST /api/v1/teacher/students/:id/review-archive/export`
- `GET /api/v1/reports/:id`
- `GET /api/v1/reports/:id/download`

### 6.2 前端消费契约

- 班级报告弹窗预览与导出消费同一组 `from_date / to_date`：
  - 页面 route query 是时间窗 owner
  - 导出弹窗只消费当前页面窗口
  - 后端仍是默认值和校验的唯一 owner
- 赛事导出发起端消费 `ReportExportData`，并通过统一轮询逻辑等待 `ready/failed`。
- 学生复盘归档页面直接消费 `ReviewArchiveData`，其中：
  - `summary` 用于概览区
  - `timeline/evidence` 用于证据面板
  - `writeups/manual_reviews` 用于反思与审核区
  - `teacher_observations.items` 用于观察条

## 7. 兼容与迁移

- 这三类导出都走统一 `reports` 表，不新增专用导出任务表。
- 当前 `class` 支持 `pdf / excel`，`contest_export` 与 `review_archive` 仅支持 `json`；若后续扩展 `zip/pdf/xlsx`，应继续复用 `reports` 生命周期和同一套数据构建逻辑。
- 本专题不把赛事导出定义为“完整系统备份”，后续若有数据库或对象存储级备份，应单独归入运维或备份专题。
