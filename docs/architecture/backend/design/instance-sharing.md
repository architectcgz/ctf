# 题目实例共享策略设计

> 状态：已采用

## 1. 设计概览

平台对可启动题目的实例分配采用题目级策略控制，后端按题目配置决定实例的创建作用域、复用规则、可见性和管理权限。

当前固定支持三种实例共享策略：

- `per_user`：按用户隔离实例
- `per_team`：按队伍共享实例
- `shared`：按题目或按竞赛共享实例

这套设计的目标是把“是否共享、按谁共享”变成题目配置的一部分，并把最终作用域持久化到实例记录里，避免运行时推断失真。

这里还有一个明确边界：

- `shared` 只解决运行环境复用，不解决答案隔离
- 只要题目选择 `shared`，平台就视为该题接受“答案可共享”这个事实
- 如果题目要求防止用户互相转发答案，就不应使用 `shared`

## 2. 配置模型

### 2.1 题目配置

`challenges.instance_sharing` 是实例分配策略的唯一配置来源。

- 类型：字符串枚举
- 默认值：`per_user`
- 可选值：`per_user` / `per_team` / `shared`

题目在创建、更新和查询时都直接暴露这个字段，管理端和学员端看到的是同一套语义。

### 2.2 实例记录

`instances.share_scope` 记录实例实际采用的共享作用域。

- 类型：字符串枚举
- 可选值：`per_user` / `per_team` / `shared`

`instance_sharing` 表示题目配置，`share_scope` 表示运行时解析后的最终结果。这样做是为了支持上下文回退，例如 `per_team` 在没有队伍信息时回退为 `per_user`。

## 3. 作用域解析规则

实例启动时，后端先读取题目配置，再结合当前上下文解析出最终作用域。

### 3.1 练习模式

- `per_user`：按 `user_id + challenge_id` 建立实例
- `per_team`：练习模式没有队伍上下文，回退为 `per_user`
- `shared`：按 `challenge_id` 共享实例

### 3.2 普通竞赛

- `per_user`：按 `contest_id + user_id + challenge_id` 建立实例
- `per_team`：若报名记录绑定队伍，则按 `contest_id + team_id + challenge_id` 共享；没有队伍时回退为 `per_user`
- `shared`：按 `contest_id + challenge_id` 共享实例

### 3.3 AWD

AWD 保持原有队伍共享语义，优先按队伍复用，不受普通题目共享策略回退逻辑影响。

## 4. 唯一性与复用规则

实例唯一索引按 `share_scope` 分流，保证每种作用域下只有一份活动实例。

- 练习个人实例：`user_id + challenge_id`，仅限 `share_scope = per_user`
- 竞赛个人实例：`contest_id + user_id + challenge_id`，仅限 `share_scope = per_user`
- 竞赛队伍实例：`contest_id + team_id + challenge_id`，仅限 `share_scope = per_team`
- 练习共享实例：`challenge_id`，仅限 `share_scope = shared`
- 竞赛共享实例：`contest_id + challenge_id`，仅限 `share_scope = shared`

实例启动命中已有活动实例时，不重复创建容器，直接返回已有实例。

## 5. 生命周期与管理规则

### 5.1 `per_user` / `per_team`

- 按现有启动、延时、销毁语义运行
- 用户只影响自己或自己队伍所绑定的实例
- AWD 队伍服务实例由 AWD 赛程和队伍工作台托管，不允许在“我的实例”链路手动延时或销毁
- 当 Docker daemon 重启或宿主机重启导致运行中容器丢失时，后台运行时对账会保留原实例记录和作用域字段，将实例重新置为 `pending`，由实例调度器重建容器

### 5.2 `shared`

共享实例的生命周期由平台统一管理，不开放给普通用户做实例级控制。

- 用户再次点击“启动”时，如果实例已存在，则刷新 `expires_at`
- 刷新逻辑为 `max(current expires_at, now + default_ttl)`
- 普通用户不可手动延时
- 普通用户不可手动销毁
- 实例由后台 TTL 清理回收
- 运行时丢失恢复只重建容器，不改变共享作用域或重新分配实例归属
- 教师和管理员保留统一管理能力

### 5.3 运行时丢失恢复边界

运行时丢失恢复只处理“数据库仍认为实例活动，但 Docker 容器已经缺失或退出”的情况。恢复时必须保留原实例的业务身份，不能重新解析成新的作用域：

- 练习实例继续保留 `user_id + challenge_id + share_scope`
- 普通竞赛实例继续保留 `contest_id + user_id/team_id + challenge_id + share_scope`
- AWD 队伍实例继续保留 `contest_id + team_id + service_id + share_scope`

恢复写入只允许清空运行时字段并把状态改回 `pending`：

- 清空：`container_id / network_id / runtime_details / access_url`
- 保留：`user_id / contest_id / team_id / challenge_id / service_id / share_scope / nonce / host_port / expires_at`

这样 Jeopardy 练习题不会被错误关联到 AWD `service_id`，AWD 服务实例也不会退回到旧的 `challenge_id` 入口。

## 6. 可见性与访问控制

### 6.1 实例列表可见性

- 个人实例：仅实例所属用户可见
- 队伍实例：同队用户可见
- 练习共享实例：普通用户可见
- 竞赛共享实例：仅当前竞赛已报名用户可见

### 6.2 实例访问控制

实例访问权限与实例列表权限保持一致，避免“看得到但进不去”或“看不到但能进”的状态分裂。

### 6.3 前端管理行为

前端根据 `share_scope` 控制实例卡片和实例列表页的操作按钮：

- `share_scope = shared`：隐藏“延时”“销毁”
- `contest_mode = awd` 且 `share_scope = per_team`：隐藏“延时”“销毁”，显示“系统托管”
- 仍保留“启动/进入”按钮，作为进入共享实例和自动续期的入口

## 7. 配置约束

`shared` 不是所有题型都能启用，当前实现明确限制为可安全共享的题目。

允许启用 `shared` 的题目：

- `flag_type = static`
- `flag_type = regex`
- `flag_type = manual_review`

禁止启用 `shared` 的情况：

- `flag_type = dynamic`
- 题目拓扑中存在 `inject_flag = true`

限制原因很直接：动态 flag 和实例内 flag 注入依赖实例创建期写入环境变量，这与共享实例的多用户访问模型冲突。

## 8. 判题边界

`shared` 只影响实例复用，不改变判题语义。

- `flag_type = static`：仍按固定 Flag 校验
- `flag_type = regex`：仍按统一正则校验
- `flag_type = manual_review`：仍进入同一套人工审核流程

平台不为共享实例引入额外的“按用户区分答案”语义。也就是说：

- 共享实例上的静态答案、正则答案本身可能被转发
- 这不是平台异常，而是 `shared` 模式的设计前提
- 如果题目要求答案隔离或需要防止直接转发，就应选择 `per_user` 或 `per_team`

## 9. 题型适用建议

实例共享策略不是按题目分类名决定，而是按“运行时是否会产生用户间状态干扰”和“是否接受答案可传播”决定。

当前平台建议固定如下：

- `Web` 题默认使用 `per_user`
- `Pwn` 题默认使用 `per_user`
- 团队赛中确有稳定协作需求时，优先考虑 `per_team`
- `shared` 只用于无状态、只读、不会发生用户间污染且接受答案可传播的题

更适合 `shared` 的题通常包括：

- 加密 / 解密
- 编码 / 解码
- 纯计算型脚本题
- 只读查询型服务
- 不依赖登录态、用户写入和持久化数据的轻量题

不适合 `shared` 的题通常包括：

- 大多数 `Web` 题
- 大多数 `Pwn` 题
- 会写数据库、文件、缓存或 Session 的题
- 一个用户的操作会改变其他用户观察结果的题
- 要求平台层面防止答案被转发的题

## 10. 管理端与学员端表现

### 10.1 管理端

管理端保留 `shared` 选项，但要把约束直接表达出来：

- 选择 `shared` 时，界面明确提示“共享实例只适用于无状态题，该模式不提供用户级答案隔离”
- 若管理员需要隔离答案，应改用 `per_user` 或 `per_team`
- 如果同时配置 `dynamic` 或带 `inject_flag` 的拓扑，后端直接拒绝保存

### 10.2 学员端

学员端只表达“这是共享实例”，不表达“答案隔离”或“每人唯一答案”。

- 共享实例继续隐藏“延时”“销毁”
- AWD 队伍服务实例继续展示在“我的实例”中，但不提供“延时”“销毁”，实例生命周期由 AWD 队伍工作台和赛程管理
- 共享实例仍可通过“启动/进入”进入
- 页面文案不对共享模式作额外的唯一性承诺

## 11. 接口影响

### 11.1 题目接口

以下接口增加 `instance_sharing` 字段：

- 管理端题目创建
- 管理端题目更新
- 管理端题目详情
- 学员题目详情

### 11.2 实例接口

以下接口增加 `share_scope` 字段；实例列表同时返回 `contest_mode`，用于区分 AWD 队伍服务实例的托管行为。

- 实例详情
- 实例列表
- 启动实例响应

## 12. 运行边界

这套设计解决的是实例数量和实例作用域问题，不负责共享应用内部的数据隔离，也不负责为共享实例提供用户级答案分流。

当前边界明确如下：

- 平台负责实例创建、实例复用、访问控制和生命周期管理
- 共享题的业务隔离仍由题目自身保证
- 不在这一层引入容器池预热、题内多租户数据隔离或一次性会话沙箱
- 不为共享实例额外设计专用提交流程来弥补静态答案传播

## 13. 当前落地结果

当前后端设计已经固定为：

- 题目通过 `instance_sharing` 声明实例分配策略
- 实例通过 `share_scope` 记录最终运行作用域
- 普通练习、普通竞赛和 AWD 使用同一套共享策略框架
- `shared` 实例由平台统一续期和清理
- `shared` 只适用于无状态、可安全共享且接受答案可传播的题
- 不适合共享的题型在配置阶段直接拒绝
