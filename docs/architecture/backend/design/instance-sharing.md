# 题目实例共享策略设计

## 背景

当前平台对需要启动靶机的题目，默认仍以“每个用户一份实例”为主。现有实现里只有 AWD 竞赛会按队伍复用实例，普通练习和普通竞赛仍会随着用户数线性增加容器数量，直接推高宿主机内存占用。

代码层面已经具备一部分基础能力：

- 有实例去重和唯一索引，能避免同一作用域重复起多份实例
- 有启动调度、全局活动实例上限、单用户并发上限
- 有 TTL 过期回收和孤儿容器清理

但真正缺的仍是“实例分配策略”这一层：平台还不能根据题目特征决定该题应该按用户隔离、按队伍共享，还是按场景共享服务。

## 目标

本轮只补最有收益、且可以稳定上线的一层能力：

1. 为题目增加实例分配策略字段
2. 支持三种策略：
   - `per_user`
   - `per_team`
   - `shared`
3. 将该策略接入题目创建/更新、题目查询、实例启动、实例可见性、实例访问控制和实例管理
4. 保持现有 AWD 共享行为兼容
5. 对不适合共享的题目显式拒绝，而不是隐式放过

## 非目标

本轮不做下面这些内容：

- 赛前容器池预热
- 镜像预拉取调度
- 每连接一次性 jail / sandbox
- 把共享服务改造成多租户数据隔离应用
- 对现有题目自动推断并批量迁移策略

这些能力会写入设计文档和后续计划，但不在本轮实现里做半成品。

## 可选方案

### 方案 A：仅继续按用户实例，调小资源限额

优点：

- 改动最小
- 不需要处理共享实例的访问控制和管理语义

缺点：

- 不能解决“同题多人启动”造成的容器数量线性增长
- 只是把单实例做小，不是把实例数量降下来
- 对内存高水位问题帮助有限

不推荐。

### 方案 B：所有可启动题统一共享服务

优点：

- 容器数最少
- 对 Web / API 题的资源收益最大

缺点：

- 会直接破坏需要实例级 Flag、状态隔离和环境污染隔离的题
- 当前平台没有多租户数据隔离层，也没有共享实例访问租约模型
- 一刀切风险太高

不推荐。

### 方案 C：题目显式声明实例分配策略，平台按策略选择作用域

优点：

- 能覆盖 `per_user / per_team / shared` 三种主流策略
- 与现有实例表、作用域去重、运行时调度模型兼容
- 可以先在适合共享的题目上启用，不强迫所有题迁移

缺点：

- 需要补模型、迁移、访问控制和前端显示字段
- `shared` 需要限制到可安全共享的题型

推荐采用。

## 推荐方案

采用方案 C，并把 `shared` 收敛成一版可控实现：

- `per_user`
  - 默认值
  - 保持当前练习模式和普通竞赛的行为
- `per_team`
  - 在团队赛事中按 `contest + team + challenge` 共享
  - 无团队上下文时回退为 `per_user`
- `shared`
  - 练习模式下按 `challenge` 共享
  - 竞赛模式下按 `contest + challenge` 共享
  - 只允许静态 Flag、正则 Flag、人工判题这类不依赖实例内动态 Flag 注入的题
  - 普通用户可以访问，但不能手动销毁或手动续期
  - 再次点击“启动”时，如果共享实例已存在，则自动刷新有效期

## 数据设计

### `challenges`

新增字段：

- `instance_sharing`
  - 类型：字符串
  - 默认值：`per_user`
  - 枚举：`per_user` / `per_team` / `shared`

该字段是题目的行为配置，属于挑战管理主数据，应该放在 `challenges` 表而不是临时扩展 JSON 中。

### `instances`

新增字段：

- `share_scope`
  - 类型：字符串
  - 默认值：`per_user`
  - 枚举：`per_user` / `per_team` / `shared`

原因：

- 题目策略只描述“应该如何分配”
- 实例记录需要明确写下“这份实例是按什么作用域创建的”
- 后续做可见性、访问控制、索引约束和管理限制时，都需要直接依赖实例自身的作用域信息

## 索引设计

现有唯一索引只覆盖“个人实例”“竞赛用户实例”“竞赛队伍实例”，不区分共享模式。新增字段后需要改成按 `share_scope` 分流：

- 个人实例唯一：`user_id + challenge_id`，仅限 `share_scope = per_user` 且非 contest
- 竞赛用户实例唯一：`contest_id + user_id + challenge_id`，仅限 `share_scope = per_user`
- 竞赛队伍实例唯一：`contest_id + team_id + challenge_id`，仅限 `share_scope = per_team`
- 练习共享实例唯一：`challenge_id`，仅限 `share_scope = shared` 且非 contest
- 竞赛共享实例唯一：`contest_id + challenge_id`，仅限 `share_scope = shared`

## 行为设计

### 启动实例

实例启动时先根据题目策略和上下文解析出真正的作用域：

- 练习题
  - `per_user` -> 用户作用域
  - `per_team` -> 回退到用户作用域
  - `shared` -> 全局共享作用域
- 竞赛题
  - `per_user` -> 竞赛用户作用域
  - `per_team` -> 若报名记录有队伍，则走队伍作用域，否则回退到用户作用域
  - `shared` -> 竞赛共享作用域
- AWD
  - 保持现有行为优先，即按队伍共享

### 共享实例续期

`shared` 实例不开放“用户手动续期”接口。原因是共享实例的生命周期不应该由某个用户手工控制。

替代策略：

- 若用户再次点击启动，且命中已有共享实例，则自动把 `expires_at` 刷新到 `max(current expires_at, now + default_ttl)`

### 销毁策略

`shared` 实例对普通用户不开放手动销毁。

原因：

- 共享实例被一个用户销毁，会直接影响其他用户
- 当前平台没有“租约计数”和“最后活跃用户退出才销毁”的共享服务管理层

因此本轮约定：

- `shared` 仅允许后台 TTL 清理
- 教师/管理员保留统一管理能力

## 兼容性限制

### `shared` 的准入限制

本轮只允许下列题目使用 `shared`：

- `flag_type = static`
- `flag_type = regex`
- `flag_type = manual_review`

下列情况一律拒绝：

- `flag_type = dynamic`
- 拓扑节点存在 `inject_flag = true`

原因很直接：当前平台的动态 Flag 注入发生在实例创建阶段，并写入容器环境变量。共享实例天生不适合这种机制。

### `per_team` 的回退

如果题目配置为 `per_team`，但当前上下文拿不到队伍信息，则回退到 `per_user`，避免把启动流程做成硬失败。

## API 影响

### 题目管理接口

在创建、更新、详情响应中增加：

- `instance_sharing`

### 学员题目详情

增加：

- `instance_sharing`

用于前端在题目详情页解释当前题目的实例策略，并决定共享实例是否显示管理操作。

### 实例接口

在实例响应和实例列表中增加：

- `share_scope`

用于前端判断：

- 当前实例是否为共享实例
- 是否展示“延时”“销毁”按钮

## 前端行为

本轮前端只做最小必要同步：

- 题目详情页和实例列表页识别 `share_scope`
- 对 `shared` 实例隐藏“延时”“销毁”
- 启动按钮仍保留，作为“进入共享实例 / 自动保活”的入口

题目管理页的完整策略配置 UI 不作为本轮必做项；本轮先把后端 API、类型和展示打通。

## 测试策略

至少覆盖下面几类用例：

1. 题目创建/更新
   - `shared + dynamic` 被拒绝
   - `shared + static/regex/manual_review` 可以保存
2. 练习模式
   - 两个用户启动 `shared` 题，命中同一实例
   - 用户可以在实例列表中看到共享实例
3. 普通竞赛
   - `per_team` 题在同队内复用
   - `shared` 题在同一场竞赛内复用
4. 实例管理
   - `shared` 实例不允许普通用户销毁或延时
   - 再次启动共享实例会自动刷新有效期
5. 访问控制
   - 共享练习实例允许普通用户访问
   - 共享竞赛实例只允许本场竞赛已报名用户访问

## 分阶段后续工作

### 下一阶段

- 容器池预热配置真正接入运行时调度
- 赛前镜像预拉取
- 按题或按镜像做热点题预热

### 更长期

- 为 Web / API 题提供“共享服务 + 租户隔离数据”运行模式
- 为 Pwn 题提供短生命周期会话沙箱
- 为共享实例增加租约、最后活跃时间和安全回收机制

## 验收标准

- 管理端和后端接口可以为题目配置实例分配策略
- `shared` 题在练习模式和普通竞赛模式下都能复用实例
- `per_team` 题在非 AWD 团队竞赛中能按队伍复用
- 共享实例对普通用户不可手动销毁、不可手动续期
- 现有 AWD 队伍共享行为不回归
- 相关后端模块测试通过
