# 共享抽屉头部错层修补应升级为壳层重构

## Status

implemented

## Context

通知中心右侧抽屉头部位置问题连续两次被修在 NotificationDropdown 调用方，通过 :deep 和 !important 反向重排共享壳层结构，导致视觉无变化且样式所有权继续漂移。此次已在 SlideOverDrawer 内完成结构性修复，但这一类问题的处理原则尚未沉淀到 durable frontend rule。后续遇到共享弹层头部、关闭按钮、图标、标题区排布问题时，应优先检查共享壳层 DOM 结构和 CSS 变量边界，而不是在业务组件中继续增加 deep override 补丁。

## Problem

- 错误发生在共享抽屉壳层 `SlideOverDrawer`，但修复连续落在业务调用方 `NotificationDropdown`。
- 这种修法会把共享结构问题伪装成“局部样式问题”，继续制造 `:deep()`、`!important`、DOM 顺序反排和无效变量残留。
- 结果是：
  - 页面看起来“改了代码但没有变化”
  - 共享壳层和调用方的样式所有权持续漂移
  - 后续其他抽屉调用点也无法复用这次修复

## Suggested Direction

- 在 frontend / review 规则里补一条共享弹层排版判断：
  - 若图标、标题区、关闭按钮、分隔线、头部额外区的结构关系不对，先检查共享壳层 DOM 和 CSS 变量。
  - 只有当共享壳层结构正确、调用方只是需要主题变量或局部内容排版差异时，才允许在业务组件中做局部覆盖。
- 对共享弹层类问题，避免在调用方新增以下补丁作为首选方案：
  - `:deep()` 反向重排共享壳层节点
  - `!important` 压共享样式
  - 为已经失效的共享变量继续赋值
