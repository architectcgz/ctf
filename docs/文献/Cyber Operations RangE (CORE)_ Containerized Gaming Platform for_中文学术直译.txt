摘要
网络安全游戏通过夺旗（capture-the-flag，CTF）竞赛、战争游戏（war games）以及取证挑战而得到普及。通过提供真实的动手实践经验，网络安全竞赛在支持课堂学习、专业技能发展以及场景测试方面具有巨大潜力。为了发挥这一潜力，支撑性的游戏化平台需要克服如下问题：i）复杂的虚拟化基础设施，需要大量资源与管理；ii）在竞赛环境中缺乏对实时网络作战（cyberoperations）的支持；以及 iii）实现独立的、实时的计分系统，用于跟踪进展并确定参赛者的结果。为应对这些问题，我们开发了 Cyber Operations RangE（CORE）——一个容器化的网络安全游戏化平台。本文给出了 CORE 平台用于实时网络作战竞赛的设计、实现与应用。

1. 引言
网络安全竞赛是安全爱好者、学生、教育者以及专业人员最为流行的动手实践练习形式［1］。其形式涵盖夺旗（CTF）竞赛、战争游戏，或取证挑战［2］、［3］。CTF 的游戏结构为每位参赛者（通常以团队形式组织）提供相同的挑战，涵盖操作系统安全、网络/软件/网络应用安全、密码学与散列函数、逆向工程以及网络取证等领域。竞赛组织通常采用“jeopardy-style（答题闯关式）”或“attack-defense（攻防式）”的竞赛格式［4］。在答题闯关式竞赛中，并不存在预先规定的解题顺序，参赛者可以在每一轮等时长的竞赛回合中自由推进。在攻防式竞赛中，参赛者会获得相同的游戏盒（虚拟机），其中预置了漏洞；参赛者既需要修补自身漏洞，同时还要利用其他参赛者的游戏盒［5］。几乎所有 CTF 平台均基于虚拟机，并且可以是集中式的（即承载现场参赛者），也可以是分散式的（即承载远程参赛者，参赛者通过虚拟专用网络（VPN）连接）。

战争游戏与答题闯关式 CTF 竞赛类似，但随着参赛者推进，挑战难度会逐步提升。在取证挑战中，参赛者同样会被提供网络取证领域的相同问题。这些问题由参赛者独立解决，并将解答提交至远程游戏平台。不同于 CTF，取证挑战持续数月并按年举办。团队在竞赛结束后依据其解答进行评分。

传统上，答题闯关式 CTF 竞赛一直是最受欢迎的网络取证游戏化活动［4］。尽管这类 CTF 能够支持课堂学习与专业技能发展［6］、［7］，但攻防式竞赛在体验式学习方面更为有效，因为其代表了更为真实的网络安全场景［8］、［9］。尽管具有这一优势，攻防式 CTF 在网络安全游戏化中却较少出现，因为它们需要复杂的游戏基础设施，其中包括虚拟化、计分管理任务以及大量资源［2］［4］。此外，参赛者还被期望在管理任务与挑战之间多任务切换，而非专注于竞赛本身。

为此，我们开发了一种用于网络作战竞赛的新型、基于容器的游戏基础设施——网络作战靶场（Cyber Operations RangE，CORE）。CORE 平台通过以 Docker 容器与实时计分引擎替代复杂的虚拟化托管基础设施，从而消除了其负担。其管理被高度简化，并支持快速实例化不同规模的竞赛以及大量挑战。据我们所知，CORE 是首个超越经典攻防式 CTF 概念的平台，能够提供攻击、防御以及实时取证的真实网络作战体验。参赛者的评分不仅取决于其提交的旗标（flag），还取决于其在竞赛中采用的战术方法。不同于基于关卡的游戏进度，CORE 迫使参赛者立即采取网络作战行动。他们需要将实时取证作为防御策略的一部分来执行，这不同于传统取证挑战中被排除在外的时间约束。本文其余部分将描述 CORE 平台的设计与技术实现，讨论其潜在应用范围，并概述未来的测试与开发计划。

2. 相关工作
大多数攻防式 CTF 借鉴了 iCTF 竞赛的设计［10］。游戏盒被打包为虚拟机，参赛者可以本地访问（在其硬件上，即集中式游戏基础设施），或通过 SSH 远程访问（托管在竞赛服务器上，即分散式游戏基础设施）。文献［4］的作者指出，虚拟化基础设施需要大量资源与管理：在固定服务数量的条件下，承载竞赛所需的内存使用量会随着玩家（虚拟机）数量成比例增加，从 5 名玩家约 5GB 直至 40 名玩家约 40GB。

为克服虚拟化与多任务问题，他们修改 iCTF 设计，开发了基于 Docker 容器的竞赛基础设施［4］。所提出的游戏平台以 Docker 容器替代虚拟机，并在基础设施中加入了容器镜像仓库、游戏服务器以及旗标卷（flags volume）模块。容器镜像仓库存储参赛者的游戏盒——服务容器与漏洞利用（exploit）容器宿主——并在单台服务器上提供与游戏服务器上镜像的周期性同步。服务容器宿主是预定义的竞赛游戏盒，而漏洞利用容器则由每位参赛者（或团队）基于自定义漏洞利用集合开发。竞赛由若干游戏回合组成，每一回合包括：服务容器同步、漏洞利用容器同步、植入旗标、运行漏洞利用以及检索旗标。其玩法与软件开发中的缺陷修复高度相似，参赛者根据其修复问题（即捕获所植入旗标）的能力进行评分。

CORE 平台同样利用 Docker 容器来解决虚拟化复杂性以及资源与管理需求过高的问题。相较于其他竞赛［4］，CORE 不需要使用专有的虚拟机自动化软件。如 3.1.1.3 节所述，CORE 编写了自有的定制自动化（依赖开源软件），用于为运行容器的极简操作系统进行配置与供应。其他虚拟化环境（例如国家系统安全与信息保障中心（CSSIA））依赖网络开发组（NDG）的闭源软件；该公司公布的订阅费用为每名参与者 30 至 175 美元［11］。

与［4］对比，CORE 的游戏化更类似实时网络作战，以应对缺乏此类网络安全游戏体验的问题。不同于 SANS NetWars［12］，CORE 参赛者在竞赛期间不需要按关卡推进，他们可以立即投入网络作战行动。CORE 并未将攻击（漏洞利用容器）与防御（服务容器）任务分离并由游戏服务器充当“红队”，而是将攻击与防御包含在一个单体的游戏盒（容器）中，从而允许参赛者实时参与。CORE 还加入实时取证任务，以补充防御过程中的攻击溯源，并据此构造另一次响应攻击。

除上述两项设计外，CORE 还集成了实时计分引擎，用于每秒跟踪玩家进展。其实现目的是让玩家能够看到其机器是否已被攻陷，并验证其是否发现预置的恶意程序。与其他 CTF 环境相比，其计分要么基于旗标是否被捕获，要么基于某项服务是否仍在运行（每隔若干分钟检查一次）。对许多参赛者而言，这种计分实现使得游戏节奏更慢，并在其被攻陷后给予恢复时间。在 CORE 环境中，参赛者会因检查失败/成功而被惩罚或奖励。计分引擎的这一方面使 CORE 具备更为真实的攻防竞赛设定。

3. CORE——基础设施运维
3.1 系统架构
CORE 架构如图 1 所示。CORE 假设参赛者为远程用户；因此，客户端的安全套接层（Secure Socket Layer，SSL）VPN 流量会被路由到 CORE 的边缘请求路由层。SSL 卸载是必要的，用于减轻基座节点（base nodes）上的 CPU 处理负担。竞赛流量随后通过运行在 OSI 第 4 层的负载均衡对等节点进一步路由至基座节点层，在该层中，每位参赛者被分发至其竞赛游戏盒（以 Docker 容器实现）。图中描绘了共计四个环境或“pod”，游戏盒驻留其中。CORE 能够部署并行的生产环境与开发环境（标注为 ENV-A-PRD/ENV-B-PRD 与 ENV-A-DEV/ENV-B-DEV），以支持灵活开发以及在不同网络安全竞赛之间进行动态切换。在高层次上，CORE 架构的动机有二：1）确保水平可扩展性；2）支持竞赛级练习隔离（基础设施安全）。为实现这两个目标，编写定制化系统自动化工具对我们而言至关重要。

3.1.1 平台可扩展性
水平可扩展性（horizontal scalability）指的是通过“横向扩展（scaling out）”系统基础设施以处理系统负载增加的实践。换言之，我们并不试图“纵向扩展（scale up）”（即垂直扩展）系统资源。这使得高可用性与敏捷性成为可能：我们不再依赖单台高性能服务器处理全部流量负载，而是能够在资源配置较低的多台可用服务器之间动态地进行负载均衡。我们接受平台会发生故障这一事实，因此采用能够容忍多个个体故障的弹性技术。大型数据中心运营商已对相关技术进行了长期研究［13］。

在我们的环境中，平台可扩展性通过多种方法实现：高可用负载均衡器、分布式数据结构集群、定制系统自动化工具、网络化存储基础设施以及容器网络织构（network fabrics）。

3.1.1.1 请求负载均衡
水平可扩展架构通常包含负载均衡器（load balancers，LB），用于将用户路由到不同的应用端点。CORE 负载均衡器的设计目标为：（1）提供 OSI 第 4 层负载均衡服务；（2）具备高可用性；（3）将用户路由到且仅路由到一个游戏盒（容器）。在我们的环境中，OSI 第 4 层的 LB 服务是必要的，因为容器会绑定到预定义的 TCP/UDP 端口范围以提供 HTTP 流量服务。将端口作为应用节点的标识，从策略上消除了生产子网出现 IPv4 地址耗尽的任何可能性。此外，在 CORE 设计中，第 4 层在逻辑上是合理的，因为基座节点（以 IPv4 地址标识）承载了多个容器（以多个 TCP/UDP 端口标识）。高可用性通过在架构中集成 N+1（主动-被动）冗余并在 LB 之间运行虚拟路由器冗余协议（Virtual Router Redundancy Protocol，VRRP）来实现。N+1 冗余是我们为消除架构中潜在单点故障所采取的第一步，尽管该故障切换机制可能存在单点。随着规模扩展，我们计划在特定环境层面执行 1+1（主动-主动）冗余。通过 VRRP，我们可以定义一个虚拟 IP 地址（Virtual IP address，VIP），当某个 LB 意外停止时，VIP 会迁移到被动 LB。用户通过环境的门户域名进行访问（其 DNS A 记录为 VIP），并在 HTTP 路径中携带特定用户标识符。LB 的访问控制列表（Access Control Lists，ACLs）会匹配该 UID，并在尚未建立其他连接的情况下将用户负载均衡到该容器。如果请求无效，LB 将向客户端返回 HTTP 5XX 错误。由于 LB 感知竞赛盒，我们可以设置任一时刻到容器的连接数上限为 1——这也有助于实现练习隔离目标。通过这种方式，我们提供了另一层安全性，以防止用户尝试连接到其他容器。

3.1.1.2 分布式数据结构集群
CORE 的另一项确保水平可扩展性的方面是分布式数据结构（键-值）集群。多个基础设施组件（例如分布式锁机制或参赛者容器之间的覆盖网络技术）依赖该集群节点进行存储。目前，数据结构集群与提供生产流量的基座节点共同部署。随着需求增长，我们计划将集群迁移到专用服务器上，从而：a）数据结构节点故障的影响半径（blast radius）不会影响生产环境；b）专用固态硬盘可提高随机 IOPS（Input/Output per Second）性能；c）具备独立于生产基座节点对集群进行扩展的能力。

数据结构集群是我们系统架构中的关键组件。因此，由于其有状态特性，我们必须确保其具备高可用性与弹性。高可用性由底层算法［14］实现，该算法赋能集群服务执行领导者选举与节点健康检查：在初始引导（bootstrapping）之后，集群能够确定节点是否失败，因为健康检查以固定时间间隔发送。如果集群的大多数返回非 HTTP 200 状态码，则值班工程师会被呼叫。数据弹性通过该算法追求复制日志（replicated logs）来实现。对于大小为 N 的集群，故障容忍度为（N-1）/2。因此，我们以奇数规模运行集群，以确保当多个基座节点不可用时仍能向集群写入日志。如果无法满足最小多数条件，复制日志将不会提交到集群，集群健康状态将退化。我们部署数据结构集群的最终结果是一个容错的、复制的状态机。

3.1.1.3 Pod 抽象
如图 1 所示，我们将每个“环境”隔离到其自身的“pod”之中。尽管这些术语可互换使用，但 CORE 中的 pod 特指容器编排基础设施中的 pod。随着我们部署容器编排方案，这一点将更为明显。将容器的功能抽象为由多个容器组成的分组，在我们的网络靶场中在逻辑上是合理的。我们将一个 pod 视为一次练习（exercise）。这将有助于构建一个集中式用户界面，竞赛运营者可以在该界面中启动、运行并拆除竞赛。此外，竞赛运营者可以控制计算、存储以及网络资源的限制，并在必要时进行额外供应。否则，随着基础设施规模扩展，负责管理竞赛的运营者将不得不手动运行脚本并执行临时任务，这将变得繁琐。容器编排能够将 pod 跨多个基座节点扩展，从而移除为管理特定竞赛而构建工具的必要性。

3.1.1.4 系统自动化工具
我们广泛实施了专有的“基础设施即代码（infrastructure-as-code）”举措。这样做具有诸多益处（超出本文范围），例如减少人为错误。举例而言，我们利用 YAML 清单覆盖基座节点的供应流程，从而消除任何手动引导。此外，我们还开发了定制基础设施工具，用于自动化已授权的远程命令执行。软件部署也被纳入我们的工具化工作范围。截至本文撰写时，我们估计在基座节点供应与部署方面，约 90% 的系统管理任务已经实现自动化。

3.1.1.5 网络化存储基础设施
在部署 Docker 容器时，我们注意到若干与存储占用有关的问题。其中最突出的为 Docker 镜像使用情况——平均而言，一个支持“山顶之王（king of the hill）”练习的容器在基座节点上消耗约 1GB 磁盘空间。随着更多练习被部署，这将变得沉重——尤其当其他练习每个镜像消耗 1–5GB 时更是如此。考虑到行业对极简容器的立场，每个镜像 1–5GB 的磁盘占用令人不安，并且在基座节点存储方面不可扩展。我们将测试让基座节点指向网络化存储，而非让 Docker 镜像驻留在每个基座节点的本地。多家公司已经开发了相关解决方案，例如 NetApp 通过 NFS 提供的方案［15］。或者，诸如 JFrog 的 Artifactory［16］或 Sonatype 的 Nexus［17］等产品支持集中式的私有 Docker 镜像仓库。

网络化存储的另一项益处是能够将容器服务（例如计分引擎）部署到一个集中位置，基座节点可从该位置拉取（pull），而不是由运营者的工作站推送（push）副本。因此，我们可以在部署方面进一步收紧基座节点网络安全（例如，基座节点仅允许访问存储共享）。

3.1.1.6 容器网络织构
为满足参赛者盒之间网络互连的需求，我们在容器之间部署了第 3 层覆盖网络。举例而言，在“山顶之王”练习中，所有参赛者被置于同一网络段，从而使攻防操作能够发生，而无需横向移动（pivot）到另一个网络段。其他练习（例如攻防场景）可以被置于不同的网络段。我们正在评估虚拟可扩展局域网（Virtual eXtensible Local Area Network，VXLAN）［18］以提供第 2 层网络分段。在 VXLAN 部署中，我们能够静态定义虚拟网络标识符（Virtual Network Identifiers，VNIs），使每个练习映射到且仅映射到一个 VNI。第 3 层隧道端点（L3 tunnel endpoints），即 VTEP（Virtual Tunnel End Points，虚拟隧道端点），执行带标签帧的封装与解封装。只有目标 VTEP 会接收帧。随后，第 2 层帧将根据常规转发规则继续转发。

3.2 基础设施安全与监控
由于 CORE 旨在以隔离方式处理恶意计算机网络操作，我们在 CORE 的网络与系统基础设施中采取了纵深防御（defense-in-depth）的设计方法。基于时间序列的监控使我们能够快速定位系统异常与性能瓶颈。

3.2.1 SSL VPN 基础设施
所有用户必须建立到我们生产网络的 SSL VPN 会话（见图 1）；分离隧道（split tunneling）将从客户端视角确保与其他相关网络的隔离。目前，这些 VPN 会话在托管于机房共置（co-location）的防火墙上终止；但鉴于 SSL 终止所需的 CPU 资源，我们正在探索高可用 VPN 方案。

3.2.2 内部服务安全
如前所述，负载均衡器将被加固，具体而言，将在负载均衡服务上实现访问控制列表（ACLs）。参赛者将被分配一个可访问的容器（当其注册参赛时将向其提供用户凭据）。此外，我们还提及采用最少连接（least connection）负载均衡算法，以确保任一时刻到某个容器仅有一个用户会话处于活动状态。我们也在探索在所有 CORE 链路上运行传输层安全（Transport Layer Security）的可能性。

3.2.3 基座节点安全
标准主机网络流量过滤将通过集成健壮的 iptables 规则集来实现，并将作为系统供应自动化的一部分进行评估。我们所选用的基座节点操作系统具备自动主机更新（见下一节），且与 CentOS、Debian 或 Fedora 等发行版相比，其整体功能足迹更小。此外，我们预计将通过 sysctl 执行与安全相关的内核参数调优。

3.2.4 自动化主机更新
确保基础设施各操作系统得到正确补丁更新不仅对安全至关重要，也关系到系统可靠性与性能。因此，我们在所有基座节点上集成了自动化补丁更新。我们遇到的问题之一是在更新发生时确保数据结构集群保持完整。当主机因重启而下线时，会获取一个驻留在数据结构集群上的二元信号量（binary semaphore），并在主机恢复为活动服务后释放。这可确保在维护期间集群保持健康。

3.2.5 系统监控
为监控 CORE 基座节点的性能，我们实现了基于时间序列的云监控解决方案；当某个基座节点或其上的核心基础设施服务发生故障或处于退化状态时，该方案将告警给“值班工程师（on-call engineer）”。我们大量依赖 StatsD［19］来收集基座节点上的系统性能指标。监控阈值持续被细化，以确保值班工程师能够获知任何系统异常；同时，我们正在探索对这类事件自动化响应的工作。我们也在评估用于系统日志的数据采集器以及告警/归档/分析解决方案。

4. CORE——竞赛平台
4.1 竞赛实现
CORE 支持实时网络作战竞赛，使参赛者能够攻击、防御并执行实时取证任务。典型的 CORE 竞赛实例组织为两个 45 分钟的竞赛时段，并在其间设置一个 15 分钟的中场休息。CORE 的概念验证（proof-of-concept）竞赛基础设施如图 2 所示。

从网络流量视角来看，CORE 管理员通过 SSH 访问管理控制台，而参赛者通过 HTTP 访问竞赛控制台。我们意识到参赛者可能试图捕获竞赛流量；我们计划对 REST API 实施认证，从而禁止客户端在未认证情况下发起 GET 或 POST 请求。在后续日期，将在竞赛管理员节点上实现 HTTPS。

4.1.1 竞赛管理员控制台
CORE 竞赛管理员控制台具有两项主要功能：1）容器管理（如第 3 节所述）；2）通过竞赛控制台进行竞赛管理。在竞赛控制台内存在若干模块：
● 计时引擎——以 time_remaining:stage 的格式维护全局竞赛时间，并通过 RESTful API 将其返回给计分后端（Scoring Backend）
● 旗标引擎——维护参赛者已捕获旗标的当前状态
● MySQL 数据库——维护关于当前参赛者及其当前得分的竞赛信息；其通过 RESTful API 与计分后端交互
● RESTful API——连接计分、计时与旗标进度功能的中间件；并通过计分前端（使用 websockets）向参赛者提供竞赛进展
● 计分前端——竞赛仪表板

CORE 管理员拥有一份预编译的已订阅参赛者名单，竞赛访问应据此进行配置与开通。一旦游戏盒被分配完成，竞赛控制台将运行脚本以启动上述每个模块，并通过仪表板宣布竞赛开始。启动脚本也会为每个游戏盒添加一组动态配置的、唯一的旗标集合。

4.1.2 CORE 游戏盒
每位参赛者会被分配一个游戏盒——Docker 容器——该容器运行 Ubuntu 14.04、XFCE 桌面（通过 VNC）、SSH 以及 Python 2.7。CORE 当前使用 Ubuntu 14.04 作为基础镜像，但这一点很容易更改。启用 XFCE 桌面环境，以便参赛者能够通过 VNC 服务以图形方式访问其游戏盒。SSH 也被启用，作为参赛者之间除标准管理操作之外的带外（out-of-band）攻击与防御工具。启用 Python 2.7 以提供脚本化与自动化攻击/防御/取证分析的能力。在竞赛期间，参赛者将查看计分前端以跟踪竞赛进度并提交旗标。

4.1.3 CORE 计分
竞赛计分涵盖：1）捕获其他参赛者的旗标（攻击）；2）找到自身旗标并加固其机器（防御）；以及 3）完成取证任务。每个旗标依据发现难度进行加权。竞赛累计得分计算为所捕获旗标得分与所执行取证任务得分之和。CORE 的计分将参赛者在游戏盒上对服务进行启动/停止/重配置的任何行为、对当前文件系统的操纵，或对系统日志的审阅视为取证任务，并对此提供额外得分。

为获得实时得分，计分后端会枚举位于其基座节点上的所有运行中容器，并通过 telnet 连接计时脚本以确定计分何时开始/结束。随后，计分引擎开始对每个容器上执行的动态取证任务集合，按每 5 秒一次的频率执行一组预定义检查。检查通过 Docker exec 命令对每个容器执行，该命令允许从宿主机对容器进行远程命令执行。计分引擎的目标是使这些检查能够实时进行。我们当前通过 4 个进程的进程池实现计时，容器在这些进程之间平均分配。一次完整检查循环当前耗时 5 秒。在每个循环结束时，计分引擎通过 REST API 将得分上报至 MySQL 数据库，在该库中其被加到旗标得分上并被宣布到计分前端。当前的计分引擎实现（多线程）在 6 个 vCPU 资源（Intel(R) Xeon(TM) CPU @ 3.50GHz）上使用了 10% 的服务器资源。为运行测试，REST API 列出任意测试命令，计分引擎检索这些命令并将其加入检查列表。这些检查通过 Docker 的 exec 在每个容器上运行，并以 subprocess 方式执行。为实现对每个容器的实时计分，我们使用工作者池（pool of workers）异步运行检查。我们计划通过使用诸如 gevent 与 celery（分布式任务队列库）之类的多进程框架修改实现，以优化计分引擎。一旦我们实现容器编排，竞赛管理将通过 Web 界面完成。旗标引擎将被修改为接受要加入到机器中的任何旗标以及旗标位置。计分引擎将从 REST API 获取全部旗标信息，并使用 Docker exec 命令将旗标注入到其各自位置。所有当前竞赛得分在配置文件中预定义。

用户可能通过从 Docker 容器逃逸到宿主环境来攻陷计分引擎。我们通过为每个竞赛容器分配用户命名空间（user namespaces）来缓解该问题。在发生容器逃逸的情况下，用户必须在 Docker 容器宿主机内执行提权操作（privilege escalation）。

4.2 竞赛结构
在 CORE 中，传统的攻防回合被两个 45 分钟回合与一个 15 分钟中场休息所替代。引入中场休息是为了与参赛者进行复盘。这在 CORE 平台用于教育或培训目的时尤为重要，因为 CORE 管理员（教育者/培训者）能够提供额外帮助以达成竞赛目标。在每一回合中，参赛者被敦促执行如下步骤：
● 侦察（Reconnaissance）——在第一阶段对初始游戏盒进行发现，以及对其他参赛者进行发现
● 攻击/防御（Attacking / Defending）——在其他人的游戏盒上捕获旗标，并删除其自身上的旗标
● 实时取证任务（Live forensic tasks）——系统日志审阅、系统重配置

参赛者可以自由选择其策略以及将如何执行上述每一步。CORE 的理念是激励参赛者完成这些步骤，以便最大化其得分（或者在 CORE 用于教育或专业培训时最大化学习结果）。

5. CORE——测试与路线图
我们正在从客户端视角探索 CORE 的若干关键绩效指标（Key Performance Indicators，KPI）。鉴于其作为面向其他高等教育机构的软件即服务（Software-as-a-Service，SaaS）解决方案的前景，我们高度关注客户端性能指标（例如 HTTP Time-To-Last-Byte（TTLB））。我们目前正在考虑在美国数据中心 POP（points of presence，接入点）中部署的多种代理上进行合成应用监控（synthetic application monitoring，即模拟客户端浏览器行为），以便获得数据中心基础设施之外的外部网络可见性。由此，我们将能够快速识别并修复应用性能瓶颈。

CORE 已在我们德保罗大学本地的开发环境中进行了初步评估。在一次近期负载测试中，合计 15 个参赛者盒在同一时间并发运行一个练习，开发基座节点的 6 个 vCPU 资源（Intel(R) Xeon(TM) CPU @ 3.50GHz）约有 26% 被使用。包含练习的参赛者容器在开发基座节点上消耗了 1.283GB 内存。与［4］的结果相比，CORE 通过将承载 15 名玩家且服务数量固定的整个竞赛时间内的内存消耗平均值保持在相对恒定的 641.93MB，表现出相似且有前景的性能。

在未来的 CORE 开发中，我们计划允许参赛者以团队形式组织并进行竞赛。我们也计划引入机器人（bots）对参赛者运行预定义漏洞利用。我们的目标是优化计分引擎，使其在实时条件下异步运行。我们计划工作的另一方面是 CORE 平台的可靠性（dependability）与弹性（resilience），以防止容器故障以及系统调用操纵（system calls manipulation）。

6. 结论
CORE 平台的开发旨在克服组织攻防式 CTF 时当前的管理负担，以及实时条件下缺乏真实网络作战竞赛的问题。CORE 通过将参赛者的游戏盒容器化，并以侦察、攻击、防御与取证任务构造竞赛结构，从而实现这些设计目标。

CORE 平台可用于多种目的：不同规模的竞赛、教育性的体验式学习、专业培训与发展，以及技能集评估。教育者可以使用 CORE 教授攻击、防御以及多种网络作战战术。开发者、系统管理员与安全专业人员可以使用 CORE 在其组织内部托管竞赛，以一种有趣且具竞争性的方式促进安全意识。CORE 也可有益于安全开发生命周期，因为其提供了一个灵活的平台，用于定制化威胁分析与场景测试。
