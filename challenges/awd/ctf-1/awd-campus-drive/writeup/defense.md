# 防守题解

## 可修改位置

- 平台防守入口实际对应的业务代码在 `/workspace/src/challenge_app.py`
- 题目包内的源文件位置是 `docker/workspace/src/challenge_app.py`
- 运行时和 checker 文件仍在保护边界内，主要修补都应落在这个业务文件里

## 主要漏洞点

### 1. 预览接口直接信任路径参数

`preview()` 从查询参数直接拿 `path`，然后拼到 `DATA_DIR / raw_path` 后读取。只要传入带目录跳转的内容，就可能读到上传目录以外的文件。

这题最关键的防守点就是这里。

推荐修法有两类：

- 更稳的做法：把预览入口改成文件 ID，而不是文件路径
- 如果你想保留现有参数形式，至少也要在 `resolve()` 后确认目标文件仍位于上传目录内

只做 `secure_filename()` 还不够，因为预览接口当前根本不是围绕固定文件 ID 设计的。

### 2. 上传校验存在双扩展绕过

`upload()` 里现在相当于允许 `.jpg.php` 这种双扩展文件混进去。攻击方可以借这个入口把危险脚本伪装成合法文件。

推荐修法：

- 不要只看最后一个后缀
- 同时检查完整后缀链
- 对 `.php`、`.phtml`、`.phar`、`.py`、`.sh` 这类危险后缀做显式拒绝

如果继续保留上传功能，比较稳的方式是：

- 原始展示名单独保存
- 落盘文件统一换成随机文件名
- 下载和预览都只按内部文件 ID 走，不再按用户上传时的名字直接访问

### 3. 上传目录同时承担业务存储和访问定位

当前首页、下载、预览都直接围绕真实文件名工作，这会让“路径安全”和“业务展示名”耦合在一起，越修越容易漏。

更适合演示赛提交的方式是：

- 上传时生成随机 `storage_name`
- 在 `/workspace/data` 下额外保存一份 `展示名 -> storage_name` 的映射
- 列表页展示原始文件名，但下载/预览只使用内部 ID

## 推荐修改函数

- `upload()`
- `download()`
- `preview()`
- 如有需要，补一个专门负责路径校验或文件 ID 解析的辅助函数

## 保活约束

修补后不要把业务直接修没，至少要保留：

- `/health` 正常返回 200
- `/api/flag` 继续支持平台 checker 的 PUT / GET
- 文件列表仍可查看
- 合法 `.txt` / 图片 / PDF 仍可上传
- 已上传文件仍可下载
- 预览功能仍存在，只是不再接受危险路径

## 交付判断

这题修完后，至少应满足：

1. `../`、绝对路径、伪造文件名都不能通过预览接口读文件
2. `.jpg.php` 这类双扩展上传会被拒绝
3. 正常文本文件仍能完成上传、列表、下载和预览闭环
