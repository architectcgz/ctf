# 防守题解

## 可修改位置

- 平台防守入口实际对应的业务代码在 `/workspace/src/challenge_app.py`
- 题目包内的源文件位置是 `docker/workspace/src/challenge_app.py`
- 这题的数据目录是 `/workspace/data`，你可以通过改业务代码调整表结构、鉴权方式和固件索引方式

## 主要漏洞点

### 1. 设备密钥是全局弱默认值

`valid_device_key()` 现在只拿请求头去对比单个环境变量，默认值还是固定的 `device-demo-key`。这意味着知道默认密钥的人可以伪装任意设备写遥测。

推荐修法：

- 不再使用全局默认设备密钥
- 改成按设备保存独立密钥
- 校验时至少同时绑定 `device_id` 和 `device_key`

如果你不想把改动做得太大，也至少要去掉固定 fallback，让密钥不再是一眼可猜的常量。

### 2. Topic 授权只靠字符串前缀

`topic_read()` 现在用 `allow` 查询参数做字符串前缀判断。攻击方只要构造类似前缀，就有机会跨设备、跨队伍读数据。

推荐修法：

- 把 topic 按结构拆成 `team_id / device_id / rest`
- 只接受结构合法的 topic
- 查询和写入都按拆出来的结构字段比对，不再靠 `startswith()`

这题的关键不是“多写几个前缀判断”，而是把 topic 从字符串技巧改成结构化授权。

### 3. 命令下发没有管理员鉴权

`command()` 现在谁都能调用，只要知道接口就能下发控制命令。这个入口至少需要一个后台侧凭据。

最小修法：

- 为 `/api/command` 增加管理员 token 校验
- token 可以来自环境变量，或者你在代码里明确改成自己的高强度值
- 未通过校验时直接返回 403

### 4. 固件读取可以拼路径

`firmware()` 直接拿 `name` 拼接文件路径，攻击方可以借这个入口探测或读取不该暴露的文件。

推荐修法：

- 固件下载只接受业务定义的固件名称
- 先通过数据库或固定映射表把“逻辑名 -> 实际文件名”查出来
- 然后只读取白名单内的文件

## 推荐修改函数

- `valid_device_key()` 或等价的设备鉴权函数
- `telemetry()`
- `topic_read()`
- `command()`
- `firmware()`

## 保活约束

修补时至少要保留这些能力：

- `/health` 正常返回 200
- `/api/flag` 继续支持平台 checker 的 PUT / GET
- 合法设备仍能上报遥测
- 合法 topic 仍能查询
- 命令下发入口仍存在，只是需要管理员鉴权
- 合法固件仍可读取

## 交付判断

这题修完后，至少应满足：

1. 固定默认密钥不能再伪装设备
2. 伪造 topic 前缀不能跨边界读取数据
3. 未带管理员凭据时不能直接调用 `/api/command`
4. 固件下载不能再通过路径拼接读任意文件
